# Argo Workflows와 Argo Events로 이벤트 기반 Kubernetes 워크플로우 자동화 구축하기

## 소개: 왜 이벤트 기반 워크플로우인가

Kubernetes 환경에서 배치 작업, ML 파이프라인, CI/CD 프로세스를 운영하다 보면 하나의 공통된 문제에 직면하게 됩니다. 바로 "언제, 무엇이, 어떤 조건에서 실행되어야 하는가"를 선언적으로 관리하는 것입니다. 크론잡으로 시작한 스케줄링은 금세 복잡한 의존성 그래프로 변하고, 쉘 스크립트로 엮은 파이프라인은 모니터링도, 재시도도, 확장도 어려워집니다.

Argo Workflows는 CNCF 졸업 프로젝트로, Kubernetes 위에서 DAG(Directed Acyclic Graph) 기반의 워크플로우를 컨테이너 단위로 실행하는 엔진입니다. 각 작업이 독립된 Pod으로 실행되므로 격리성과 확장성을 자연스럽게 확보할 수 있습니다. 여기에 Argo Events를 결합하면, 외부 이벤트(GitHub 웹훅, S3 오브젝트 생성, Kafka 메시지, 크론 스케줄 등)가 발생했을 때 자동으로 워크플로우를 트리거하는 완전한 이벤트 기반 아키텍처를 구현할 수 있습니다.

이 글에서는 Argo Workflows 3.6의 핵심 개선사항을 살펴보고, Argo Events와의 통합을 통해 프로덕션에서 사용할 수 있는 이벤트 기반 워크플로우 자동화 파이프라인을 처음부터 끝까지 구축합니다. 단순히 YAML을 나열하는 것이 아니라, **왜** 이런 아키텍처가 필요한지, 각 설계 결정의 트레이드오프는 무엇인지, 그리고 실제 프로덕션에서 겪게 되는 함정과 해결책까지 다룹니다.

## Argo Workflows 3.6 핵심 개선사항

### 대용량 파라미터 자동 오프로딩

Argo Workflows 3.6에서 가장 주목할 만한 변화는 대용량 파라미터의 자동 오프로딩 기능입니다. 이전 버전에서는 워크플로우 스텝 간에 전달되는 파라미터가 모두 etcd에 저장되었습니다. Kubernetes의 모든 리소스는 etcd에 저장되는데, etcd는 단일 값의 크기를 기본 1.5MB로 제한합니다. ML 파이프라인에서 모델 메타데이터나 대규모 설정 파일을 스텝 간에 전달할 때 이 제한에 자주 부딪혔습니다.

3.6부터는 파라미터 크기가 임계값을 초과하면 자동으로 외부 저장소(S3, GCS, Azure Blob)에 오프로드됩니다. 워크플로우 작성자는 코드를 변경할 필요가 없습니다. 컨트롤러 설정만으로 투명하게 동작합니다.

```yaml
# configmap: workflow-controller-configmap
apiVersion: v1
kind: ConfigMap
metadata:
  name: workflow-controller-configmap
  namespace: argo
data:
  config: |
    artifactRepository:
      s3:
        bucket: argo-artifacts
        endpoint: s3.amazonaws.com
        region: ap-northeast-2
    # 256KB 초과 파라미터는 자동으로 S3에 오프로드
    parameterOffloadThreshold: "256Ki"
```

이것이 중요한 이유는 etcd의 부하를 줄이는 것 이상입니다. 대규모 클러스터에서 수천 개의 워크플로우가 동시에 실행될 때, etcd에 대한 읽기/쓰기 부하가 전체 Kubernetes 클러스터의 안정성에 영향을 미칩니다. 파라미터 오프로딩은 이 병목을 근본적으로 해결합니다.

### 템플릿 스케줄링 힌트와 Pod 스펙 패치

3.6에서는 워크플로우 레벨뿐 아니라 개별 템플릿 레벨에서도 스케줄링 힌트를 지정할 수 있게 되었습니다. GPU가 필요한 ML 추론 스텝은 GPU 노드에, 데이터 전처리 스텝은 일반 노드에 배치하는 것이 자연스러워졌습니다.

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Workflow
metadata:
  generateName: ml-pipeline-
spec:
  entrypoint: pipeline
  templates:
    - name: pipeline
      dag:
        tasks:
          - name: preprocess
            template: preprocess-data
          - name: train
            template: train-model
            dependencies: [preprocess]
          - name: evaluate
            template: evaluate-model
            dependencies: [train]

    - name: preprocess-data
      # CPU 집약적 작업 → 일반 노드
      nodeSelector:
        node-type: compute
      container:
        image: data-pipeline:v2.3
        command: [python, preprocess.py]
        resources:
          requests:
            cpu: "4"
            memory: 8Gi

    - name: train-model
      # GPU 필요 → GPU 노드 스케줄링
      nodeSelector:
        accelerator: nvidia-a100
      tolerations:
        - key: nvidia.com/gpu
          operator: Exists
          effect: NoSchedule
      container:
        image: ml-trainer:v1.8
        command: [python, train.py]
        resources:
          requests:
            cpu: "8"
            memory: 32Gi
            nvidia.com/gpu: "1"
          limits:
            nvidia.com/gpu: "1"

    - name: evaluate-model
      container:
        image: ml-evaluator:v1.2
        command: [python, evaluate.py]
        resources:
          requests:
            cpu: "2"
            memory: 4Gi
```

### 향상된 재시도와 에러 핸들링

프로덕션 워크플로우에서 가장 중요한 것 중 하나는 장애 복원력입니다. 3.6에서는 재시도 정책이 더욱 세밀해졌습니다. 특히 `expression` 기반 재시도 조건을 통해 특정 에러 코드에만 재시도하거나, 특정 로그 패턴이 감지되었을 때만 재시도하는 것이 가능합니다.

```yaml
# 세밀한 재시도 정책 정의
retryStrategy:
  limit: 5
  retryPolicy: OnError
  backoff:
    duration: "30s"
    factor: 2
    maxDuration: "10m"
  # 표현식 기반 재시도 조건: 429(Rate Limit)나 5xx 에러만 재시도
  expression: "lastRetry.status == 429 || (lastRetry.status >= 500 && lastRetry.status < 600)"
```

이런 세밀한 제어가 왜 중요할까요? 외부 API를 호출하는 워크플로우에서 인증 오류(401)에 대해 재시도하는 것은 무의미할 뿐만 아니라 계정이 잠길 위험까지 있습니다. 반면 Rate Limit(429)이나 일시적 서버 오류(503)는 지수 백오프로 재시도하면 대부분 해결됩니다.

## Argo Events 아키텍처 이해하기

Argo Events는 세 가지 핵심 컴포넌트로 구성됩니다: **EventSource**, **EventBus**, **Sensor**. 이 세 컴포넌트가 어떻게 협력하는지 이해하는 것이 이벤트 기반 워크플로우를 설계하는 핵심입니다.

**EventSource**는 외부 이벤트를 수집하는 진입점입니다. GitHub 웹훅 엔드포인트를 노출하거나, S3 버킷의 변경을 감시하거나, Kafka 토픽을 구독합니다. 수집된 이벤트는 CloudEvents 형식으로 정규화되어 EventBus로 전송됩니다.

**EventBus**는 이벤트의 전송 계층입니다. NATS JetStream 또는 Kafka를 백엔드로 사용하며, EventSource에서 발행한 이벤트를 Sensor로 전달합니다. 이벤트 버스를 별도로 두는 이유는 이벤트 소스와 소비자를 디커플링하기 위해서입니다. EventSource가 10개이고 Sensor가 20개라면, 직접 연결 시 200개의 연결이 필요하지만 EventBus를 거치면 30개면 충분합니다.

**Sensor**는 이벤트를 필터링하고, 조건을 평가한 뒤, 실제 액션(워크플로우 트리거, HTTP 요청, Kafka 메시지 발행 등)을 수행합니다. 여러 이벤트의 논리적 조합(AND, OR)으로 복합 조건을 만들 수 있다는 점이 강력합니다.

```yaml
# EventBus 설정: NATS JetStream 기반
apiVersion: argoproj.io/v1alpha1
kind: EventBus
metadata:
  name: default
  namespace: argo-events
spec:
  jetstream:
    version: "2.10.11"
    replicas: 3
    settings: |
      max_memory_store: 1Gi
      max_file_store: 10Gi
    streamConfig: |
      maxAge: 72h
      replicas: 3
      storage: file
    persistence:
      storageClassName: gp3
      accessMode: ReadWriteOnce
      volumeSize: 20Gi
```

왜 JetStream인가? 기본 NATS는 at-most-once 전달만 보장하지만, JetStream은 at-least-once 전달과 메시지 지속성을 보장합니다. 프로덕션에서 이벤트 하나를 놓치는 것은 파이프라인 전체의 무결성을 해치므로, JetStream의 지속성 보장은 필수입니다.

## 실전: GitHub 웹훅으로 CI 파이프라인 트리거하기

가장 일반적인 사용 사례부터 시작합니다. GitHub 리포지토리에 Push가 발생하면 자동으로 빌드, 테스트, 이미지 빌드, 배포까지 이어지는 CI/CD 파이프라인을 구축합니다.

### EventSource: GitHub 웹훅 수신

```yaml
apiVersion: argoproj.io/v1alpha1
kind: EventSource
metadata:
  name: github-webhook
  namespace: argo-events
spec:
  service:
    ports:
      - port: 12000
        targetPort: 12000
  github:
    app-repo:
      repositories:
        - owner: myorg
          names:
            - backend-api
            - frontend-app
      webhook:
        endpoint: /push
        port: "12000"
        method: POST
      # 특정 이벤트만 수신
      events:
        - push
        - pull_request
      # 웹훅 시크릿으로 요청 검증
      webhookSecret:
        name: github-webhook-secret
        key: secret
      # API 토큰 (웹훅 자동 등록에 필요)
      apiToken:
        name: github-api-token
        key: token
      # 요청 본문의 Content-Type
      contentType: json
      # 연결 유지를 위한 설정
      insecure: false
      active: true
```

### Sensor: 이벤트 필터링과 워크플로우 트리거

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Sensor
metadata:
  name: ci-pipeline-trigger
  namespace: argo-events
spec:
  dependencies:
    - name: github-push
      eventSourceName: github-webhook
      eventName: app-repo
      filters:
        data:
          # main 브랜치 Push만 필터링
          - path: body.ref
            type: string
            value:
              - "refs/heads/main"
          # 봇 커밋 제외
          - path: body.head_commit.author.username
            type: string
            comparator: "!="
            value:
              - "github-actions[bot]"

  triggers:
    - template:
        name: ci-workflow
        argoWorkflow:
          operation: submit
          source:
            resource:
              apiVersion: argoproj.io/v1alpha1
              kind: Workflow
              metadata:
                generateName: ci-pipeline-
                namespace: argo
              spec:
                serviceAccountName: argo-workflow-sa
                entrypoint: ci-pipeline
                arguments:
                  parameters:
                    - name: repo-url
                    - name: commit-sha
                    - name: repo-name
                    - name: branch
                templates:
                  - name: ci-pipeline
                    dag:
                      tasks:
                        - name: checkout
                          template: git-checkout
                        - name: test
                          template: run-tests
                          dependencies: [checkout]
                        - name: lint
                          template: run-lint
                          dependencies: [checkout]
                        - name: build-image
                          template: build-push-image
                          dependencies: [test, lint]
                        - name: deploy
                          template: deploy-staging
                          dependencies: [build-image]

                  - name: git-checkout
                    container:
                      image: alpine/git:v2.43.0
                      command: [sh, -c]
                      args:
                        - |
                          git clone {{workflow.parameters.repo-url}} /workspace
                          cd /workspace
                          git checkout {{workflow.parameters.commit-sha}}
                      volumeMounts:
                        - name: workspace
                          mountPath: /workspace

                  - name: run-tests
                    container:
                      image: golang:1.23-bookworm
                      command: [sh, -c]
                      args:
                        - |
                          cd /workspace
                          go test -v -race -coverprofile=coverage.out ./...
                          go tool cover -func=coverage.out
                      volumeMounts:
                        - name: workspace
                          mountPath: /workspace
                    retryStrategy:
                      limit: 2
                      retryPolicy: OnError

                  - name: run-lint
                    container:
                      image: golangci/golangci-lint:v1.62
                      command: [sh, -c]
                      args:
                        - |
                          cd /workspace
                          golangci-lint run --timeout 5m ./...
                      volumeMounts:
                        - name: workspace
                          mountPath: /workspace

                  - name: build-push-image
                    container:
                      image: gcr.io/kaniko-project/executor:v1.23.0
                      args:
                        - --dockerfile=/workspace/Dockerfile
                        - --context=/workspace
                        - --destination=registry.example.com/{{workflow.parameters.repo-name}}:{{workflow.parameters.commit-sha}}
                        - --cache=true
                        - --cache-repo=registry.example.com/{{workflow.parameters.repo-name}}/cache
                      volumeMounts:
                        - name: workspace
                          mountPath: /workspace
                        - name: docker-config
                          mountPath: /kaniko/.docker

                  - name: deploy-staging
                    container:
                      image: bitnami/kubectl:1.31
                      command: [sh, -c]
                      args:
                        - |
                          kubectl set image deployment/{{workflow.parameters.repo-name}} \
                            app=registry.example.com/{{workflow.parameters.repo-name}}:{{workflow.parameters.commit-sha}} \
                            -n staging
                          kubectl rollout status deployment/{{workflow.parameters.repo-name}} \
                            -n staging --timeout=300s

                volumeClaimTemplates:
                  - metadata:
                      name: workspace
                    spec:
                      accessModes: [ReadWriteOnce]
                      storageClassName: gp3
                      resources:
                        requests:
                          storage: 5Gi

          # 이벤트 데이터에서 파라미터 추출
          parameters:
            - src:
                dependencyName: github-push
                dataKey: body.repository.clone_url
              dest: spec.arguments.parameters.0.value
            - src:
                dependencyName: github-push
                dataKey: body.after
              dest: spec.arguments.parameters.1.value
            - src:
                dependencyName: github-push
                dataKey: body.repository.name
              dest: spec.arguments.parameters.2.value
            - src:
                dependencyName: github-push
                dataKey: body.ref
              dest: spec.arguments.parameters.3.value
```

이 구성의 핵심은 **이벤트 데이터에서 워크플로우 파라미터로의 매핑**입니다. GitHub 웹훅 페이로드의 `body.after`(커밋 SHA), `body.repository.clone_url`(리포 URL) 등의 필드를 추출하여 워크플로우의 입력 파라미터로 주입합니다. 이렇게 하면 워크플로우 템플릿 자체는 범용적으로 유지하면서, 트리거 시점에 동적으로 컨텍스트를 전달할 수 있습니다.

## S3 이벤트 기반 데이터 처리 파이프라인

CI/CD 다음으로 흔한 사용 사례는 S3에 파일이 업로드되었을 때 자동으로 데이터 처리 파이프라인을 실행하는 것입니다. 예를 들어, 데이터 팀이 CSV 파일을 S3에 업로드하면 자동으로 검증, 변환, 데이터 웨어하우스 적재까지 이어지는 ETL 파이프라인을 구축할 수 있습니다.

```yaml
# S3 EventSource
apiVersion: argoproj.io/v1alpha1
kind: EventSource
metadata:
  name: s3-data-upload
  namespace: argo-events
spec:
  s3:
    raw-data-upload:
      bucket:
        name: data-lake-raw
      events:
        - s3:ObjectCreated:Put
        - s3:ObjectCreated:CompleteMultipartUpload
      filter:
        prefix: "incoming/"
        suffix: ".csv"
      endpoint: s3.ap-northeast-2.amazonaws.com
      region: ap-northeast-2
      accessKey:
        name: s3-credentials
        key: accessKey
      secretKey:
        name: s3-credentials
        key: secretKey
```

```yaml
# ETL Sensor
apiVersion: argoproj.io/v1alpha1
kind: Sensor
metadata:
  name: etl-pipeline-trigger
  namespace: argo-events
spec:
  dependencies:
    - name: new-data-file
      eventSourceName: s3-data-upload
      eventName: raw-data-upload
      filters:
        data:
          # 10MB 이상 파일만 파이프라인 실행 (작은 파일은 배치 처리)
          - path: notification.0.s3.object.size
            type: number
            comparator: ">="
            value:
              - "10485760"
  triggers:
    - template:
        name: etl-workflow
        argoWorkflow:
          operation: submit
          source:
            resource:
              apiVersion: argoproj.io/v1alpha1
              kind: Workflow
              metadata:
                generateName: etl-
                namespace: argo
              spec:
                entrypoint: etl-pipeline
                arguments:
                  parameters:
                    - name: s3-key
                    - name: s3-bucket
                    - name: file-size
                templates:
                  - name: etl-pipeline
                    dag:
                      tasks:
                        - name: validate
                          template: validate-schema
                        - name: transform
                          template: transform-data
                          dependencies: [validate]
                        - name: load
                          template: load-warehouse
                          dependencies: [transform]
                        - name: notify
                          template: send-notification
                          dependencies: [load]

                  - name: validate-schema
                    container:
                      image: data-tools:v3.1
                      command: [python, -c]
                      args:
                        - |
                          import boto3
                          import pandas as pd
                          import sys

                          s3 = boto3.client('s3')
                          obj = s3.get_object(
                              Bucket='{{workflow.parameters.s3-bucket}}',
                              Key='{{workflow.parameters.s3-key}}'
                          )
                          df = pd.read_csv(obj['Body'], nrows=100)

                          # 필수 컬럼 검증
                          required = ['id', 'timestamp', 'value', 'category']
                          missing = [c for c in required if c not in df.columns]
                          if missing:
                              print(f"Missing columns: {missing}")
                              sys.exit(1)

                          # 데이터 타입 검증
                          print(f"Schema validated: {len(df.columns)} columns")
                          print(f"Sample rows: {len(df)}")
                      resources:
                        requests:
                          cpu: "1"
                          memory: 2Gi

                  - name: transform-data
                    container:
                      image: data-tools:v3.1
                      command: [python, transform.py]
                      args:
                        - --input=s3://{{workflow.parameters.s3-bucket}}/{{workflow.parameters.s3-key}}
                        - --output=s3://data-lake-processed/{{workflow.parameters.s3-key}}
                        - --format=parquet
                        - --compression=zstd
                      resources:
                        requests:
                          cpu: "4"
                          memory: 16Gi
                    retryStrategy:
                      limit: 3
                      backoff:
                        duration: "1m"
                        factor: 2

                  - name: load-warehouse
                    container:
                      image: data-tools:v3.1
                      command: [python, load.py]
                      args:
                        - --source=s3://data-lake-processed/{{workflow.parameters.s3-key}}
                        - --target=warehouse.analytics.processed_data
                      env:
                        - name: DB_CONNECTION
                          valueFrom:
                            secretKeyRef:
                              name: warehouse-credentials
                              key: connection-string

                  - name: send-notification
                    container:
                      image: curlimages/curl:8.7.1
                      command: [sh, -c]
                      args:
                        - |
                          curl -X POST "$SLACK_WEBHOOK" \
                            -H 'Content-Type: application/json' \
                            -d '{
                              "text": "✅ ETL 완료: {{workflow.parameters.s3-key}} ({{workflow.parameters.file-size}} bytes)"
                            }'
                      env:
                        - name: SLACK_WEBHOOK
                          valueFrom:
                            secretKeyRef:
                              name: slack-webhook
                              key: url
          parameters:
            - src:
                dependencyName: new-data-file
                dataKey: notification.0.s3.object.key
              dest: spec.arguments.parameters.0.value
            - src:
                dependencyName: new-data-file
                dataKey: notification.0.s3.bucket.name
              dest: spec.arguments.parameters.1.value
            - src:
                dependencyName: new-data-file
                dataKey: notification.0.s3.object.size
              dest: spec.arguments.parameters.2.value
```

파일 크기 필터(`>= 10MB`)를 둔 이유가 있습니다. 소규모 파일은 개별 워크플로우로 처리하면 Pod 스케줄링 오버헤드가 실제 처리 시간보다 클 수 있습니다. 작은 파일은 별도의 크론 기반 배치 워크플로우로 모아서 처리하는 것이 효율적입니다.

## 복합 이벤트 패턴: 여러 조건을 결합하기

실제 프로덕션에서는 단일 이벤트가 아니라 여러 이벤트의 조합으로 워크플로우를 트리거해야 하는 경우가 많습니다. 예를 들어, "모든 마이크로서비스의 테스트가 통과하고, 보안 스캔이 완료된 후에만 프로덕션 배포를 진행"하는 시나리오입니다.

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Sensor
metadata:
  name: production-deploy-gate
  namespace: argo-events
spec:
  dependencies:
    - name: tests-passed
      eventSourceName: ci-events
      eventName: test-results
      filters:
        data:
          - path: body.status
            type: string
            value: ["passed"]
    - name: security-scan
      eventSourceName: ci-events
      eventName: security-scan
      filters:
        data:
          - path: body.vulnerabilities.critical
            type: number
            comparator: "=="
            value: ["0"]
    - name: approval
      eventSourceName: slack-events
      eventName: deploy-approval
      filters:
        data:
          - path: body.action
            type: string
            value: ["approved"]

  # AND 조건: 세 가지 이벤트가 모두 충족되어야 트리거
  # afterAll: 모든 의존성이 순서와 관계없이 충족되면 실행
  triggers:
    - template:
        name: production-deploy
        conditions: "tests-passed && security-scan && approval"
        argoWorkflow:
          operation: submit
          source:
            resource:
              apiVersion: argoproj.io/v1alpha1
              kind: Workflow
              metadata:
                generateName: prod-deploy-
              spec:
                entrypoint: blue-green-deploy
                templates:
                  - name: blue-green-deploy
                    dag:
                      tasks:
                        - name: deploy-green
                          template: deploy
                          arguments:
                            parameters:
                              - name: target
                                value: green
                        - name: smoke-test
                          template: smoke-test
                          dependencies: [deploy-green]
                        - name: switch-traffic
                          template: switch
                          dependencies: [smoke-test]
                        - name: cleanup-blue
                          template: cleanup
                          dependencies: [switch-traffic]

                  - name: deploy
                    inputs:
                      parameters:
                        - name: target
                    container:
                      image: deployer:v2.0
                      command: [./deploy.sh]
                      args: ["--target", "{{inputs.parameters.target}}"]

                  - name: smoke-test
                    container:
                      image: test-runner:v1.5
                      command: [./smoke-test.sh]
                      args: ["--endpoint", "https://green.example.com"]
                    retryStrategy:
                      limit: 3
                      backoff:
                        duration: "10s"

                  - name: switch
                    container:
                      image: traffic-manager:v1.2
                      command: [./switch-traffic.sh]
                      args: ["--from", "blue", "--to", "green"]

                  - name: cleanup
                    container:
                      image: deployer:v2.0
                      command: [./cleanup.sh]
                      args: ["--target", "blue"]
```

`conditions: "tests-passed && security-scan && approval"` 표현식이 핵심입니다. 단순 AND뿐만 아니라 `(A && B) || C` 같은 복잡한 논리 조합도 가능합니다. 이를 통해 "긴급 핫픽스는 보안 스캔을 건너뛰되, 일반 배포는 모든 게이트를 통과해야 한다"는 정책도 선언적으로 표현할 수 있습니다.

## 프로덕션 운영을 위한 모니터링과 가시성

워크플로우 자동화를 프로덕션에 운영하려면 모니터링이 필수입니다. Argo Workflows는 Prometheus 메트릭을 네이티브로 노출하며, 커스텀 메트릭도 워크플로우 레벨에서 정의할 수 있습니다.

```yaml
# 워크플로우에 커스텀 메트릭 추가
apiVersion: argoproj.io/v1alpha1
kind: Workflow
metadata:
  generateName: monitored-pipeline-
spec:
  entrypoint: pipeline
  metrics:
    prometheus:
      - name: pipeline_duration_seconds
        help: "Total pipeline execution time"
        labels:
          - key: pipeline_type
            value: "etl"
          - key: status
            value: "{{workflow.status}}"
        gauge:
          value: "{{workflow.duration}}"
      - name: pipeline_runs_total
        help: "Total number of pipeline runs"
        labels:
          - key: pipeline_type
            value: "etl"
          - key: status
            value: "{{workflow.status}}"
        counter:
          value: "1"
  templates:
    - name: pipeline
      dag:
        tasks:
          - name: step1
            template: process
    - name: process
      metrics:
        prometheus:
          - name: step_memory_usage_bytes
            help: "Peak memory usage of step"
            gauge:
              value: "{{outputs.parameters.peak-memory}}"
      container:
        image: processor:v1.0
        command: [sh, -c]
        args:
          - |
            # 작업 실행
            python process.py
            # 메모리 사용량 기록
            peak_mem=$(cat /sys/fs/cgroup/memory.peak 2>/dev/null || echo "0")
            echo $peak_mem > /tmp/peak-memory
      outputs:
        parameters:
          - name: peak-memory
            valueFrom:
              path: /tmp/peak-memory
```

Grafana 대시보드에서 모니터링해야 할 핵심 메트릭은 다음과 같습니다:

- **워크플로우 성공/실패율**: `argo_workflows_count{status="Succeeded"}` vs `{status="Failed"}`
- **평균 실행 시간**: 파이프라인별 P50, P95, P99 레이턴시
- **Pod 스케줄링 대기 시간**: 리소스 부족으로 인한 병목 감지
- **재시도 횟수**: 특정 스텝의 재시도가 급증하면 외부 의존성 문제 의심
- **동시 실행 워크플로우 수**: 클러스터 용량 계획에 필수

> **운영 팁**: Argo Workflows 컨트롤러의 `--workflow-workers` 플래그로 동시 처리 워커 수를 조절할 수 있습니다. 기본값은 32이지만, 대규모 클러스터에서는 64~128까지 올리면 처리량이 크게 개선됩니다. 단, etcd와 API Server에 대한 부하도 비례하여 증가하므로 모니터링과 함께 점진적으로 조정하세요.

## 비용 최적화: Spot 인스턴스와 리소스 관리

대규모 워크플로우를 운영할 때 비용은 무시할 수 없는 요소입니다. Argo Workflows와 Kubernetes의 기능을 활용하면 비용을 크게 절감할 수 있습니다.

```yaml
# Spot 인스턴스를 활용한 비용 최적화 워크플로우
apiVersion: argoproj.io/v1alpha1
kind: Workflow
metadata:
  generateName: cost-optimized-
spec:
  entrypoint: batch-process
  # Pod GC로 완료된 Pod 자동 정리 → 리소스 회수
  podGC:
    strategy: OnPodCompletion
    deleteDelayDuration: "5m"
  # TTL로 워크플로우 객체 자동 정리 → etcd 부하 감소
  ttlStrategy:
    secondsAfterCompletion: 86400  # 완료 후 24시간 뒤 삭제
    secondsAfterFailure: 604800    # 실패 시 7일 유지 (디버깅)

  templates:
    - name: batch-process
      dag:
        tasks:
          - name: process-batch
            template: spot-tolerant-worker
            withSequence:
              count: "100"

    - name: spot-tolerant-worker
      # Spot 인스턴스 활용
      tolerations:
        - key: kubernetes.io/spot
          operator: Equal
          value: "true"
          effect: NoSchedule
      nodeSelector:
        node-lifecycle: spot
      # Spot 중단 시 자동 재시도
      retryStrategy:
        limit: 5
        retryPolicy: Always
        backoff:
          duration: "30s"
          factor: 2
      container:
        image: batch-worker:v2.1
        command: [python, worker.py]
        args: ["--batch-id", "{{item}}"]
        resources:
          requests:
            cpu: "2"
            memory: 4Gi
          # requests와 limits를 동일하게 → Guaranteed QoS
          # Spot 중단 외의 OOM 방지
          limits:
            cpu: "2"
            memory: 4Gi
```

Spot 인스턴스를 활용하면 On-Demand 대비 60~90% 비용을 절감할 수 있습니다. 핵심은 **워크플로우가 멱등(idempotent)해야 한다**는 것입니다. Spot 인스턴스는 언제든 중단될 수 있으므로, 각 스텝이 중간 결과를 외부 저장소에 체크포인팅하고, 재시도 시 이전 진행 상태부터 재개할 수 있어야 합니다.

`podGC`와 `ttlStrategy`도 비용 최적화의 중요한 요소입니다. 완료된 Pod과 워크플로우 객체를 제때 정리하지 않으면, 클러스터에 수만 개의 Completed Pod이 쌓이고 etcd 용량을 소모합니다. 특히 대규모 배치 워크플로우에서는 `podGC.strategy: OnPodCompletion`을 반드시 설정하세요.

## 보안 베스트 프랙티스

이벤트 기반 워크플로우는 외부 이벤트를 받아들이므로 보안에 특히 신경써야 합니다. 핵심 보안 원칙을 정리합니다.

```yaml
# 최소 권한 ServiceAccount
apiVersion: v1
kind: ServiceAccount
metadata:
  name: argo-workflow-sa
  namespace: argo
---
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: argo-workflow-role
  namespace: argo
rules:
  # 워크플로우 실행에 필요한 최소 권한만 부여
  - apiGroups: [""]
    resources: ["pods", "pods/log"]
    verbs: ["get", "list", "watch"]
  - apiGroups: [""]
    resources: ["configmaps"]
    verbs: ["get"]
  - apiGroups: [""]
    resources: ["secrets"]
    verbs: ["get"]
    # 특정 시크릿만 접근 허용
    resourceNames:
      - "docker-config"
      - "s3-credentials"
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: argo-workflow-binding
  namespace: argo
subjects:
  - kind: ServiceAccount
    name: argo-workflow-sa
roleRef:
  kind: Role
  name: argo-workflow-role
  apiGroup: rbac.authorization.k8s.io
```

> **보안 필수 체크리스트:**
> - 모든 웹훅 EventSource에 시크릿 검증 활성화
> - 워크플로우 ServiceAccount는 최소 권한 원칙 적용
> - 컨테이너는 non-root 사용자로 실행 (`securityContext.runAsNonRoot: true`)
> - 네트워크 정책으로 워크플로우 Pod의 외부 접근 제한
> - 시크릿은 External Secrets Operator나 Vault로 관리
> - PodSecurityStandard의 `restricted` 프로파일 적용 고려

## 마이그레이션 전략: Jenkins/GitLab CI에서 Argo로

기존 CI/CD 시스템에서 Argo로 마이그레이션할 때는 빅뱅 전환보다 점진적 접근이 안전합니다. 먼저 새 프로젝트나 배치 작업부터 Argo로 시작하고, 안정성이 검증된 후 기존 파이프라인을 하나씩 이전합니다. Jenkins의 Groovy 파이프라인은 Argo의 DAG 템플릿으로 거의 1:1 매핑이 가능합니다. GitLab CI의 stage/job 구조도 마찬가지입니다.

가장 큰 차이점은 실행 환경입니다. Jenkins는 에이전트 노드에서 실행되지만, Argo는 모든 스텝이 독립된 컨테이너입니다. 이는 격리성과 재현성 면에서 명확한 이점이지만, 스텝 간 파일 공유를 위해 Volume(PVC 또는 emptyDir)을 명시적으로 설정해야 합니다. Jenkins에서 당연히 되던 워크스페이스 공유가 Argo에서는 설계가 필요한 부분입니다.

## 결론

Argo Workflows와 Argo Events의 조합은 Kubernetes 네이티브 워크플로우 자동화의 사실상 표준이 되어가고 있습니다. 선언적 워크플로우 정의, 이벤트 기반 트리거, 세밀한 재시도 정책, 그리고 Kubernetes 생태계와의 자연스러운 통합은 기존 CI/CD 도구나 크론 기반 접근법이 제공하지 못하는 수준의 안정성과 확장성을 제공합니다.

핵심은 단순히 도구를 도입하는 것이 아니라, 이벤트 기반 사고방식으로 전환하는 것입니다. "이 작업은 언제 실행되는가"가 아니라 "이 작업은 어떤 이벤트에 반응하는가"로 관점을 바꾸면, 시스템의 결합도가 낮아지고 각 컴포넌트의 테스트 가능성과 재사용성이 높아집니다. Argo Workflows 3.6의 개선사항들과 Argo Events의 강력한 이벤트 필터링은 이 비전을 프로덕션에서 현실화하는 데 필요한 도구를 제공합니다.
