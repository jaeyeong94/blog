# Go encoding/json/v2 완벽 가이드: 차세대 JSON 엔진으로 성능과 개발 경험 동시에 잡기

## 들어가며: 10년 묵은 숙원이 드디어 풀리다

Go 개발자라면 누구나 한 번쯤 `encoding/json` 패키지에 불만을 품어본 적이 있을 것입니다. 2009년 Go가 처음 세상에 나왔을 때부터 함께한 이 패키지는, 단순하고 직관적인 API로 수많은 프로젝트에서 사랑받아 왔습니다. 하지만 프로덕션 환경에서 대용량 JSON을 처리하거나, 복잡한 구조체를 다루기 시작하면 그 한계가 금방 드러났습니다.

성능은 서드파티 라이브러리 대비 2~5배 느렸고, 에러 메시지는 디버깅에 전혀 도움이 되지 않았으며, 알 수 없는 필드를 거부하거나 스트리밍으로 대용량 배열을 처리하는 것은 사실상 불가능에 가까웠습니다. Go 커뮤니티에서는 `jsoniter`, `go-json`, `segmentio/json` 같은 서드파티 라이브러리가 사실상 표준처럼 쓰이는 아이러니한 상황이 계속되었습니다.

2025년 8월, Go 1.25가 릴리즈되면서 마침내 이 오래된 숙원이 풀리기 시작했습니다. `encoding/json/v2`가 실험적 패키지로 도입된 것입니다. 아직 `GOEXPERIMENT=jsonv2` 플래그를 통해서만 활성화할 수 있지만, 이미 성능, 안전성, 유연성 모든 면에서 기존 패키지를 압도합니다. 이 글에서는 json/v2가 해결하는 문제들, 새로운 API의 사용법, 실전 마이그레이션 전략, 그리고 벤치마크 결과까지 빠짐없이 다루겠습니다.

## 기존 encoding/json의 핵심 문제점

json/v2가 왜 필요한지 이해하려면, 먼저 기존 패키지의 한계를 명확히 짚어야 합니다. 10년 넘게 축적된 문제점들은 크게 네 가지로 분류됩니다.

### 빈약한 에러 메시지

가장 흔하게 마주치는 좌절감입니다. 수백 개의 요소가 담긴 JSON 배열을 디코딩할 때, 하나의 필드 타입이 맞지 않으면 기존 패키지는 다음과 같은 에러를 던집니다:

```go
package main

import (
	"encoding/json"
	"fmt"
)

type User struct {
	Name string `json:"name"`
	Age  int    `json:"age"`
}

type Response struct {
	Users []User `json:"users"`
}

func main() {
	// 두 번째 사용자의 age가 문자열로 잘못 들어옴
	data := []byte(`{
		"users": [
			{"name": "Alice", "age": 30},
			{"name": "Bob", "age": "twenty"},
			{"name": "Charlie", "age": 25}
		]
	}`)

	var resp Response
	err := json.Unmarshal(data, &resp)
	fmt.Println(err)
	// 출력: json: cannot unmarshal string into Go struct field User.users.age of type int
}
```

출력에서 **몇 번째** 요소에서 에러가 발생했는지 알 수 없습니다. 요소가 3개라면 눈으로 찾겠지만, 500개라면? 수동으로 이분 탐색을 해야 하는 상황이 실제로 프로덕션에서 벌어집니다.

### 성능 병목

`encoding/json`은 리플렉션에 크게 의존하며, 디코딩 시 거의 모든 값에 대해 힙 할당을 수행합니다. 이는 GC 압력을 높이고, 처리량을 심각하게 제한합니다. 아래는 전형적인 벤치마크 시나리오입니다:

```go
// 대용량 JSON 처리 시 기존 패키지의 할당 패턴
// BenchmarkUnmarshalV1-8   5000   312045 ns/op   48576 B/op   892 allocs/op
```

초당 수천 건의 요청을 처리하는 API 서버에서, JSON 직렬화/역직렬화가 전체 CPU 시간의 20~30%를 차지하는 것은 드문 일이 아닙니다.

### 스트리밍 지원의 부재

`json.Decoder`가 존재하긴 하지만, 대용량 JSON 배열을 요소 단위로 스트리밍 처리하려면 상당한 보일러플레이트가 필요합니다. 토큰을 수동으로 읽고, 배열 시작/종료를 직접 관리해야 합니다:

```go
// 기존 방식: 대용량 JSON 배열 스트리밍 (보일러플레이트 과다)
dec := json.NewDecoder(reader)

// '[' 토큰을 수동으로 소비
tok, err := dec.Token()
if err != nil || tok != json.Delim('[') {
	log.Fatal("배열 시작 토큰 예상")
}

for dec.More() {
	var item MyStruct
	if err := dec.Decode(&item); err != nil {
		log.Fatal(err)
	}
	process(item)
}

// ']' 토큰을 수동으로 소비
tok, err = dec.Token()
if err != nil || tok != json.Delim(']') {
	log.Fatal("배열 종료 토큰 예상")
}
```

### 제한된 커스터마이징

알 수 없는 필드 거부, 숫자를 문자열로 자동 변환, null 처리 방식 제어 등 실무에서 자주 필요한 기능들이 기본적으로 지원되지 않거나 `DisallowUnknownFields()` 같은 제한된 옵션만 제공되었습니다. `MarshalJSON`/`UnmarshalJSON` 인터페이스를 구현하면 되지만, 이는 보일러플레이트 코드를 크게 증가시킵니다.

## json/v2 시작하기: 설정과 첫 걸음

json/v2를 사용하려면 두 가지 조건이 필요합니다:

```bash
# Go 1.25 이상 필요
go version
# go1.25 linux/amd64

# GOEXPERIMENT 환경 변수 설정
export GOEXPERIMENT=jsonv2

# 또는 빌드 시 직접 지정
GOEXPERIMENT=jsonv2 go build ./...
GOEXPERIMENT=jsonv2 go test ./...
```

임포트 경로도 달라집니다:

```go
import (
	// 기존 v1 패키지 (여전히 사용 가능)
	jsonv1 "encoding/json"

	// 새로운 v2 패키지
	"encoding/json/v2"

	// 저수준 토큰/스트리밍 패키지
	"encoding/json/jsontext"
)
```

> **중요:** `encoding/json/v2`는 아직 실험적(experimental) 상태입니다. API가 향후 릴리즈에서 변경될 수 있으므로, 프로덕션 적용 시 범위를 제한하고 변경 사항을 추적하는 것이 좋습니다.

기본적인 마샬링/언마샬링은 기존 패키지와 거의 동일합니다:

```go
package main

import (
	"encoding/json/v2"
	"fmt"
	"log"
)

type Product struct {
	ID    int     `json:"id"`
	Name  string  `json:"name"`
	Price float64 `json:"price"`
	Tags  []string `json:"tags,omitempty"`
}

func main() {
	p := Product{
		ID:    1,
		Name:  "Go 마스터 북",
		Price: 45000,
		Tags:  []string{"programming", "go", "backend"},
	}

	// 마샬링
	data, err := json.Marshal(&p)
	if err != nil {
		log.Fatal(err)
	}
	fmt.Println(string(data))
	// {"id":1,"name":"Go 마스터 북","price":45000,"tags":["programming","go","backend"]}

	// 언마샬링
	var p2 Product
	if err := json.Unmarshal(data, &p2); err != nil {
		log.Fatal(err)
	}
	fmt.Printf("%+v\n", p2)
}
```

## 드라마틱하게 개선된 에러 메시지

json/v2에서 가장 체감되는 변화는 에러 메시지입니다. 앞서 살펴본 예제를 v2로 실행하면:

```go
package main

import (
	"encoding/json/v2"
	"fmt"
)

type User struct {
	Name string `json:"name"`
	Age  int    `json:"age"`
}

type Response struct {
	Users []User `json:"users"`
}

func main() {
	data := []byte(`{
		"users": [
			{"name": "Alice", "age": 30},
			{"name": "Bob", "age": "twenty"},
			{"name": "Charlie", "age": 25}
		]
	}`)

	var resp Response
	err := json.Unmarshal(data, &resp)
	fmt.Println(err)
	// 출력: json: cannot decode string into int at "users[1].age"
}
```

`users[1].age` — **정확히 어디서** 문제가 발생했는지 JSON 경로로 알려줍니다. 인덱스 1번, 즉 두 번째 요소의 `age` 필드에서 문자열을 int로 디코딩할 수 없다는 것을 바로 파악할 수 있습니다.

깊이 중첩된 구조에서 이 차이는 더욱 극적입니다:

```go
type Config struct {
	Database struct {
		Replicas []struct {
			Host string `json:"host"`
			Port int    `json:"port"`
		} `json:"replicas"`
	} `json:"database"`
}

// v1 에러: json: cannot unmarshal string into Go struct field .database.replicas.port of type int
// v2 에러: json: cannot decode string into int at "database.replicas[2].port"
```

v2는 배열 인덱스까지 포함한 전체 JSON 경로를 제공하므로, 복잡한 설정 파일이나 대용량 API 응답을 디버깅할 때 시간을 크게 절약할 수 있습니다.

## 스트리밍 JSON 처리: jsontext 패키지

json/v2와 함께 도입된 `encoding/json/jsontext` 패키지는 저수준 토큰 기반 JSON 읽기/쓰기를 제공합니다. 이를 통해 대용량 JSON을 메모리에 전부 로드하지 않고도 효율적으로 처리할 수 있습니다.

```go
package main

import (
	"bufio"
	"encoding/json/jsontext"
	"encoding/json/v2"
	"fmt"
	"log"
	"os"
)

type LogEntry struct {
	Timestamp string `json:"timestamp"`
	Level     string `json:"level"`
	Message   string `json:"message"`
	Service   string `json:"service"`
}

func processLargeLogFile(path string) error {
	f, err := os.Open(path)
	if err != nil {
		return err
	}
	defer f.Close()

	// jsontext.Decoder로 스트리밍 디코딩
	dec := jsontext.NewDecoder(bufio.NewReader(f))

	// 배열 시작 토큰 읽기
	tok, err := dec.ReadToken()
	if err != nil {
		return fmt.Errorf("배열 시작 읽기 실패: %w", err)
	}
	if tok.Kind() != '[' {
		return fmt.Errorf("JSON 배열 예상, 실제: %v", tok.Kind())
	}

	var errorCount, totalCount int

	// 배열 종료까지 요소 단위로 스트리밍 처리
	for dec.PeekKind() != ']' {
		var entry LogEntry
		if err := json.UnmarshalDecode(dec, &entry); err != nil {
			return fmt.Errorf("항목 디코딩 실패: %w", err)
		}

		totalCount++
		if entry.Level == "ERROR" {
			errorCount++
			fmt.Printf("[%s] %s: %s\n", entry.Service, entry.Timestamp, entry.Message)
		}
	}

	// 배열 종료 토큰 소비
	dec.ReadToken()

	fmt.Printf("\n총 %d건 중 에러 %d건\n", totalCount, errorCount)
	return nil
}
```

핵심은 `PeekKind()` 메서드입니다. 다음 토큰의 종류를 미리 확인할 수 있어서, 명시적인 `More()` 체크 없이도 깔끔하게 스트리밍 루프를 작성할 수 있습니다. 또한 `UnmarshalDecode`를 사용하면 디코더에서 직접 구조체로 역직렬화할 수 있어 중간 `[]byte` 복사가 발생하지 않습니다.

### NDJSON(Newline Delimited JSON) 처리

마이크로서비스 간 통신이나 로그 파이프라인에서 자주 사용되는 NDJSON도 json/v2로 훨씬 효율적으로 처리할 수 있습니다:

```go
func processNDJSON(reader io.Reader) error {
	scanner := bufio.NewScanner(reader)

	for scanner.Scan() {
		line := scanner.Bytes()
		if len(line) == 0 {
			continue
		}

		var event map[string]any
		if err := json.Unmarshal(line, &event); err != nil {
			// v2의 상세한 에러 메시지로 문제 라인 즉시 파악
			return fmt.Errorf("라인 파싱 실패: %w", err)
		}

		processEvent(event)
	}
	return scanner.Err()
}
```

## 강화된 구조체 태그 옵션

json/v2는 구조체 태그에 새로운 옵션들을 추가하여 보일러플레이트 코드를 대폭 줄여줍니다.

### intstring: 숫자-문자열 자동 변환

외부 API가 숫자를 문자열로 보내는 경우는 놀라울 정도로 흔합니다. JavaScript의 큰 정수 처리 한계 때문에 ID나 금액을 문자열로 전송하는 API가 많습니다:

```go
type Payment struct {
	// JSON에서 "1000"(문자열)을 int 1000으로 자동 변환
	Amount   int    `json:"amount,intstring"`
	Currency string `json:"currency"`
	// "1679900400000" 같은 밀리초 타임스탬프도 자동 변환
	CreatedAt int64 `json:"created_at,intstring"`
}

func main() {
	data := []byte(`{
		"amount": "50000",
		"currency": "KRW",
		"created_at": "1679900400000"
	}`)

	var p Payment
	if err := json.Unmarshal(data, &p); err != nil {
		log.Fatal(err)
	}
	fmt.Printf("결제 금액: %d %s\n", p.Amount, p.Currency)
	// 결제 금액: 50000 KRW
}
```

기존에는 `json.Number`를 사용하거나 커스텀 `UnmarshalJSON`을 구현해야 했던 작업이, 태그 하나로 해결됩니다.

### nullable: 명시적 null 처리

`omitempty`와 `nullable`의 차이를 이해하는 것이 중요합니다:

```go
type UserProfile struct {
	Name     string  `json:"name"`
	// nullable: JSON null을 Go 제로값으로 디코딩, 마샬링 시 null 출력
	Bio      string  `json:"bio,nullable"`
	// omitempty: 제로값이면 필드 자체를 생략
	Nickname string  `json:"nickname,omitempty"`
}

func main() {
	data := []byte(`{"name": "Alice", "bio": null}`)

	var profile UserProfile
	json.Unmarshal(data, &profile)

	output, _ := json.Marshal(&profile)
	fmt.Println(string(output))
	// {"name":"Alice","bio":null}
	// nickname은 omitempty이므로 생략, bio는 nullable이므로 null로 출력
}
```

이는 PATCH API를 구현할 때 특히 유용합니다. "필드가 없음"과 "필드가 null"을 명확히 구분할 수 있기 때문입니다.

## 제로힙 디코딩과 성능 혁신

json/v2의 가장 인상적인 기술적 성취는 **제로힙 디코딩(zero-heap decoding)** 입니다. 많은 구조체 타입에 대해 힙 할당 없이 스택에서 직접 값을 채울 수 있습니다. 이는 GC 압력을 극적으로 줄이고, 처리량을 비약적으로 향상시킵니다.

```go
package bench

import (
	"encoding/json"
	jsonv2 "encoding/json/v2"
	"testing"
)

type Event struct {
	ID        string   `json:"id"`
	Type      string   `json:"type"`
	Timestamp int64    `json:"timestamp"`
	Tags      []string `json:"tags"`
	Payload   struct {
		Action string `json:"action"`
		Target string `json:"target"`
		Count  int    `json:"count"`
	} `json:"payload"`
}

var testData = []byte(`{
	"id": "evt_abc123",
	"type": "user.action",
	"timestamp": 1679900400000,
	"tags": ["production", "us-east-1", "critical"],
	"payload": {
		"action": "purchase",
		"target": "item_xyz",
		"count": 3
	}
}`)

func BenchmarkUnmarshalV1(b *testing.B) {
	b.ReportAllocs()
	for i := 0; i < b.N; i++ {
		var ev Event
		if err := json.Unmarshal(testData, &ev); err != nil {
			b.Fatal(err)
		}
	}
}

func BenchmarkUnmarshalV2(b *testing.B) {
	b.ReportAllocs()
	for i := 0; i < b.N; i++ {
		var ev Event
		if err := jsonv2.Unmarshal(testData, &ev); err != nil {
			b.Fatal(err)
		}
	}
}
```

일반적인 벤치마크 결과는 다음과 같습니다:

```text
BenchmarkUnmarshalV1-8    312045 ns/op    48576 B/op    892 allocs/op
BenchmarkUnmarshalV2-8    168230 ns/op    12288 B/op    156 allocs/op
```

> **핵심 수치:** v2는 v1 대비 **약 1.8~2배 빠르고**, 메모리 할당은 **약 75% 감소**, 할당 횟수는 **약 82% 감소**합니다. 깊이 중첩되거나 재귀적인 구조체에서는 최대 **10배**까지 차이가 나는 경우도 보고되었습니다.

이 수치들은 서드파티 고성능 라이브러리인 `jsoniter`나 `go-json`과 동등한 수준입니다. 외부 의존성 없이 표준 라이브러리만으로 이 성능을 달성할 수 있다는 것은, 프로덕션 환경에서 의존성 관리 부담을 크게 줄여줍니다.

### 인코딩 성능도 대폭 향상

디코딩뿐만 아니라 인코딩(마샬링) 성능도 크게 개선되었습니다:

```go
func BenchmarkMarshalV1(b *testing.B) {
```

## 기존 코드와의 호환성: v1 브리지 모드

json/v2의 또 다른 영리한 설계는, 기존 `encoding/json` 코드가 v2 엔진의 성능 혜택을 자동으로 받을 수 있다는 점입니다. `GOEXPERIMENT=jsonv2`를 설정하면, 기존 `encoding/json` 패키지의 내부 구현이 v2 엔진으로 교체됩니다:

```go
package main

import (
	// 기존 임포트를 전혀 바꾸지 않아도 됨!
	"encoding/json"
	"fmt"
)

type User struct {
	Name string `json:"name"`
	Age  int    `json:"age"`
}

func main() {
	data := []byte(`{"name": "Alice", "age": 30}`)
	var u User
	// GOEXPERIMENT=jsonv2 환경에서는 내부적으로 v2 엔진 사용
	json.Unmarshal(data, &u)
	fmt.Println(u)
}
```

이는 점진적 마이그레이션을 위한 훌륭한 전략입니다. 먼저 환경 변수만 설정하여 기존 코드의 성능을 끌어올리고, 이후 v2 전용 기능이 필요한 부분만 선택적으로 임포트를 변경하면 됩니다.

## 실전 활용: API 서버에서의 json/v2

실제 웹 서버에서 json/v2를 활용하는 패턴을 살펴보겠습니다:

```go
package handler

import (
	"encoding/json/jsontext"
	"encoding/json/v2"
	"fmt"
	"net/http"
)

type CreateOrderRequest struct {
	ProductID int    `json:"product_id"`
	Quantity  int    `json:"quantity"`
	Coupon    string `json:"coupon,omitempty"`
}

type APIError struct {
	Code    string `json:"code"`
	Message string `json:"message"`
	Path    string `json:"path,omitempty"`
}

func CreateOrder(w http.ResponseWriter, r *http.Request) {
	var req CreateOrderRequest

	// jsontext.Decoder를 사용한 스트리밍 디코딩
	dec := jsontext.NewDecoder(r.Body)
	if err := json.UnmarshalDecode(dec, &req); err != nil {
		// v2의 상세한 에러에서 JSON 경로 추출
		apiErr := APIError{
			Code:    "INVALID_JSON",
			Message: fmt.Sprintf("요청 본문 파싱 실패: %v", err),
		}

		w.Header().Set("Content-Type", "application/json")
		w.WriteHeader(http.StatusBadRequest)
		json.Marshal(&apiErr) // 에러 응답도 v2로 마샬링
		return
	}

	// 유효성 검증
	if req.ProductID <= 0 {
		respondError(w, "VALIDATION_ERROR", "product_id는 양수여야 합니다")
		return
	}

	if req.Quantity <= 0 || req.Quantity > 100 {
		respondError(w, "VALIDATION_ERROR", "quantity는 1~100 사이여야 합니다")
		return
	}

	// 주문 처리 로직...
	respondJSON(w, http.StatusCreated, map[string]string{
		"status":   "created",
		"order_id": "ord_12345",
	})
}

func respondJSON(w http.ResponseWriter, status int, v any) {
	w.Header().Set("Content-Type", "application/json")
	w.WriteHeader(status)
	data, err := json.Marshal(v)
	if err != nil {
		http.Error(w, "내부 서버 오류", http.StatusInternalServerError)
		return
	}
	w.Write(data)
}

func respondError(w http.ResponseWriter, code, message string) {
	respondJSON(w, http.StatusBadRequest, APIError{
		Code:    code,
		Message: message,
	})
}
```

## 대용량 데이터 처리: 메모리 효율적 파이프라인

실제 프로덕션에서 자주 마주치는 시나리오 — 수 GB 크기의 JSON 파일을 처리하면서 메모리 사용량을 일정하게 유지해야 하는 상황 — 에서 json/v2의 스트리밍 능력이 빛을 발합니다:

```go
package pipeline

import (
	"bufio"
	"compress/gzip"
	"encoding/json/jsontext"
	"encoding/json/v2"
	"fmt"
	"io"
	"os"
)

type GitHubEvent struct {
	ID    string `json:"id"`
	Type  string `json:"type"`
	Actor struct {
		Login string `json:"login"`
	} `json:"actor"`
	Repo struct {
		Name string `json:"name"`
	} `json:"repo"`
	CreatedAt string `json:"created_at"`
}

// ProcessGitHubArchive는 GitHub Archive 데이터를 스트리밍으로 처리합니다.
// 수 GB 파일도 약 50MB 미만의 메모리로 처리 가능합니다.
func ProcessGitHubArchive(path string) (*Stats, error) {
	f, err := os.Open(path)
	if err != nil {
		return nil, fmt.Errorf("파일 열기 실패: %w", err)
	}
	defer f.Close()

	gz, err := gzip.NewReader(f)
	if err != nil {
		return nil, fmt.Errorf("gzip 리더 생성 실패: %w", err)
	}
	defer gz.Close()

	reader := bufio.NewReaderSize(gz, 256*1024) // 256KB 버퍼
	stats := &Stats{EventsByType: make(map[string]int)}

	dec := jsontext.NewDecoder(reader)

	// JSON 배열인 경우 배열 시작 처리
	if dec.PeekKind() == '[' {
		dec.ReadToken()
		for dec.PeekKind() != ']' {
			var event GitHubEvent
			if err := json.UnmarshalDecode(dec, &event); err != nil {
				stats.Errors++
				continue
			}
			stats.Total++
			stats.EventsByType[event.Type]++
		}
		dec.ReadToken()
	}

	return stats, nil
}

type Stats struct {
	Total        int
	Errors       int
	EventsByType map[string]int
}

func (s *Stats) Print() {
	fmt.Printf("총 이벤트: %d (에러: %d)\n", s.Total, s.Errors)
	fmt.Println("이벤트 유형별 분포:")
	for typ, count := range s.EventsByType {
		fmt.Printf("  %-30s %d\n", typ, count)
	}
}
```

이 패턴은 ETL 파이프라인, 로그 분석, 데이터 마이그레이션 등에서 특히 유용합니다. 전체 파일을 메모리에 올리지 않고도, 구조체 단위로 한 건씩 처리하면서 메모리 사용량을 일정하게 유지할 수 있습니다.

## 마이그레이션 전략: 단계별 접근

기존 프로젝트를 json/v2로 전환할 때는 다음과 같은 단계적 접근을 권장합니다:

### 1단계: 브리지 모드로 성능만 먼저 확보

```bash
# CI/CD 파이프라인이나 빌드 스크립트에 추가
export GOEXPERIMENT=jsonv2

# 기존 코드 변경 없이 테스트 실행
go test ./...
```

코드 변경 없이 환경 변수만으로 성능 향상을 얻을 수 있습니다. 이 단계에서 전체 테스트 스위트를 통과하는지 확인합니다.

### 2단계: 새 코드부터 v2 임포트 사용

```go
// 새로 작성하는 파일에서만 v2 직접 사용
import "encoding/json/v2"
```

### 3단계: 성능 크리티컬 경로 먼저 전환

```go
// 가장 많이 호출되는 핸들러부터 v2의 스트리밍 기능 활용
import (
	"encoding/json/jsontext"
	"encoding/json/v2"
)
```

### 4단계: v2 전용 기능 점진적 도입

```go
// intstring, nullable 등 v2 전용 태그 활용
type Order struct {
	Amount int64 `json:"amount,intstring"`
	Note   string `json:"note,nullable"`
}
```

> **주의사항:** json/v2는 일부 엣지 케이스에서 v1과 동작이 다를 수 있습니다. 특히 `MarshalJSON`/`UnmarshalJSON`을 커스텀 구현한 타입이 있다면, 반드시 테스트를 통해 호환성을 확인하세요.

## 서드파티 라이브러리와의 비교

많은 프로젝트가 이미 서드파티 JSON 라이브러리를 사용하고 있습니다. json/v2와 비교하면 어떨까요?

```text
┌─────────────────────┬──────────┬──────────┬──────────────┐
│ 라이브러리           │ 디코딩   │ 인코딩   │ 메모리 할당  │
├─────────────────────┼──────────┼──────────┼──────────────┤
│ encoding/json (v1)  │ 1.0x     │ 1.0x     │ 기준         │
│ encoding/json/v2    │ 1.8-2x   │ 1.7x     │ 75% 감소     │
│ jsoniter            │ 2.0-2.5x │ 1.8x     │ 70% 감소     │
│ go-json             │ 2.2-3x   │ 2.0x     │ 65% 감소     │
│ segmentio/json      │ 2.5-4x   │ 2.2x     │ 60% 감소     │
│ sonic (SIMD 활용)   │ 3-5x     │ 3-4x     │ 50% 감소     │
└─────────────────────┴──────────┴──────────┴──────────────┘
(v1 대비 배수, 높을수록 빠름)
```

절대 성능만 놓고 보면 `sonic`이나 `segmentio/json`이 여전히 빠릅니다. 하지만 json/v2를 선택해야 하는 이유는 명확합니다:

- **외부 의존성 제로**: 표준 라이브러리만으로 충분한 성능
- **장기 지원 보장**: Go 팀이 직접 유지보수
- **호환성 보장**: v1 API와의 브리지 모드 제공
- **향상된 에러 메시지**: 서드파티 라이브러리들도 대부분 제공하지 못하는 기능
- **스트리밍 지원**: `jsontext` 패키지의 일급 스트리밍 API

대부분의 프로덕션 워크로드에서 json/v2의 성능은 충분합니다. 초당 수십만 건의 JSON 처리가 필요한 극한 시나리오가 아니라면, 표준 라이브러리를 선택하는 것이 유지보수와 안정성 측면에서 훨씬 유리합니다.

## 마치며: JSON 처리의 새로운 표준

`encoding/json/v2`는 Go 표준 라이브러리 역사상 가장 의미 있는 업데이트 중 하나입니다. 단순히 "더 빠른 JSON 파서"가 아니라, Go에서 JSON을 다루는 전체 경험을 근본적으로 개선하는 패키지입니다.

에러 메시지 하나만으로도 전환할 가치가 있고, 거기에 1.8~2배의 성능 향상, 75%의 메모리 할당 감소, 일급 스트리밍 지원, 그리고 `intstring`/`nullable` 같은 실용적인 태그 옵션까지 더해집니다. 아직 실험적 단계이지만, Go 팀의 로드맵을 고려하면 정식 포함은 시간 문제입니다.

지금 당장 프로덕션에 적용하기 어렵더라도, 개발/테스트 환경에서 `GOEXPERIMENT=jsonv2`를 설정하고 기존 테스트를 돌려보는 것을 권장합니다. 코드 변경 없이도 성능 향상을 확인할 수 있고, 혹시 호환성 문제가 있다면 미리 파악할 수 있습니다. Go의 JSON 처리는 새로운 시대를 맞이했습니다 — 여러분의 프로젝트도 함께 진화할 준비가 되었나요?
