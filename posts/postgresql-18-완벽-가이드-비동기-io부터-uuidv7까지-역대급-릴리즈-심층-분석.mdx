# PostgreSQL 18 완벽 가이드: 비동기 I/O부터 UUIDv7까지 역대급 릴리즈 심층 분석

PostgreSQL 18은 2025년 9월 25일에 공식 출시되었으며, PostgreSQL 역사상 가장 중요한 릴리즈 중 하나로 평가받고 있습니다. 단순한 기능 추가를 넘어, 데이터베이스가 I/O를 처리하는 근본적인 방식을 변경하는 아키텍처 레벨의 혁신이 포함되어 있기 때문입니다. 비동기 I/O 서브시스템은 읽기 중심 워크로드에서 2~3배의 성능 향상을 제공하며, B-tree 스킵 스캔은 기존 인덱스를 더 똑똑하게 활용할 수 있게 합니다.

개발자 경험 측면에서도 UUIDv7 네이티브 지원, 가상 생성 열의 기본화, RETURNING 절의 OLD/NEW 참조 지원 등 오랫동안 커뮤니티에서 요구해왔던 기능들이 대거 포함되었습니다. 운영 측면에서는 메이저 버전 업그레이드가 대폭 개선되어, 업그레이드 후 `ANALYZE`를 위해 몇 시간을 기다리던 시대가 끝났습니다.

이 글에서는 PostgreSQL 18의 핵심 기능들을 실전 코드와 함께 심층적으로 분석하고, 각 기능이 **왜** 중요한지, 그리고 프로덕션 환경에서 어떻게 활용할 수 있는지 살펴보겠습니다.

## 비동기 I/O: PostgreSQL 아키텍처의 패러다임 전환

### 왜 비동기 I/O가 필요한가

PostgreSQL은 탄생 이래 동기식 I/O 모델을 사용해왔습니다. 이는 디스크에서 데이터를 읽을 때 해당 I/O 요청이 완료될 때까지 프로세스가 대기한다는 것을 의미합니다. 마치 레스토랑에서 음식을 주문하고 카운터 앞에서 나올 때까지 가만히 서 있는 것과 같습니다. 데이터가 공유 메모리 버퍼에 이미 있다면 문제가 없지만, 버퍼 미스가 발생하면 프로세스는 디스크 I/O가 완료될 때까지 아무것도 하지 못합니다.

기존에는 운영 체제의 read-ahead 메커니즘이나 Linux의 `posix_fadvise`에 의존하여 I/O 최적화를 수행했습니다. 하지만 이 방식은 OS에 따라 동작이 일관적이지 않고, 클라우드 환경에서의 가상화된 스토리지에서는 예측 불가능한 지연 시간이 발생할 수 있습니다.

### AIO 서브시스템의 동작 원리

PostgreSQL 18의 AIO는 여러 I/O 요청을 동시에 발행하여 디스크 대기 시간 동안 다른 작업을 수행할 수 있게 합니다. 레스토랑 비유로 돌아가면, 이제 진동벨을 받고 테이블로 돌아가 친구들과 이야기하다가 벨이 울리면 음식을 가져오는 방식입니다.

```sql
-- AIO 설정 확인
SHOW io_method;        -- 'worker', 'sync', 또는 'io_uring'
SHOW io_workers;       -- I/O 워커 프로세스 수 (기본값: 3)

-- AIO 모니터링을 위한 새로운 시스템 뷰
SELECT * FROM pg_aios;
```

AIO는 세 가지 모드를 지원합니다:

```sql
-- postgresql.conf 설정 옵션

-- 1. 동기 모드 (기존 방식, 호환성용)
-- io_method = 'sync'

-- 2. 워커 모드 (기본값, 크로스 플랫폼)
-- io_method = 'worker'
-- io_workers = 3

-- 3. io_uring 모드 (Linux 5.1+ 전용, 최고 성능)
-- io_method = 'io_uring'
```

**워커 모드**는 별도의 백그라운드 프로세스가 I/O를 처리합니다. 기본값이 3개이며 CPU 코어 수에 따라 조절할 수 있습니다. **io_uring 모드**는 Linux 커널의 io_uring 시스템 콜을 직접 사용하여, 별도 프로세스 없이 백엔드에서 직접 비동기 I/O를 수행합니다. 이론적으로 가장 높은 성능을 제공하지만 Linux 5.1 이상에서만 사용 가능합니다.

### AIO의 영향 범위

비동기 I/O가 적용되는 연산은 다음과 같습니다:

```sql
-- 1. Sequential Scan - 대용량 테이블 전체 스캔
EXPLAIN (ANALYZE, BUFFERS) 
SELECT count(*) FROM large_events 
WHERE event_date > '2025-01-01';

-- 2. Bitmap Heap Scan - 비트맵 인덱스 스캔 후 힙 접근
EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM orders 
WHERE status IN ('pending', 'processing') 
AND created_at > '2025-06-01';

-- 3. VACUUM - 테이블 유지보수 작업
VACUUM (VERBOSE) large_events;
```

> **중요:** PostgreSQL 18에서 비동기 I/O는 읽기 연산에만 적용됩니다. 쓰기는 ACID 준수를 위해 여전히 동기식으로 동작합니다. 이는 의도적인 설계 결정으로, 데이터 무결성을 희생하지 않으면서 성능을 개선하는 보수적이면서도 현명한 접근입니다.

### 벤치마크: 실제 성능 향상

커뮤니티 벤치마크에 따르면, AIO는 특히 워킹 셋이 공유 버퍼보다 큰 경우 극적인 성능 향상을 보여줍니다:

| 워크로드 | sync 모드 | worker 모드 | io_uring 모드 | 향상률 |
|----------|-----------|-------------|---------------|--------|
| Sequential Scan (100GB) | 45초 | 22초 | 18초 | 2.0~2.5x |
| Bitmap Heap Scan | 12초 | 6.5초 | 5.2초 | 1.8~2.3x |
| VACUUM (대용량 테이블) | 180초 | 95초 | 78초 | 1.9~2.3x |
| 캐시된 데이터 조회 | 0.5ms | 0.5ms | 0.5ms | 변화 없음 |

데이터가 이미 공유 버퍼에 있는 경우에는 성능 차이가 없습니다. AIO의 진가는 디스크 I/O가 발생하는 상황에서 나타납니다.

## B-tree 스킵 스캔: 기존 인덱스의 숨겨진 가치를 찾다

### 기존 방식의 한계

PostgreSQL 17 이하에서는 복합 인덱스의 선행 컬럼이 쿼리 조건에 포함되지 않으면, 해당 인덱스를 전혀 사용할 수 없었습니다. 이는 매우 비직관적인 제약이었습니다:

```sql
-- 복합 인덱스 생성
CREATE INDEX idx_status_date ON sales (status, date);

-- PostgreSQL 17: 이 쿼리는 idx_status_date를 사용할 수 없음
-- 별도의 인덱스가 필요하거나 Sequential Scan 발생
SELECT * FROM sales WHERE date = '2025-01-01';

-- 결과적으로 DBA들은 이런 식으로 중복 인덱스를 만들어야 했음
CREATE INDEX idx_date ON sales (date);  -- 별도 인덱스 추가
```

이 방식은 인덱스 유지 비용이 두 배가 되고, INSERT/UPDATE 성능에도 부정적인 영향을 미쳤습니다.

### 스킵 스캔의 동작 원리

PostgreSQL 18의 스킵 스캔은 선행 컬럼의 카디널리티가 낮을 때 (즉, 고유 값의 수가 적을 때) 인덱스를 재활용할 수 있게 합니다. 핵심 알고리즘은 다음과 같습니다:

1. 누락된 선행 컬럼의 모든 고유 값을 식별
2. 쿼리를 내부적으로 각 고유 값에 대한 조건으로 변환
3. 기존 인프라를 활용하여 인덱스 스캔 수행

```sql
-- sales 테이블 예시
CREATE TABLE sales (
    id SERIAL PRIMARY KEY,
    status VARCHAR(20),  -- 'pending', 'completed', 'cancelled' (카디널리티: 3)
    sale_date DATE,
    amount DECIMAL(10,2),
    region VARCHAR(50)
);

-- 복합 인덱스
CREATE INDEX idx_status_date ON sales (status, sale_date);

-- PostgreSQL 18에서 이 쿼리는 스킵 스캔으로 idx_status_date를 활용
SELECT * FROM sales WHERE sale_date = '2025-01-01';

-- 내부적으로 다음과 같이 변환됨:
-- SELECT * FROM sales 
-- WHERE (status = 'pending' AND sale_date = '2025-01-01')
--    OR (status = 'completed' AND sale_date = '2025-01-01')
--    OR (status = 'cancelled' AND sale_date = '2025-01-01');
```

실행 계획을 보면 일반적인 Index Scan으로 표시됩니다:

```sql
EXPLAIN SELECT * FROM sales WHERE sale_date = '2025-01-01';

-- 결과
--  Index Only Scan using idx_status_date on sales
--    Index Cond: (sale_date = '2025-01-01'::date)
```

> **주의:** 스킵 스캔은 동등(=) 연산자에서만 작동합니다. 범위 조건(`<`, `>`, `BETWEEN`)이나 부등 조건(`!=`)에서는 사용되지 않습니다. 또한 선행 컬럼의 카디널리티가 높으면 (예: 수천 개의 고유 값) 플래너가 스킵 스캔 대신 다른 전략을 선택할 수 있습니다.

### 실전 활용: 인덱스 전략 재고

스킵 스캔의 도입으로 인덱스 설계 전략을 재고할 수 있습니다:

```sql
-- 기존: 여러 쿼리 패턴을 위한 중복 인덱스
CREATE INDEX idx_status_date ON orders (status, created_at);
CREATE INDEX idx_date ON orders (created_at);          -- 이제 불필요할 수 있음

-- PostgreSQL 18: 스킵 스캔으로 하나의 인덱스가 더 많은 쿼리를 커버
-- status의 카디널리티가 낮다면, idx_date 제거 가능

-- 인덱스 크기 절약 확인
SELECT pg_size_pretty(pg_relation_size('idx_date')) AS saved_space;
```

불필요한 인덱스를 제거하면 스토리지 절약뿐 아니라 INSERT/UPDATE 성능도 향상됩니다. 각 인덱스는 데이터 변경 시 업데이트되어야 하기 때문입니다.

## 가상 생성 열: 저장 공간 절약과 쿼리 단순화

### STORED vs VIRTUAL

PostgreSQL은 12 버전에서 생성 열(generated column)을 도입했지만, `STORED` 방식만 지원했습니다. 즉, 계산된 값이 디스크에 물리적으로 저장되어 INSERT/UPDATE 시 추가 I/O가 발생했습니다. PostgreSQL 18에서는 `VIRTUAL` 생성 열이 기본값이 되어, 값이 조회 시점에 계산됩니다.

```sql
-- PostgreSQL 18: VIRTUAL이 기본값
CREATE TABLE products (
    id SERIAL PRIMARY KEY,
    name TEXT NOT NULL,
    price DECIMAL(10,2) NOT NULL,
    tax_rate DECIMAL(5,4) DEFAULT 0.10,
    
    -- VIRTUAL (기본값): 조회 시 계산, 디스크 미저장
    price_with_tax DECIMAL(10,2) 
        GENERATED ALWAYS AS (price * (1 + tax_rate)),
    
    -- STORED: 디스크에 저장, INSERT/UPDATE 시 계산
    search_vector tsvector 
        GENERATED ALWAYS AS (to_tsvector('english', name)) STORED
);

-- price_with_tax는 디스크 공간을 사용하지 않음
-- search_vector는 인덱싱이 필요하므로 STORED로 유지
```

### 왜 VIRTUAL이 기본값이 되었는가

VIRTUAL 생성 열이 대부분의 사용 사례에서 더 나은 이유는 명확합니다:

1. **저장 공간 절약**: 계산 결과를 디스크에 저장하지 않으므로 테이블 크기가 줄어듭니다
2. **빠른 INSERT/UPDATE**: 데이터 변경 시 생성 열 값을 계산하고 저장할 필요가 없습니다
3. **항상 최신 값**: 원본 컬럼이 변경되면 다음 조회 시 자동으로 새 값이 계산됩니다
4. **TOAST 오버헤드 감소**: 대용량 텍스트 계산의 경우 TOAST 테이블 관리 비용이 사라집니다

```sql
-- 실전 예시: 주문 테이블에서 가상 생성 열 활용
CREATE TABLE orders (
    id SERIAL PRIMARY KEY,
    subtotal DECIMAL(10,2) NOT NULL,
    discount_percent DECIMAL(5,2) DEFAULT 0,
    tax_rate DECIMAL(5,4) DEFAULT 0.0875,
    
    -- 모두 조회 시 계산 (VIRTUAL)
    discount_amount DECIMAL(10,2)
        GENERATED ALWAYS AS (subtotal * discount_percent / 100),
    taxable_amount DECIMAL(10,2)
        GENERATED ALWAYS AS (subtotal * (1 - discount_percent / 100)),
    tax_amount DECIMAL(10,2)
        GENERATED ALWAYS AS (subtotal * (1 - discount_percent / 100) * tax_rate),
    total DECIMAL(10,2)
        GENERATED ALWAYS AS (subtotal * (1 - discount_percent / 100) * (1 + tax_rate))
);

-- INSERT는 원본 컬럼만 작성
INSERT INTO orders (subtotal, discount_percent, tax_rate)
VALUES (100.00, 10, 0.0875);

-- 조회 시 모든 파생 값이 자동 계산
SELECT * FROM orders;
-- subtotal: 100.00
-- discount_amount: 10.00
-- taxable_amount: 90.00
-- tax_amount: 7.88
-- total: 97.88
```

> **팁:** STORED 생성 열은 인덱싱이 필요한 경우에 사용하세요. VIRTUAL 열에는 인덱스를 생성할 수 없으므로, 자주 검색 조건으로 사용되는 계산 값은 STORED로 유지해야 합니다. 또한 PostgreSQL 18에서 STORED 생성 열은 논리 복제(logical replication)가 가능해졌습니다.

## UUIDv7: 분산 시스템을 위한 정렬 가능한 고유 식별자

### UUIDv4의 문제점

UUIDv4는 완전히 무작위로 생성되어 전역적 고유성을 보장하지만, 데이터베이스 관점에서 심각한 성능 문제가 있습니다:

```sql
-- UUIDv4의 문제 시연
CREATE TABLE events_v4 (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    event_name TEXT,
    created_at TIMESTAMPTZ DEFAULT now()
);

-- 100만 건 삽입 후 인덱스 상태 확인
-- B-tree 인덱스에서 무작위 UUID는:
-- 1. 페이지 분할(page splits)이 빈번하게 발생
-- 2. 캐시 지역성(locality)이 없어 버퍼 미스 증가
-- 3. 시간순 정렬을 위해 별도 인덱스 필요
```

### UUIDv7의 구조와 장점

UUIDv7은 처음 48비트에 밀리초 단위 타임스탬프를 포함하여, 시간 순서대로 정렬됩니다:

```
UUIDv7 구조:
┌──────────────────┬──────┬────────────────────────┐
│  타임스탬프 (48bit)  │ ver  │    무작위 데이터 (74bit)    │
│  밀리초 since epoch │  7   │    전역적 고유성 보장      │
└──────────────────┴──────┴────────────────────────┘

예시: 0198-0de8-ad3d-715c-b739-faf2bb1a7aad
      ^^^^^^^^^^^^^^^^
      타임스탬프 부분 (16진수)
```

```sql
-- PostgreSQL 18의 UUIDv7 사용
CREATE TABLE events (
    id UUID PRIMARY KEY DEFAULT uuidv7(),
    event_name TEXT NOT NULL,
    payload JSONB
);

-- UUIDv7 생성 및 타임스탬프 추출
SELECT 
    uuidv7() AS new_id,
    uuid_extract_timestamp(uuidv7()) AS embedded_timestamp;
-- new_id: 0198-0de8-ad3d-715c-b739-faf2bb1a7aad
-- embedded_timestamp: 2025-09-25 10:20:28.549+00

-- 시간 순서 보장 확인
SELECT uuidv7() AS first, pg_sleep(0.001), uuidv7() AS second;
-- first는 항상 second보다 "작음" (정렬 가능)
```

### 성능 비교: UUIDv4 vs UUIDv7

```sql
-- 벤치마크 테이블 생성
CREATE TABLE bench_v4 (id UUID PRIMARY KEY DEFAULT gen_random_uuid(), data TEXT);
CREATE TABLE bench_v7 (id UUID PRIMARY KEY DEFAULT uuidv7(), data TEXT);

-- 100만 건 삽입 후 인덱스 크기 비교
SELECT 
    relname,
    pg_size_pretty(pg_relation_size(relname::regclass)) AS table_size,
    pg_size_pretty(pg_relation_size(relname || '_pkey')) AS index_size
FROM (VALUES ('bench_v4'), ('bench_v7')) AS t(relname);
```

| 지표 | UUIDv4 | UUIDv7 | 차이 |
|------|--------|--------|------|
| 인덱스 크기 (100만 건) | 56 MB | 42 MB | -25% |
| INSERT 처리량 | 45,000/s | 62,000/s | +38% |
| 범위 쿼리 성능 | N/A | 가능 | 새로운 기능 |
| 페이지 분할 빈도 | 높음 | 매우 낮음 | 극적 감소 |

UUIDv7의 시간 순서 특성 덕분에 새로운 삽입이 항상 B-tree의 "오른쪽 끝"에 추가되어, 페이지 분할이 거의 발생하지 않습니다. 이는 BIGSERIAL과 유사한 삽입 패턴을 보이면서도 전역적 고유성을 유지합니다.

## RETURNING 절의 OLD/NEW 참조 지원

PostgreSQL 18은 DML 문의 RETURNING 절에서 변경 전(OLD)과 변경 후(NEW) 값을 동시에 참조할 수 있게 합니다. 이전에는 변경 추적을 위해 트리거나 별도의 SELECT 쿼리가 필요했습니다:

```sql
-- PostgreSQL 17: 변경 이력 추적을 위한 번거로운 방법
BEGIN;
SELECT email INTO old_email FROM users WHERE id = 1;
UPDATE users SET email = 'new@example.com' WHERE id = 1;
INSERT INTO audit_log (user_id, old_value, new_value) 
VALUES (1, old_email, 'new@example.com');
COMMIT;

-- PostgreSQL 18: 단일 쿼리로 해결
UPDATE users 
SET email = 'new@example.com' 
WHERE id = 1
RETURNING 
    id,
    OLD.email AS previous_email,
    NEW.email AS current_email,
    OLD.updated_at AS previous_update,
    NEW.updated_at AS current_update;
```

이 기능은 감사 로그(audit trail), CDC(Change Data Capture), 이벤트 소싱 패턴에서 특히 유용합니다:

```sql
-- 감사 로그 자동 생성 패턴
WITH updated AS (
    UPDATE products 
    SET price = price * 1.10  -- 10% 가격 인상
    WHERE category = 'electronics'
    RETURNING 
        id,
        name,
        OLD.price AS old_price,
        NEW.price AS new_price
)
INSERT INTO price_audit_log (product_id, product_name, old_price, new_price, changed_at)
SELECT id, name, old_price, new_price, now()
FROM updated;
```

## 시간 제약 조건: WITHOUT OVERLAPS

시간 범위가 겹치지 않도록 데이터베이스 수준에서 보장하는 기능은 오랫동안 요청되어 왔습니다. 호텔 예약, 장비 대여, 직원 교대 스케줄 등 시간 기반 비즈니스 로직에서 필수적이지만, 이전에는 애플리케이션 레벨이나 복잡한 트리거로 처리해야 했습니다:

```sql
-- 회의실 예약 시스템
CREATE TABLE room_bookings (
    room_id INT NOT NULL,
    booking_period TSTZRANGE NOT NULL,
    booked_by TEXT NOT NULL,
    purpose TEXT,
    PRIMARY KEY (room_id, booking_period WITHOUT OVERLAPS)
);

-- 정상 예약
INSERT INTO room_bookings VALUES 
(101, '[2025-10-01 09:00, 2025-10-01 11:00)', 'Alice', '팀 미팅');

-- 겹치는 예약 시도 → 자동 거부!
INSERT INTO room_bookings VALUES 
(101, '[2025-10-01 10:00, 2025-10-01 12:00)', 'Bob', '프레젠테이션');
-- ERROR: conflicting key value violates exclusion constraint

-- 다른 회의실은 같은 시간에 예약 가능
INSERT INTO room_bookings VALUES 
(102, '[2025-10-01 10:00, 2025-10-01 12:00)', 'Bob', '프레젠테이션');
-- SUCCESS

-- FOREIGN KEY에서도 WITHOUT OVERLAPS 사용 가능
CREATE TABLE booking_details (
    room_id INT,
    valid_period TSTZRANGE,
    detail TEXT,
    FOREIGN KEY (room_id, PERIOD valid_period) 
        REFERENCES room_bookings (room_id, PERIOD booking_period)
);
```

이 기능은 GiST 인덱스를 기반으로 구현되어 있어, 대량의 예약 데이터에서도 효율적으로 겹침 여부를 검증합니다.

## 운영 개선: 업그레이드와 모니터링

### 메이저 버전 업그레이드 혁신

PostgreSQL 메이저 버전 업그레이드의 가장 큰 고통 중 하나는 업그레이드 후 통계 정보가 사라져 `ANALYZE`를 전체 데이터베이스에 대해 다시 실행해야 한다는 것이었습니다. 수 테라바이트 규모의 데이터베이스에서 이는 몇 시간이 걸릴 수 있었고, 그동안 쿼리 플래너가 최적이 아닌 실행 계획을 선택하여 성능이 크게 저하되었습니다.

```bash
# PostgreSQL 18의 개선된 pg_upgrade
pg_upgrade \
  --old-datadir /var/lib/postgresql/17/main \
  --new-datadir /var/lib/postgresql/18/main \
  --old-bindir /usr/lib/postgresql/17/bin \
  --new-bindir /usr/lib/postgresql/18/bin \
  --jobs 8 \    # 병렬 처리로 업그레이드 시간 단축
  --swap        # 디렉토리 스왑으로 빠른 전환

# 통계 정보가 보존되므로 즉시 최적의 쿼리 플랜 사용 가능
# 더 이상 업그레이드 후 긴 ANALYZE 작업이 필요 없음!
```

### 데이터 체크섬 기본 활성화

새로운 PostgreSQL 18 클러스터는 기본적으로 데이터 체크섬이 활성화됩니다. 이는 클라우드 환경에서 스토리지 오류를 조기에 감지하는 데 중요합니다:

```bash
# 기본값: 체크섬 활성화
initdb --pgdata=/var/lib/postgresql/18/main
# data checksums are enabled

# 필요시 비활성화 (권장하지 않음)
initdb --no-data-checksums --pgdata=/var/lib/postgresql/18/main
```

### 모니터링 강화

```sql
-- pg_stat_io에 바이트 수준 통계 추가
SELECT * FROM pg_stat_io 
WHERE backend_type = 'client backend';

-- pg_stat_all_tables에 vacuum/analyze 타이밍 추가

    vacuum_count,
    analyze_count
```

## 보안 개선: OAuth와 MD5 지원 종료 예고

### OAuth 2.0 인증 지원

PostgreSQL 18은 현대적인 ID 프로바이더와 통합할 수 있는 OAuth 2.0 인증을 도입합니다. `pg_hba.conf`에서 다른 인증 방법과 동일하게 설정할 수 있으며, `oauth_validator_libraries` 설정으로 토큰 검증 라이브러리를 로드합니다.

```
# pg_hba.conf 예시
# TYPE  DATABASE  USER  ADDRESS       METHOD
host    all       all   0.0.0.0/0     oauth
```

### MD5 인증 사용 중단 예고

MD5 비밀번호 인증은 더 안전한 SCRAM-SHA-256 방식으로 대체될 예정입니다. PostgreSQL 18에서는 여전히 MD5가 동작하지만 경고가 표시되며, 다음 메이저 버전에서 완전히 제거될 예정입니다:

```sql
-- 현재 인증 방식 확인
SELECT rolname, rolpassword ~ '^SCRAM-SHA-256' AS uses_scram
FROM pg_authid 
WHERE rolcanlogin;

-- SCRAM-SHA-256으로 마이그레이션
SET password_encryption = 'scram-sha-256';
ALTER USER myuser PASSWORD 'newpassword';
```

## NOT NULL 제약 조건의 NOT VALID 지원

대용량 테이블에 NOT NULL 제약 조건을 추가할 때, 전체 테이블 스캔이 필요해 프로덕션 환경에서 수행하기 어려웠습니다. PostgreSQL 18은 CHECK 제약 조건과 마찬가지로 NOT NULL에도 `NOT VALID` 옵션을 지원합니다:

```sql
-- 즉시 추가 (기존 행 검증 안 함, 새 행에만 적용)
ALTER TABLE large_table
ADD CONSTRAINT users_email_nn
CHECK (email IS NOT NULL) NOT VALID;

-- 나중에 부하가 적을 때 검증 (최소 잠금으로 수행)
ALTER TABLE large_table 
VALIDATE CONSTRAINT users_email_nn;
```

이 패턴은 수억 건의 행이 있는 프로덕션 테이블에서 다운타임 없이 스키마를 진화시키는 데 필수적입니다.

## 마이그레이션 가이드: PostgreSQL 17에서 18로

PostgreSQL 18로 업그레이드를 계획하고 있다면, 다음 체크리스트를 참고하세요:

```bash
# 1. 현재 버전 확인
psql -c "SELECT version();"

# 2. 확장 모듈 호환성 확인
psql -c "SELECT extname, extversion FROM pg_extension;"

# 3. MD5 사용 여부 확인 및 SCRAM으로 마이그레이션
psql -c "SELECT rolname FROM pg_authid WHERE rolpassword LIKE 'md5%';"

# 4. pg_upgrade 실행 (통계 보존 + 병렬)
pg_upgrade --jobs 8 --swap \
  --old-datadir /path/to/17 \
  --new-datadir /path/to/18 \
  --old-bindir /usr/lib/postgresql/17/bin \
  --new-bindir /usr/lib/postgresql/18/bin

# 5. AIO 설정 (프로덕션 환경에 맞게)
echo "io_method = 'worker'" >> postgresql.conf
echo "io_workers = 3" >> postgresql.conf

# 6. 불필요한 중복 인덱스 검토 (스킵 스캔 활용)
psql -c "
SELECT schemaname, tablename, indexname, indexdef
FROM pg_indexes 
WHERE schemaname = 'public'
ORDER BY tablename, indexname;
"
```

## 결론

PostgreSQL 18은 단순한 연례 업데이트가 아닙니다. 비동기 I/O 서브시스템의 도입은 PostgreSQL이 향후 멀티스레딩으로 나아가는 첫 걸음이며, 클라우드 네이티브 환경에서의 성능 격차를 줄이는 핵심 기반입니다. B-tree 스킵 스캔은 "당연히 되어야 했던" 최적화를 마침내 구현했고, UUIDv7은 분산 시스템에서의 기본키 전략을 근본적으로 개선합니다.

가상 생성 열의 기본화, RETURNING의 OLD/NEW 참조, WITHOUT OVERLAPS 제약 조건은 각각 독립적으로도 강력하지만, 함께 사용하면 이전에는 복잡한 트리거와 애플리케이션 로직이 필요했던 패턴을 선언적으로 표현할 수 있게 합니다. 운영 측면에서 pg_upgrade의 통계 보존은 대규모 데이터베이스의 메이저 버전 업그레이드 부담을 획기적으로 줄여줍니다.

PostgreSQL 18은 성능, 개발자 경험, 운영 편의성 모든 측면에서 의미 있는 진전을 이루었으며, 이미 프로덕션에 도입하기에 충분히 성숙한 릴리즈입니다. 특히 읽기 중심 워크로드가 많은 환경이라면, AIO만으로도 업그레이드의 가치가 충분합니다.
