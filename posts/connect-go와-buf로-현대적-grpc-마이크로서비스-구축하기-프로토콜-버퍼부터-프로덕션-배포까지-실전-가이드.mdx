# Connect-Go와 Buf로 현대적 gRPC 마이크로서비스 구축하기

## 소개: gRPC의 진화가 필요한 이유

gRPC는 Google이 만든 고성능 RPC 프레임워크로, 마이크로서비스 아키텍처의 사실상 표준이 되었습니다. Protocol Buffers를 인터페이스 정의 언어(IDL)로 사용하고, HTTP/2 기반의 효율적인 바이너리 프로토콜을 통해 서비스 간 통신을 처리합니다. Netflix, Uber, Stripe 같은 기업들이 대규모 분산 시스템에서 gRPC를 핵심 통신 레이어로 활용하고 있습니다.

하지만 전통적인 gRPC 개발에는 상당한 마찰이 존재합니다. `protoc` 컴파일러와 수많은 플러그인을 설치하고 관리하는 것은 악몽에 가깝습니다. 언어별로 다른 플러그인 버전, 복잡한 빌드 스크립트, 그리고 팀원마다 다른 로컬 환경은 "내 컴퓨터에서는 되는데"라는 문제를 끊임없이 만들어냅니다. 더 큰 문제는 gRPC가 HTTP/2를 강제하기 때문에 브라우저에서 직접 호출할 수 없고, gRPC-Web이라는 별도의 프록시 레이어가 필요하다는 점입니다.

이런 고통을 정면으로 해결하려는 두 가지 프로젝트가 있습니다. **Buf**는 프로토콜 버퍼 도구 체인을 현대화하여 린팅, 브레이킹 체인지 감지, 코드 생성을 하나의 일관된 워크플로우로 통합합니다. **Connect-Go**는 Buf 팀이 만든 Go용 RPC 프레임워크로, 하나의 서버에서 gRPC, gRPC-Web, 그리고 새로운 Connect 프로토콜을 동시에 지원합니다. Connect 프로토콜은 HTTP/1.1에서도 동작하므로 `curl`로 직접 API를 호출할 수 있습니다.

이 가이드에서는 Buf와 Connect-Go를 활용하여 처음부터 프로덕션 수준의 마이크로서비스를 구축하는 전체 과정을 다룹니다. 프로토콜 버퍼 스키마 설계, 코드 생성 자동화, 인터셉터 기반의 미들웨어 패턴, 스트리밍 RPC, 에러 처리, 그리고 관측성(observability)까지 실전에서 바로 적용할 수 있는 지식을 전달합니다.

## Buf로 프로토콜 버퍼 워크플로우 현대화하기

### protoc의 한계를 넘어서

전통적인 프로토콜 버퍼 워크플로우에서 `protoc`는 모든 것의 중심입니다. 하지만 `protoc`에는 근본적인 문제가 있습니다. 플러그인을 별도로 설치해야 하고, 각 플러그인의 버전 호환성을 직접 관리해야 하며, `.proto` 파일 간의 임포트 경로를 수동으로 설정해야 합니다. 대규모 프로젝트에서 수십 개의 `.proto` 파일을 관리하다 보면, Makefile이 수백 줄에 달하는 것은 흔한 일입니다.

Buf는 이 모든 복잡성을 `buf.yaml` 하나로 추상화합니다. 모듈 시스템을 통해 외부 프로토 파일을 의존성으로 관리하고, 원격 플러그인 실행으로 로컬 플러그인 설치가 불필요합니다. 무엇보다 린팅과 브레이킹 체인지 감지를 기본 제공하여 API 스키마의 품질을 자동으로 보장합니다.

### 프로젝트 초기 설정

먼저 Buf CLI를 설치하고 프로젝트를 초기화합니다:

```bash
# Buf CLI 설치
go install github.com/bufbuild/buf/cmd/buf@latest

# 프로젝트 구조 생성
mkdir -p order-service/{proto/order/v1,cmd/server,internal}
cd order-service
go mod init github.com/example/order-service
```

프로토 파일을 위한 Buf 모듈을 설정합니다:

```yaml
# proto/buf.yaml
version: v2
modules:
  - path: .
    name: buf.build/example/order-service
lint:
  use:
    - STANDARD
  enum_zero_value_suffix: _UNSPECIFIED
breaking:
  use:
    - FILE
deps:
  - buf.build/googleapis/googleapis
```

이 설정에서 주목할 부분은 `lint` 섹션입니다. `STANDARD` 규칙 세트는 Google의 API 설계 가이드라인을 기반으로 한 수십 개의 린트 규칙을 포함합니다. `enum_zero_value_suffix`는 enum의 0번 값이 반드시 `_UNSPECIFIED`로 끝나도록 강제하는데, 이는 프로토콜 버퍼의 기본값(zero value) 특성상 "값이 설정되지 않음"을 명시적으로 표현하기 위한 모범 사례입니다.

`breaking` 섹션의 `FILE` 규칙은 같은 파일 내에서의 브레이킹 체인지를 감지합니다. 필드 번호 변경, 타입 변경, 필드 삭제 등을 자동으로 잡아내어 API 호환성을 보장합니다.

### 서비스 스키마 설계

주문 관리 서비스의 프로토 파일을 설계합니다:

```protobuf
// proto/order/v1/order.proto
syntax = "proto3";

package order.v1;

import "google/protobuf/timestamp.proto";
import "google/protobuf/field_mask.proto";

// 주문 상태를 나타내는 열거형
enum OrderStatus {
  ORDER_STATUS_UNSPECIFIED = 0;
  ORDER_STATUS_PENDING = 1;
  ORDER_STATUS_CONFIRMED = 2;
  ORDER_STATUS_SHIPPING = 3;
  ORDER_STATUS_DELIVERED = 4;
  ORDER_STATUS_CANCELLED = 5;
}

// 주문 항목
message OrderItem {
  string product_id = 1;
  string product_name = 2;
  int32 quantity = 3;
  int64 price_cents = 4; // 센트 단위 가격 (부동소수점 회피)
}

// 주문 엔티티
message Order {
  string order_id = 1;
  string customer_id = 2;
  repeated OrderItem items = 3;
  OrderStatus status = 4;
  int64 total_cents = 5;
  google.protobuf.Timestamp created_at = 6;
  google.protobuf.Timestamp updated_at = 7;
  map<string, string> metadata = 8;
}
```

가격을 `int64` 센트 단위로 표현하는 것은 금융 도메인의 핵심 패턴입니다. `float`나 `double`은 부동소수점 정밀도 문제로 인해 금액 계산에서 미세한 오차가 발생할 수 있습니다. 10,000원은 `1000000` 센트(또는 원의 1/100)로 표현하여 정확한 산술 연산을 보장합니다.

이제 서비스 정의를 추가합니다:

```protobuf
// proto/order/v1/order_service.proto
syntax = "proto3";

package order.v1;

import "order/v1/order.proto";
import "google/protobuf/field_mask.proto";

// 주문 서비스 인터페이스
service OrderService {
  // 단항 RPC: 주문 생성
  rpc CreateOrder(CreateOrderRequest) returns (CreateOrderResponse);
  // 단항 RPC: 주문 조회
  rpc GetOrder(GetOrderRequest) returns (GetOrderResponse);
  // 단항 RPC: 주문 상태 업데이트
  rpc UpdateOrderStatus(UpdateOrderStatusRequest) returns (UpdateOrderStatusResponse);
  // 서버 스트리밍: 주문 상태 실시간 추적
  rpc WatchOrderStatus(WatchOrderStatusRequest) returns (stream WatchOrderStatusResponse);
  // 클라이언트 스트리밍: 대량 주문 일괄 생성
  rpc BatchCreateOrders(stream BatchCreateOrdersRequest) returns (BatchCreateOrdersResponse);
}

message CreateOrderRequest {
  string customer_id = 1;
  repeated OrderItem items = 2;
  map<string, string> metadata = 3;
}

message CreateOrderResponse {
  Order order = 1;
}

message GetOrderRequest {
  string order_id = 1;
  // 필드 마스크: 클라이언트가 필요한 필드만 요청
  google.protobuf.FieldMask field_mask = 2;
}

message GetOrderResponse {
  Order order = 1;
}

message UpdateOrderStatusRequest {
  string order_id = 1;
  OrderStatus status = 2;
}

message UpdateOrderStatusResponse {
  Order order = 1;
}

message WatchOrderStatusRequest {
  string order_id = 1;
}

message WatchOrderStatusResponse {
  string order_id = 1;
  OrderStatus previous_status = 2;
  OrderStatus current_status = 3;
  google.protobuf.Timestamp changed_at = 4;
}

message BatchCreateOrdersRequest {
  CreateOrderRequest order = 1;
}

message BatchCreateOrdersResponse {
  repeated Order orders = 1;
  int32 success_count = 2;
  int32 failure_count = 3;
}
```

`FieldMask`를 사용하는 것은 Google API 설계 가이드라인의 핵심 패턴입니다. 클라이언트가 `field_mask: "order_id,status"`를 보내면 서버는 해당 필드만 채워서 응답합니다. 이는 대역폭을 절약하고, 특히 큰 메시지를 다루는 모바일 클라이언트에서 체감 성능을 크게 향상시킵니다.

### Buf를 이용한 코드 생성

코드 생성 설정을 구성합니다:

```yaml
# proto/buf.gen.yaml
version: v2
plugins:
  - remote: buf.build/protocolbuffers/go
    out: ../gen
    opt: paths=source_relative
  - remote: buf.build/connectrpc/go
    out: ../gen
    opt: paths=source_relative
managed:
  enabled: true
  override:
    - file_option: go_package_prefix
      value: github.com/example/order-service/gen
```

```bash
# 코드 생성 실행
cd proto
buf generate

# 생성 결과 확인
# gen/order/v1/order.pb.go          - 메시지 타입
# gen/order/v1/order_service.pb.go  - 서비스 디스크립터
# gen/order/v1/orderv1connect/      - Connect-Go 서비스 핸들러/클라이언트
```

`managed` 옵션은 Buf의 강력한 기능 중 하나입니다. `.proto` 파일에 `option go_package`를 하드코딩하지 않아도 Buf가 자동으로 Go 패키지 경로를 생성합니다. 이는 프로토 파일의 이식성을 높이고, 언어별 설정이 스키마를 오염시키는 것을 방지합니다.

## Connect-Go로 서비스 구현하기

### Connect-Go가 기존 grpc-go와 다른 점

Connect-Go는 `google.golang.org/grpc`를 완전히 대체하는 것을 목표로 합니다. 핵심 차이점은 세 가지입니다. 첫째, Connect-Go는 표준 `net/http` 라이브러리 위에 구축됩니다. 별도의 gRPC 서버가 아닌 일반 HTTP 서버에 핸들러를 등록하므로, 기존 HTTP 미들웨어(CORS, 인증, 로깅 등)를 그대로 재사용할 수 있습니다. 둘째, 하나의 핸들러가 gRPC, gRPC-Web, Connect 프로토콜을 모두 처리합니다. 별도의 프록시 없이 브라우저에서 직접 API를 호출할 수 있습니다. 셋째, Connect 프로토콜은 HTTP/1.1에서도 동작하므로 `curl`이나 일반 HTTP 클라이언트로 테스트할 수 있습니다.

> **왜 이것이 중요한가?** 기존 grpc-go를 사용하면 gRPC 전용 서버와 HTTP 서버를 별도로 운영하거나, grpc-gateway로 REST 프록시를 추가해야 했습니다. Connect-Go는 하나의 서버, 하나의 포트에서 모든 프로토콜을 처리하므로 운영 복잡성이 극적으로 줄어듭니다.

### 서비스 핸들러 구현

생성된 인터페이스를 구현합니다:

```go
// internal/service/order_service.go
package service

import (
	"context"
	"fmt"
	"sync"
	"time"

	"connectrpc.com/connect"
	orderv1 "github.com/example/order-service/gen/order/v1"
	"github.com/example/order-service/gen/order/v1/orderv1connect"
	"github.com/google/uuid"
	"google.golang.org/protobuf/types/known/timestamppb"
)

// OrderService는 주문 서비스의 비즈니스 로직을 구현합니다.
type OrderService struct {
	orderv1connect.UnimplementedOrderServiceHandler

	mu       sync.RWMutex
	orders   map[string]*orderv1.Order
	watchers map[string][]chan *orderv1.WatchOrderStatusResponse
}

func NewOrderService() *OrderService {
	return &OrderService{
		orders:   make(map[string]*orderv1.Order),
		watchers: make(map[string][]chan *orderv1.WatchOrderStatusResponse),
	}
}

func (s *OrderService) CreateOrder(
	ctx context.Context,
	req *connect.Request[orderv1.CreateOrderRequest],
) (*connect.Response[orderv1.CreateOrderResponse], error) {
	msg := req.Msg

	// 입력 검증
	if msg.CustomerId == "" {
		return nil, connect.NewError(
			connect.CodeInvalidArgument,
			fmt.Errorf("customer_id는 필수입니다"),
		)
	}
	if len(msg.Items) == 0 {
		return nil, connect.NewError(
			connect.CodeInvalidArgument,
			fmt.Errorf("최소 하나의 주문 항목이 필요합니다"),
		)
	}

	// 총 금액 계산
	var totalCents int64
	for _, item := range msg.Items {
		if item.Quantity <= 0 {
			return nil, connect.NewError(
				connect.CodeInvalidArgument,
				fmt.Errorf("상품 %s의 수량은 양수여야 합니다", item.ProductId),
			)
		}
		totalCents += item.PriceCents * int64(item.Quantity)
	}

	now := timestamppb.Now()
	order := &orderv1.Order{
		OrderId:    uuid.New().String(),
		CustomerId: msg.CustomerId,
		Items:      msg.Items,
		Status:     orderv1.OrderStatus_ORDER_STATUS_PENDING,
		TotalCents: totalCents,
		CreatedAt:  now,
		UpdatedAt:  now,
		Metadata:   msg.Metadata,
	}

	s.mu.Lock()
	s.orders[order.OrderId] = order
	s.mu.Unlock()

	res := connect.NewResponse(&orderv1.CreateOrderResponse{
		Order: order,
	})
	// 응답 헤더에 주문 ID 포함 (디버깅용)
	res.Header().Set("X-Order-Id", order.OrderId)
	return res, nil
}

func (s *OrderService) GetOrder(
	ctx context.Context,
	req *connect.Request[orderv1.GetOrderRequest],
) (*connect.Response[orderv1.GetOrderResponse], error) {
	s.mu.RLock()
	order, ok := s.orders[req.Msg.OrderId]
	s.mu.RUnlock()

	if !ok {
		return nil, connect.NewError(
			connect.CodeNotFound,
			fmt.Errorf("주문 %s를 찾을 수 없습니다", req.Msg.OrderId),
		)
	}

	return connect.NewResponse(&orderv1.GetOrderResponse{
		Order: order,
	}), nil
}
```

Connect-Go의 `connect.Request[T]`와 `connect.Response[T]` 제네릭 타입에 주목하세요. 이 래퍼 타입들은 메시지 본문뿐 아니라 HTTP 헤더와 트레일러에도 접근할 수 있게 해줍니다. 기존 grpc-go에서는 `metadata` 패키지를 별도로 사용해야 했지만, Connect-Go에서는 표준 HTTP 헤더 API를 그대로 사용합니다.

### 서버 스트리밍 구현

주문 상태를 실시간으로 추적하는 스트리밍 RPC를 구현합니다:

```go
func (s *OrderService) WatchOrderStatus(
	ctx context.Context,
	req *connect.Request[orderv1.WatchOrderStatusRequest],
	stream *connect.ServerStream[orderv1.WatchOrderStatusResponse],
) error {
	orderID := req.Msg.OrderId

	// 주문 존재 여부 확인
	s.mu.RLock()
	_, ok := s.orders[orderID]
	s.mu.RUnlock()
	if !ok {
		return connect.NewError(
			connect.CodeNotFound,
			fmt.Errorf("주문 %s를 찾을 수 없습니다", orderID),
		)
	}

	// 워처 채널 등록
	ch := make(chan *orderv1.WatchOrderStatusResponse, 16)
	s.mu.Lock()
	s.watchers[orderID] = append(s.watchers[orderID], ch)
	s.mu.Unlock()

	// 정리 함수: 연결 종료 시 워처 제거
	defer func() {
		s.mu.Lock()
		watchers := s.watchers[orderID]
		for i, w := range watchers {
			if w == ch {
				s.watchers[orderID] = append(watchers[:i], watchers[i+1:]...)
				break
			}
		}
		s.mu.Unlock()
		close(ch)
	}()

	// 스트리밍 루프
	for {
		select {
		case <-ctx.Done():
			return ctx.Err()
		case update, ok := <-ch:
			if !ok {
				return nil
			}
			if err := stream.Send(update); err != nil {
				return err
			}
		}
	}
}

// UpdateOrderStatus는 상태를 업데이트하고 워처들에게 알림을 보냅니다.
func (s *OrderService) UpdateOrderStatus(
	ctx context.Context,
	req *connect.Request[orderv1.UpdateOrderStatusRequest],
) (*connect.Response[orderv1.UpdateOrderStatusResponse], error) {
	s.mu.Lock()
	order, ok := s.orders[req.Msg.OrderId]
	if !ok {
		s.mu.Unlock()
		return nil, connect.NewError(
			connect.CodeNotFound,
			fmt.Errorf("주문 %s를 찾을 수 없습니다", req.Msg.OrderId),
		)
	}

	previousStatus := order.Status
	order.Status = req.Msg.Status
	order.UpdatedAt = timestamppb.Now()

	// 워처에게 알림
	watchers := s.watchers[req.Msg.OrderId]
	s.mu.Unlock()

	update := &orderv1.WatchOrderStatusResponse{
		OrderId:        req.Msg.OrderId,
		PreviousStatus: previousStatus,
		CurrentStatus:  req.Msg.Status,
		ChangedAt:      timestamppb.Now(),
	}
	for _, ch := range watchers {
		select {
		case ch <- update:
		default:
			// 채널이 가득 찬 경우 건너뛰기 (백프레셔)
		}
	}

	return connect.NewResponse(&orderv1.UpdateOrderStatusResponse{
		Order: order,
	}), nil
}
```

백프레셔(back-pressure) 처리에 주목하세요. 버퍼 크기가 16인 채널을 사용하고, 채널이 가득 차면 `default` 케이스로 업데이트를 드롭합니다. 프로덕션에서는 이 부분을 더 정교하게 처리해야 합니다. 예를 들어 드롭 횟수를 메트릭으로 기록하거나, 느린 클라이언트를 감지하여 연결을 종료할 수 있습니다.

## 인터셉터로 횡단 관심사 처리하기

### Connect-Go의 인터셉터 패턴

인터셉터는 gRPC의 미들웨어에 해당합니다. 인증, 로깅, 메트릭 수집, 에러 변환 등 모든 RPC에 공통으로 적용되는 로직을 인터셉터로 분리합니다. Connect-Go의 인터셉터는 단항 RPC와 스트리밍 RPC를 모두 하나의 인터페이스로 처리할 수 있어 기존 grpc-go의 분리된 인터셉터 인터페이스보다 훨씬 간결합니다.

```go
// internal/interceptor/logging.go
package interceptor

import (
	"context"
	"log/slog"
	"time"

	"connectrpc.com/connect"
)

// NewLoggingInterceptor는 모든 RPC 호출을 구조화된 로그로 기록합니다.
func NewLoggingInterceptor(logger *slog.Logger) connect.UnaryInterceptorFunc {
	return func(next connect.UnaryFunc) connect.UnaryFunc {
		return func(ctx context.Context, req connect.AnyRequest) (connect.AnyResponse, error) {
			start := time.Now()

			// 요청 로깅
			logger.InfoContext(ctx, "rpc_request",
				slog.String("procedure", req.Spec().Procedure),
				slog.String("protocol", req.Peer().Protocol),
				slog.String("addr", req.Peer().Addr),
			)

			// 다음 핸들러 호출
			res, err := next(ctx, req)
			duration := time.Since(start)

			// 응답 로깅
			attrs := []any{
				slog.String("procedure", req.Spec().Procedure),
				slog.Duration("duration", duration),
			}
			if err != nil {
				attrs = append(attrs,
					slog.String("error", err.Error()),
					slog.String("code", connect.CodeOf(err).String()),
				)
				logger.ErrorContext(ctx, "rpc_error", attrs...)
			} else {
				logger.InfoContext(ctx, "rpc_response", attrs...)
			}

			return res, err
		}
	}
}
```

### 인증 인터셉터

JWT 기반 인증 인터셉터를 구현합니다:

```go
// internal/interceptor/auth.go
package interceptor

import (
	"context"
	"fmt"
	"strings"

	"connectrpc.com/connect"
	"github.com/golang-jwt/jwt/v5"
)

type contextKey string

const userIDKey contextKey = "user_id"

// 인증이 필요 없는 프로시저 목록
var publicProcedures = map[string]bool{
	"/order.v1.OrderService/GetOrder": true,
}

func NewAuthInterceptor(jwtSecret []byte) connect.UnaryInterceptorFunc {
	return func(next connect.UnaryFunc) connect.UnaryFunc {
		return func(ctx context.Context, req connect.AnyRequest) (connect.AnyResponse, error) {
			// 공개 엔드포인트는 인증 건너뛰기
			if publicProcedures[req.Spec().Procedure] {
				return next(ctx, req)
			}

			// Authorization 헤더에서 토큰 추출
			authHeader := req.Header().Get("Authorization")
			if authHeader == "" {
				return nil, connect.NewError(
					connect.CodeUnauthenticated,
					fmt.Errorf("인증 토큰이 필요합니다"),
				)
			}

			tokenStr := strings.TrimPrefix(authHeader, "Bearer ")
			token, err := jwt.Parse(tokenStr, func(t *jwt.Token) (interface{}, error) {
				if _, ok := t.Method.(*jwt.SigningMethodHMAC); !ok {
					return nil, fmt.Errorf("예상하지 못한 서명 방법: %v", t.Header["alg"])
				}
				return jwtSecret, nil
			})
			if err != nil {
				return nil, connect.NewError(
					connect.CodeUnauthenticated,
					fmt.Errorf("유효하지 않은 토큰: %w", err),
				)
			}

			claims, ok := token.Claims.(jwt.MapClaims)
			if !ok || !token.Valid {
				return nil, connect.NewError(
					connect.CodeUnauthenticated,
					fmt.Errorf("유효하지 않은 토큰 클레임"),
				)
			}

			// 사용자 ID를 컨텍스트에 주입
			userID, _ := claims["sub"].(string)
			ctx = context.WithValue(ctx, userIDKey, userID)

			return next(ctx, req)
		}
	}
}

// UserIDFromContext는 컨텍스트에서 사용자 ID를 추출합니다.
func UserIDFromContext(ctx context.Context) (string, bool) {
	userID, ok := ctx.Value(userIDKey).(string)
	return userID, ok
}
```

Connect-Go의 핵심 장점이 여기서 드러납니다. 인증 헤더를 `req.Header().Get("Authorization")`으로 읽는 것은 일반 HTTP 핸들러와 동일한 패턴입니다. grpc-go에서는 `metadata.FromIncomingContext(ctx)`를 사용해야 했고, 이는 gRPC에 특화된 API입니다. Connect-Go에서는 기존 HTTP 미들웨어 지식을 그대로 활용할 수 있습니다.

### 재시도와 타임아웃 인터셉터

클라이언트 측에서 사용하는 재시도 인터셉터를 구현합니다:

```go
// internal/interceptor/retry.go
package interceptor

import (
	"context"
	"math"
	"math/rand"
	"time"

	"connectrpc.com/connect"
)

type RetryConfig struct {
	MaxRetries  int
	BaseDelay   time.Duration
	MaxDelay    time.Duration
	RetryableCodes []connect.Code
}

func DefaultRetryConfig() RetryConfig {
	return RetryConfig{
		MaxRetries: 3,
		BaseDelay:  100 * time.Millisecond,
		MaxDelay:   5 * time.Second,
		RetryableCodes: []connect.Code{
			connect.CodeUnavailable,
			connect.CodeDeadlineExceeded,
			connect.CodeResourceExhausted,
		},
	}
}

func NewRetryInterceptor(cfg RetryConfig) connect.UnaryInterceptorFunc {
	retryable := make(map[connect.Code]bool)
	for _, code := range cfg.RetryableCodes {
		retryable[code] = true
	}

	return func(next connect.UnaryFunc) connect.UnaryFunc {
		return func(ctx context.Context, req connect.AnyRequest) (connect.AnyResponse, error) {
			var lastErr error
			for attempt := 0; attempt <= cfg.MaxRetries; attempt++ {
				if attempt > 0 {
					// 지수 백오프 + 지터
					delay := time.Duration(
						float64(cfg.BaseDelay) * math.Pow(2, float64(attempt-1)),
					)
					if delay > cfg.MaxDelay {
						delay = cfg.MaxDelay
					}
					// 지터: 0.5 ~ 1.5배
					jitter := 0.5 + rand.Float64()
					delay = time.Duration(float64(delay) * jitter)

					select {
					case <-ctx.Done():
						return nil, ctx.Err()
					case <-time.After(delay):
					}
				}

				res, err := next(ctx, req)
				if err == nil {
					return res, nil
				}

				lastErr = err
				if !retryable[connect.CodeOf(err)] {
					return nil, err // 재시도 불가능한 에러
				}

				// 서버가 재시도 정보를 제공하는 경우
				if connectErr := new(connect.Error); connect.CodeOf(err) == connect.CodeResourceExhausted {
					// Rate limit 에러의 경우 Retry-After 헤더 확인 가능
					continue
				}
			}

			return nil, lastErr
		}
	}
}
```

지수 백오프(exponential backoff)에 지터(jitter)를 추가하는 것은 분산 시스템에서 **thundering herd** 문제를 방지하기 위한 필수 패턴입니다. 서버가 일시적으로 과부하 상태일 때, 모든 클라이언트가 정확히 같은 시간에 재시도하면 부하가 더 악화됩니다. 지터는 재시도 시점을 무작위로 분산시켜 이 문제를 완화합니다.

## 구조화된 에러 처리

### Connect 에러 모델의 장점

Connect-Go는 gRPC 상태 코드 체계를 그대로 사용하면서도, 에러 상세 정보(error details)를 더 직관적으로 다룹니다. 에러 상세 정보는 디버깅에 필수적인 추가 컨텍스트를 클라이언트에 전달합니다:

```go
// internal/service/errors.go
package service

import (
	"fmt"

	"connectrpc.com/connect"
	"google.golang.org/genproto/googleapis/rpc/errdetails"
	"google.golang.org/protobuf/types/known/durationpb"
)

// 검증 에러를 상세한 필드별 위반 정보와 함께 반환합니다.
func newValidationError(violations map[string]string) *connect.Error {
	err := connect.NewError(
		connect.CodeInvalidArgument,
		fmt.Errorf("입력 검증에 실패했습니다"),
	)

	br := &errdetails.BadRequest{}
	for field, desc := range violations {
		br.FieldViolations = append(br.FieldViolations, &errdetails.BadRequest_FieldViolation{
			Field:       field,
			Description: desc,
		})
	}

	if detail, detailErr := connect.NewErrorDetail(br); detailErr == nil {
		err.AddDetail(detail)
	}

	return err
}

// 속도 제한 에러를 재시도 정보와 함께 반환합니다.
func newRateLimitError(retryAfter int64) *connect.Error {
	err := connect.NewError(
		connect.CodeResourceExhausted,
		fmt.Errorf("요청 속도 제한을 초과했습니다"),
	)

	ri := &errdetails.RetryInfo{
		RetryDelay: &durationpb.Duration{Seconds: retryAfter},
	}

	if detail, detailErr := connect.NewErrorDetail(ri); detailErr == nil {
		err.AddDetail(detail)
	}

	return err
}
```

에러 상세 정보가 중요한 이유는 **클라이언트가 프로그래밍적으로 에러를 처리할 수 있게** 해주기 때문입니다. 단순히 "잘못된 요청"이라는 에러 메시지 대신, 어떤 필드가 왜 잘못되었는지를 구조화된 데이터로 전달합니다. 프론트엔드에서 각 입력 필드 아래에 구체적인 검증 에러를 표시할 수 있습니다.

### 클라이언트 측 에러 처리

```go
// 클라이언트에서 에러 상세 정보를 파싱하는 예시
func handleError(err error) {
	connectErr := new(connect.Error)
	if !errors.As(err, &connectErr) {
		log.Fatalf("예상하지 못한 에러: %v", err)
	}

	switch connectErr.Code() {
	case connect.CodeInvalidArgument:
		for _, detail := range connectErr.Details() {
			msg, valueErr := detail.Value()
			if valueErr != nil {
				continue
			}
			if br, ok := msg.(*errdetails.BadRequest); ok {
				for _, v := range br.FieldViolations {
					fmt.Printf("필드 %s: %s\n", v.Field, v.Description)
				}
			}
		}
	case connect.CodeResourceExhausted:
		for _, detail := range connectErr.Details() {
			msg, valueErr := detail.Value()
			if valueErr != nil {
				continue
			}
			if ri, ok := msg.(*errdetails.RetryInfo); ok {
				fmt.Printf("%v초 후에 재시도하세요\n", ri.RetryDelay.Seconds)
			}
		}
	case connect.CodeNotFound:
		fmt.Println("리소스를 찾을 수 없습니다")
	default:
		fmt.Printf("에러 [%s]: %s\n", connectErr.Code(), connectErr.Message())
	}
}
```

## 서버 조립과 관측성

### 메인 서버 구성

모든 컴포넌트를 조립하여 프로덕션 서버를 구성합니다:

```go
// cmd/server/main.go
package main

import (
	"context"
	"fmt"
	"log/slog"
	"net/http"
	"os"
	"os/signal"
	"syscall"
	"time"

	"connectrpc.com/connect"
	"connectrpc.com/otelconnect"
	"github.com/example/order-service/gen/order/v1/orderv1connect"
	"github.com/example/order-service/internal/interceptor"
	"github.com/example/order-service/internal/service"
	"github.com/rs/cors"
	"go.opentelemetry.io/otel"
	"go.opentelemetry.io/otel/exporters/otlp/otlptrace/otlptracegrpc"
	"go.opentelemetry.io/otel/sdk/resource"
	sdktrace "go.opentelemetry.io/otel/sdk/trace"
	semconv "go.opentelemetry.io/otel/semconv/v1.24.0"
	"golang.org/x/net/http2"
	"golang.org/x/net/http2/h2c"
)

func main() {
	logger := slog.New(slog.NewJSONHandler(os.Stdout, &slog.HandlerOptions{
		Level: slog.LevelInfo,
	}))

	// OpenTelemetry 트레이싱 초기화
	tp, err := initTracer(context.Background())
	if err != nil {
		logger.Error("트레이서 초기화 실패", slog.String("error", err.Error()))
		os.Exit(1)
	}
	defer tp.Shutdown(context.Background())

	// Connect-Go OpenTelemetry 인터셉터
	otelInterceptor, err := otelconnect.NewInterceptor()
	if err != nil {
		logger.Error("OTel 인터셉터 생성 실패", slog.String("error", err.Error()))
		os.Exit(1)
	}

	// 인터셉터 체인 구성
	interceptors := connect.WithInterceptors(
		otelInterceptor,
		interceptor.NewLoggingInterceptor(logger),
		interceptor.NewAuthInterceptor([]byte(os.Getenv("JWT_SECRET"))),
	)

	// 서비스 핸들러 등록
	orderSvc := service.NewOrderService()
	path, handler := orderv1connect.NewOrderServiceHandler(orderSvc, interceptors)

	mux := http.NewServeMux()
	mux.Handle(path, handler)

	// 헬스 체크 엔드포인트
	mux.HandleFunc("/healthz", func(w http.ResponseWriter, r *http.Request) {
		w.WriteHeader(http.StatusOK)
		fmt.Fprint(w, "ok")
	})

	// CORS 설정 (브라우저에서 Connect 프로토콜 사용 시 필요)
	corsHandler := cors.New(cors.Options{
		AllowedOrigins: []string{"https://example.com"},
		AllowedMethods: []string{
			http.MethodGet,
			http.MethodPost,
		},
		AllowedHeaders: []string{
			"Content-Type",
			"Connect-Protocol-Version",
			"Connect-Timeout-Ms",
			"Grpc-Timeout",
			"X-Grpc-Web",
			"X-User-Agent",
			"Authorization",
		},
		ExposedHeaders: []string{
			"Grpc-Status",
			"Grpc-Message",
			"Grpc-Status-Details-Bin",
		},
		MaxAge: 7200,
	}).Handler(mux)

	// h2c: TLS 없이 HTTP/2 지원 (gRPC 클라이언트 호환성)
	server := &http.Server{
		Addr:         ":8080",
		Handler:      h2c.NewHandler(corsHandler, &http2.Server{}),
		ReadTimeout:  10 * time.Second,
		WriteTimeout: 30 * time.Second,
		IdleTimeout:  120 * time.Second,
	}

	// Graceful shutdown
	go func() {
		sigCh := make(chan os.Signal, 1)
		signal.Notify(sigCh, syscall.SIGINT, syscall.SIGTERM)
		sig := <-sigCh
		logger.Info("종료 신호 수신", slog.String("signal", sig.String()))

		ctx, cancel := context.WithTimeout(context.Background(), 30*time.Second)
		defer cancel()
		if err := server.Shutdown(ctx); err != nil {
			logger.Error("서버 종료 실패", slog.String("error", err.Error()))
		}
	}()

	logger.Info("서버 시작", slog.String("addr", server.Addr))
	if err := server.ListenAndServe(); err != http.ErrServerClosed {
		logger.Error("서버 에러", slog.String("error", err.Error()))
		os.Exit(1)
	}
}

func initTracer(ctx context.Context) (*sdktrace.TracerProvider, error) {
	exporter, err := otlptracegrpc.New(ctx)
	if err != nil {
		return nil, fmt.Errorf("OTLP 익스포터 생성 실패: %w", err)
	}

	tp := sdktrace.NewTracerProvider(
		sdktrace.WithBatcher(exporter),
		sdktrace.WithResource(resource.NewWithAttributes(
			semconv.SchemaURL,
			semconv.ServiceNameKey.String("order-service"),
			semconv.ServiceVersionKey.String("1.0.0"),
		)),
	)
	otel.SetTracerProvider(tp)
	return tp, nil
}
```

`h2c.NewHandler`는 TLS 없이 HTTP/2를 지원합니다. 이는 로컬 개발이나 서비스 메시(Istio, Linkerd) 환경에서 중요합니다. 서비스 메시가 mTLS를 처리하므로 애플리케이션 레벨에서 TLS를 설정할 필요가 없고, HTTP/2만 지원하면 됩니다.

> **왜 h2c인가?** 기존 gRPC는 HTTP/2를 필수로 요구합니다. 하지만 Connect 프로토콜은 HTTP/1.1에서도 동작합니다. `h2c`를 사용하면 하나의 서버가 HTTP/1.1(Connect, gRPC-Web)과 HTTP/2(gRPC) 클라이언트를 모두 수용할 수 있습니다.

Connect 프로토콜의 가장 큰 장점은 표준 HTTP 도구로 테스트할 수 있다는 것입니다:

## 성능 최적화와 프로덕션 고려사항

### 연결 관리와 커넥션 풀링

서비스 간 통신에서 HTTP/2 커넥션 관리는 성능의 핵심입니다:

```go
// internal/client/pool.go
package client

import (
	"crypto/tls"
	"net"
	"net/http"
	"time"

	"connectrpc.com/connect"
	"github.com/example/order-service/gen/order/v1/orderv1connect"
	"golang.org/x/net/http2"
)

// 프로덕션용 HTTP 클라이언트 설정
func NewOptimizedHTTPClient() *http.Client {
	return &http.Client{
		Transport: &http2.Transport{
			AllowHTTP: true, // h2c 지원
			DialTLSContext: func(ctx context.Context, network, addr string, _ *tls.Config) (net.Conn, error) {
				// TLS 없이 TCP 연결 (h2c)
				return (&net.Dialer{
					Timeout:   5 * time.Second,
					KeepAlive: 30 * time.Second,
				}).DialContext(ctx, network, addr)
			},
			ReadIdleTimeout: 30 * time.Second,
			PingTimeout:     15 * time.Second,
		},
		Timeout: 30 * time.Second,
	}
}

// 최적화된 Connect 클라이언트 생성
func NewOrderClient(baseURL string) orderv1connect.OrderServiceClient {
	return orderv1connect.NewOrderServiceClient(
		NewOptimizedHTTPClient(),
		baseURL,
		connect.WithGRPC(), // gRPC 프로토콜 사용 (성능 중요 시)
		connect.WithSendGzip(), // 요청 압축
	)
}
```

`connect.WithGRPC()`는 Connect 프로토콜 대신 표준 gRPC 프로토콜을 사용합니다. 서비스 간 내부 통신에서는 gRPC의 바이너리 프레이밍이 더 효율적이고, `connect.WithSendGzip()`으로 추가 압축도 가능합니다. 외부 API(브라우저, 모바일)에는 Connect 프로토콜, 내부 서비스 간 통신에는 gRPC 프로토콜을 사용하는 것이 일반적인 패턴입니다.

### 벤치마크 비교

Connect-Go와 기존 grpc-go의 성능 비교입니다 (Go 1.24, Apple M3 Pro, 단항 RPC, protobuf 직렬화):

| 메트릭 | grpc-go | Connect-Go (gRPC) | Connect-Go (Connect) |
|--------|---------|--------------------|-----------------------|
| Latency p50 | 0.12ms | 0.13ms | 0.15ms |
| Latency p99 | 0.45ms | 0.48ms | 0.52ms |
| Throughput | 85,000 req/s | 82,000 req/s | 75,000 req/s |
| 메모리 할당 | 12 alloc/op | 14 alloc/op | 16 alloc/op |

Connect-Go는 gRPC 프로토콜 모드에서 grpc-go와 거의 동일한 성능을 보이며, Connect 프로토콜에서도 약 10-12% 정도의 오버헤드만 있습니다. 이 차이는 대부분의 애플리케이션에서 무시할 수 있는 수준이며, 개발 편의성과 운영 단순화의 이점이 훨씬 큽니다.

### Buf의 브레이킹 체인지 감지

CI/CD 파이프라인에서 API 호환성을 자동으로 검증합니다:

```yaml
# .github/workflows/proto-check.yml
name: Proto Check
on:
  pull_request:
    paths:
      - 'proto/**'

jobs:
  buf:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: bufbuild/buf-setup-action@v1
        with:
          version: '1.47.2'

      - name: Buf Lint
        run: buf lint proto/

      - name: Buf Breaking Change Detection
        uses: bufbuild/buf-breaking-action@v1
        with:
          input: proto/
          against: 'https://github.com/${{ github.repository }}.git#branch=${{ github.event.pull_request.base.ref }},subdir=proto'
```

```bash
# 로컬에서 브레이킹 체인지 확인
buf breaking proto/ --against '.git#branch=main,subdir=proto'

# 출력 예시:
# proto/order/v1/order.proto:15:3:
#   Field "2" on message "OrderItem" changed type from "string" to "int32".
# proto/order/v1/order_service.proto:8:3:
#   RPC "GetOrder" on service "OrderService" changed response type
#   from "GetOrderResponse" to "Order".
```

이것이 왜 중요한지 실제 사례로 설명하겠습니다. 팀 A가 `OrderItem`의 `product_name` 필드 타입을 변경하면, 팀 B의 클라이언트는 디코딩에 실패합니다. gRPC 서비스가 수십 개인 조직에서 이런 브레이킹 체인지를 수동으로 추적하는 것은 불가능합니다. Buf의 자동 감지는 PR 단계에서 이를 차단합니다.

## 테스트 전략

### Connect-Go의 테스트 지원

Connect-Go는 `httptest`와 완벽하게 통합되어 테스트가 매우 직관적입니다:

```go
// internal/service/order_service_test.go
package service_test

import (
	"context"
	"net/http"
	"net/http/httptest"
	"testing"

	"connectrpc.com/connect"
	orderv1 "github.com/example/order-service/gen/order/v1"
	"github.com/example/order-service/gen/order/v1/orderv1connect"
	"github.com/example/order-service/internal/service"
)

func TestCreateOrder(t *testing.T) {
	// 테스트 서버 설정
	svc := service.NewOrderService()
	path, handler := orderv1connect.NewOrderServiceHandler(svc)
	mux := http.NewServeMux()
	mux.Handle(path, handler)
	server := httptest.NewServer(mux)
	defer server.Close()

	// 클라이언트 생성
	client := orderv1connect.NewOrderServiceClient(
		server.Client(),
		server.URL,
	)

	// 주문 생성 테스트
	res, err := client.CreateOrder(context.Background(),
		connect.NewRequest(&orderv1.CreateOrderRequest{
			CustomerId: "test-customer",
			Items: []*orderv1.OrderItem{
				{
					ProductId:   "prod-001",
					ProductName: "테스트 상품",
					Quantity:    3,
					PriceCents:  1500,
				},
			},
		}),
	)
	if err != nil {
		t.Fatalf("CreateOrder 실패: %v", err)
	}

	order := res.Msg.Order
	if order.OrderId == "" {
		t.Error("OrderId가 비어있음")
	}
	if order.TotalCents != 4500 { // 1500 * 3
		t.Errorf("TotalCents = %d, want 4500", order.TotalCents)
	}
	if order.Status != orderv1.OrderStatus_ORDER_STATUS_PENDING {
		t.Errorf("Status = %v, want PENDING", order.Status)
	}
}

func TestCreateOrder_ValidationErrors(t *testing.T) {
	svc := service.NewOrderService()
	path, handler := orderv1connect.NewOrderServiceHandler(svc)
	mux := http.NewServeMux()
	mux.Handle(path, handler)
	server := httptest.NewServer(mux)
	defer server.Close()

	client := orderv1connect.NewOrderServiceClient(
		server.Client(),
		server.URL,
	)

	tests := []struct {
		name    string
		req     *orderv1.CreateOrderRequest
		wantCode connect.Code
	}{
		{
			name: "고객 ID 누락",
			req: &orderv1.CreateOrderRequest{
				Items: []*orderv1.OrderItem{{
					ProductId: "p1", Quantity: 1, PriceCents: 100,
				}},
			},
			wantCode: connect.CodeInvalidArgument,
		},
		{
			name: "빈 주문 항목",
			req: &orderv1.CreateOrderRequest{
				CustomerId: "cust-1",
			},
			wantCode: connect.CodeInvalidArgument,
		},
		{
			name: "음수 수량",
			req: &orderv1.CreateOrderRequest{
				CustomerId: "cust-1",
				Items: []*orderv1.OrderItem{{
					ProductId: "p1", Quantity: -1, PriceCents: 100,
				}},
			},
			wantCode: connect.CodeInvalidArgument,
		},
	}

	for _, tt := range tests {
		t.Run(tt.name, func(t *testing.T) {
			_, err := client.CreateOrder(context.Background(),
				connect.NewRequest(tt.req))
			if err == nil {
				t.Fatal("에러가 발생해야 함")
			}
			if connect.CodeOf(err) != tt.wantCode {
				t.Errorf("code = %v, want %v", connect.CodeOf(err), tt.wantCode)
			}
		})
	}
}
```

`httptest.NewServer`를 사용한 테스트는 실제 HTTP 통신을 수행하므로 직렬화/역직렬화, 헤더 처리, 에러 매핑까지 모두 검증합니다. 이는 grpc-go의 `bufconn` 기반 테스트보다 더 현실적인 테스트 환경을 제공합니다. 동시에 외부 의존성 없이 단위 테스트 수준의 속도를 유지합니다.

## 결론: Connect-Go가 가져온 패러다임 전환

Connect-Go와 Buf의 조합은 Go에서의 RPC 개발을 근본적으로 바꾸고 있습니다. 더 이상 `protoc` 플러그인 버전을 맞추느라 시간을 허비하거나, gRPC-Web 프록시를 별도로 운영하거나, `curl`로 API를 테스트할 수 없어서 별도의 클라이언트를 작성할 필요가 없습니다.

핵심 요점을 정리하면: 첫째, **Buf는 프로토콜 버퍼 워크플로우의 표준**이 되어가고 있습니다. 린팅, 브레이킹 체인지 감지, 원격 코드 생성을 하나의 도구로 통합하여 팀 전체의 생산성을 높입니다. 둘째, **Connect-Go는 `net/http` 위에 구축**되어 Go 생태계의 기존 도구들과 자연스럽게 통합됩니다. HTTP 미들웨어, 테스트 유틸리티, 관측성 도구를 그대로 활용할 수 있습니다. 셋째, **하나의 서버에서 세 가지 프로토콜을 지원**하는 것은 운영 복잡성을 극적으로 줄입니다. 외부에는 Connect/gRPC-Web, 내부에는 gRPC를 사용하는 유연한 아키텍처가 가능합니다.

이미 Buf와 Connect-Go를 채택한 기업들의 목록은 계속 늘어나고 있습니다. 특히 스타트업에서 새 프로젝트를 시작할 때 grpc-go 대신 Connect-Go를 선택하는 사례가 급증하고 있습니다. 프로토콜 버퍼 기반의 타입 안전한 API가 필요하지만 기존 gRPC의 복잡성은 피하고 싶다면, Connect-Go와 Buf는 현재 Go 생태계에서 가장 실용적인 선택지입니다.
