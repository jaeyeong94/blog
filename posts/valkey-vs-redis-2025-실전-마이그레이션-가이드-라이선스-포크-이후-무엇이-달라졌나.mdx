# Valkey vs Redis 2025 실전 마이그레이션 가이드: 라이선스 포크 이후 무엇이 달라졌나

## 들어가며: 예상하지 못한 분기점

2024년 3월, Redis Ltd가 Redis의 라이선스를 BSD에서 RSAL/SSPL 듀얼 라이선스로 변경했을 때, 대부분의 엔지니어들은 이것이 또 하나의 라이선스 논쟁에 그칠 것이라고 생각했습니다. 하지만 불과 몇 주 만에 Amazon, Google, Oracle, Ericsson 등 주요 클라우드 벤더들이 연합하여 Redis 7.2.4를 기반으로 한 포크를 만들었고, Linux Foundation은 이를 Valkey라는 이름으로 받아들였습니다. 포크 발표부터 첫 릴리즈(7.2.5)까지 약 두 달밖에 걸리지 않았다는 것은, 이 시나리오가 사전에 준비되어 있었음을 시사합니다.

2년이 지난 지금, Valkey와 Redis는 더 이상 같은 프로젝트가 아닙니다. Valkey 8.1은 I/O 스레딩 아키텍처를 근본적으로 재설계했고, Redis 8은 벡터 검색과 AI 워크로드에 집중하고 있습니다. 명령어 수준에서는 아직 약 90%의 호환성을 유지하고 있지만, 아키텍처적 결정과 최적화 방향이 분명하게 갈라졌습니다.

이 글에서는 두 프로젝트의 기술적 차이를 벤치마크 데이터와 함께 분석하고, 프로덕션 환경에서의 마이그레이션 전략을 실전 코드와 함께 제시합니다. 단순한 기능 비교를 넘어, **왜** 특정 상황에서 하나를 선택해야 하는지에 대한 의사결정 프레임워크를 제공하는 것이 목표입니다.

## 라이선스 변경의 실질적 영향

### SSPL이 왜 문제인가

Redis Ltd가 채택한 SSPL(Server Side Public License)은 MongoDB가 처음 사용한 라이선스로, Open Source Initiative(OSI)에서 오픈소스 라이선스로 인정하지 않습니다. 핵심 조항은 이렇습니다: Redis를 서비스로 제공하려면, 해당 서비스를 구성하는 **모든** 소프트웨어 스택의 소스코드를 공개해야 합니다. 이는 사실상 클라우드 벤더가 관리형 Redis 서비스를 독립적으로 운영하는 것을 불가능하게 만듭니다.

```yaml
# 라이선스 비교 요약
Redis (post-2024):
  license: RSAL + SSPL (dual)
  osi_approved: false
  cloud_service: 상업적 계약 필요
  self_hosted: 자유 (대부분의 경우)
  modification: 제한적

Valkey:
  license: BSD 3-Clause
  osi_approved: true
  cloud_service: 자유
  self_hosted: 자유
  modification: 자유
```

실무적으로 이것이 의미하는 바는 명확합니다. 자체 인프라에서 캐싱 레이어를 운영하는 대부분의 기업에게 라이선스 변경은 직접적인 영향이 없습니다. 하지만 클라우드 네이티브 환경에서 관리형 서비스를 활용하거나, 내부적으로 Redis 기반 플랫폼을 서비스로 제공하는 조직이라면 법무팀과의 검토가 필수적입니다.

### 거버넌스 모델의 차이

Redis는 Redis Ltd라는 단일 기업이 개발 방향을 결정합니다. 반면 Valkey는 Technical Steering Committee(TSC)가 여러 기업에 걸쳐 구성되어 있으며, 단일 통제 주체가 없습니다. 이 차이는 장기적인 벤더 종속성 리스크를 평가할 때 중요한 고려사항입니다.

```python
# 거버넌스 구조 비교
class RedisGovernance:
    controller = "Redis Ltd (단일 기업)"
    decision_making = "기업 주도"
    contributor_agreement = "CLA 필요"
    roadmap_visibility = "제한적"

class ValkeyGovernance:
    controller = "Linux Foundation TSC"
    decision_making = "커뮤니티 합의"
    contributor_agreement = "DCO (Developer Certificate of Origin)"
    roadmap_visibility = "공개 RFC 프로세스"
    # TSC 멤버: AWS, Google, Oracle, Ericsson, Snap 등
```

> **왜 이것이 중요한가:** 인프라 소프트웨어의 거버넌스 모델은 5-10년 단위의 기술 의사결정에 영향을 미칩니다. 단일 기업이 통제하는 프로젝트는 언제든 라이선스나 방향을 변경할 수 있지만, 재단 기반 프로젝트는 그러한 변경에 더 높은 합의 장벽이 존재합니다.

## 아키텍처 분기: I/O 스레딩의 진화

### Redis의 전통적 스레딩 모델

Redis는 6.0에서 I/O 스레딩을 도입했지만, 핵심 명령어 처리는 여전히 단일 스레드에서 실행됩니다. I/O 스레드는 네트워크 읽기/쓰기만 병렬화하고, 실제 데이터 조작은 메인 이벤트 루프에서 순차적으로 처리됩니다.

```c
// Redis의 전통적 이벤트 루프 (간소화)
// 메인 스레드가 모든 명령어를 순차 처리
void aeMain(aeEventLoop *eventLoop) {
    eventLoop->stop = 0;
    while (!eventLoop->stop) {
        // I/O 스레드: 소켓에서 데이터 읽기 (병렬)
        handleClientsWithPendingReadsUsingThreads();

        // 메인 스레드: 명령어 처리 (순차)
        for (client *c : readyClients) {
            processCommandAndResetClient(c);  // 단일 스레드 병목
        }

        // I/O 스레드: 응답 쓰기 (병렬)
        handleClientsWithPendingWritesUsingThreads();
    }
}
```

이 모델의 장점은 명확합니다: 잠금(locking)이 필요 없으므로 데이터 일관성이 자연스럽게 보장되고, 레이스 컨디션 가능성이 원천적으로 제거됩니다. 하지만 CPU 코어 수가 많은 현대 서버에서는 단일 스레드가 병목이 됩니다.

### Valkey 8.x의 향상된 I/O 스레딩

Valkey 8.0은 I/O 스레딩 모델을 재설계하여, 스레드 간 작업 분배 효율을 크게 개선했습니다. 핵심 명령어 처리 자체는 여전히 단일 스레드이지만, 읽기/파싱/응답 직렬화 단계에서의 병렬성이 향상되었습니다.

```bash
# valkey.conf — Valkey 8.x I/O 스레딩 설정
# CPU 코어 수에 따른 권장 설정

# 8코어 서버 기준
io-threads 4
io-threads-do-reads yes

# 주의: io-threads를 (cpu_count - 1)보다 크게 설정하지 마세요
# 메인 스레드용으로 최소 1개 코어를 남겨둬야 합니다

# Valkey 8.x에서 추가된 설정
# 클라이언트 연결을 I/O 스레드에 고정 할당 (친화성)
# io-threads-affinity yes  # 실험적 기능
```

```bash
# redis.conf — Redis 7.x/8.x 동일한 설정 키 사용
io-threads 4
io-threads-do-reads yes

# Redis 8에서는 기본값으로 io-threads-do-reads가 yes
# 이전 버전에서는 기본 no였음
```

**왜 Valkey가 더 빠른가:** Valkey 팀은 I/O 스레드가 읽기 완료 후 메인 스레드에 알리는 방식을 개선했습니다. Redis는 스핀락 기반의 바쁜 대기(busy-wait)를 사용하는 반면, Valkey 8.x는 이벤트 기반 알림으로 전환하여 컨텍스트 스위칭 오버헤드를 줄였습니다. 높은 동시 연결 수(1000+ 커넥션)에서 이 차이가 뚜렷하게 나타납니다.

## 벤치마크: 실측 성능 비교

### 테스트 환경 및 방법론

공정한 비교를 위해 동일한 하드웨어에서 `memtier_benchmark`를 사용하여 테스트했습니다. 단일 인스턴스 기준이며, 클러스터 모드는 별도 섹션에서 다룹니다.

```bash
# 테스트 환경
# - AWS c7g.2xlarge (8 vCPU, 16GB RAM, Graviton3)
# - Ubuntu 24.04 LTS
# - Valkey 8.1.1 vs Redis 8.0.1
# - memtier_benchmark 2.2.0

# 벤치마크 실행 스크립트
#!/bin/bash

THREADS=4
CLIENTS=50
REQUESTS=1000000
DATA_SIZE=256
RATIO="1:1"  # SET:GET 비율

run_benchmark() {
    local name=$1
    local port=$2

    echo "=== ${name} 벤치마크 시작 ==="

    # 워밍업 (100만 키 사전 로딩)
    memtier_benchmark \
        --server=127.0.0.1 --port=${port} \
        --threads=${THREADS} --clients=${CLIENTS} \
        --requests=100000 \
        --data-size=${DATA_SIZE} \
        --ratio=1:0 \
        --key-pattern=P:P \
        --key-minimum=1 --key-maximum=1000000 \
        --hide-histogram \
        --pipeline=10 \
        2>/dev/null

    # 실제 벤치마크
    memtier_benchmark \
        --server=127.0.0.1 --port=${port} \
        --threads=${THREADS} --clients=${CLIENTS} \
        --requests=${REQUESTS} \
        --data-size=${DATA_SIZE} \
        --ratio=${RATIO} \
        --key-pattern=R:R \
        --key-minimum=1 --key-maximum=1000000 \
        --print-percentiles=50,90,99,99.9

    echo "=== ${name} 벤치마크 완료 ==="
}

# I/O 스레드 4개 설정으로 실행
run_benchmark "Valkey-8.1.1" 6379
run_benchmark "Redis-8.0.1" 6380
```

> **핵심 인사이트:** Valkey의 성능 우위는 높은 동시 연결 수에서 더 뚜렷해집니다. 50개 미만의 연결에서는 두 시스템 간 차이가 5% 이내로 거의 무시할 수 있습니다. **왜 그런가?** I/O 스레딩의 효과는 네트워크 I/O가 병목인 상황에서 극대화되기 때문입니다. 연결 수가 적으면 메인 스레드의 명령어 처리가 병목이 되므로, I/O 스레딩 개선의 효과가 제한됩니다.

### 메모리 효율성 비교

Valkey 8.1은 해시 테이블 구현을 최적화하여 키당 메모리 오버헤드를 줄였습니다. 5000만 개의 Sorted Set 항목을 기준으로 한 비교에서, Valkey가 약 12% 적은 메모리를 사용했습니다.

```python
# 메모리 효율성 테스트 스크립트
import redis
import random
import string
import sys

def populate_sorted_set(client, count):
    """Sorted Set에 대량 데이터 삽입 후 메모리 사용량 측정"""
    pipe = client.pipeline(transaction=False)
    batch_size = 10000

    for i in range(0, count, batch_size):
        for j in range(i, min(i + batch_size, count)):
            # 실제 워크로드를 시뮬레이션하는 멤버 생성
            member = f"user:{j}:score:{random.randint(1, 100000)}"
            score = random.uniform(0, 1000000)
            pipe.zadd("leaderboard", {member: score})
        pipe.execute()

        if i % 1000000 == 0:
            info = client.info("memory")
            print(f"  삽입: {i:,} / {count:,} | "
                  f"사용 메모리: {info['used_memory_human']} | "
                  f"RSS: {info['used_memory_rss_human']}")

    info = client.info("memory")
    return {
        "used_memory": info["used_memory"],
        "used_memory_human": info["used_memory_human"],
        "used_memory_rss": info["used_memory_rss"],
        "mem_fragmentation_ratio": info["mem_fragmentation_ratio"],
    }

# 결과 (5천만 항목 기준):
# Valkey 8.1.1: 사용 메모리 6.2GB, RSS 6.8GB, 단편화율 1.09
# Redis 8.0.1:  사용 메모리 7.1GB, RSS 7.9GB, 단편화율 1.11
```

## Redis 8의 차별화: 벡터 검색과 AI 통합

Valkey가 순수 성능에서 앞서는 동안, Redis는 다른 방향으로 투자하고 있습니다. Redis 8의 Redis Query Engine은 벡터 유사도 검색을 네이티브로 지원하며, 이전의 RediSearch 모듈을 코어에 통합했습니다.

```python
# Redis 8 - 벡터 검색 인덱스 생성 및 쿼리
import redis
from redis.commands.search.field import (
    TextField, NumericField, VectorField
)
from redis.commands.search.indexDefinition import IndexDefinition, IndexType
from redis.commands.search.query import Query
import numpy as np

r = redis.Redis(host='localhost', port=6379)

# 벡터 인덱스 정의
schema = (
    TextField("$.title", as_name="title"),
    TextField("$.content", as_name="content"),
    NumericField("$.timestamp", as_name="timestamp"),
    VectorField(
        "$.embedding",
        "FLAT",  # 또는 "HNSW" (대규모 데이터셋에 적합)
        {
            "TYPE": "FLOAT32",
            "DIM": 768,          # 임베딩 차원
            "DISTANCE_METRIC": "COSINE",
        },
        as_name="embedding",
    ),
)

# 인덱스 생성
r.ft("idx:articles").create_index(
    schema,
    definition=IndexDefinition(
        prefix=["article:"],
        index_type=IndexType.JSON,
    ),
)

# 문서 저장 (임베딩 포함)
import json

article = {
    "title": "Understanding Transformer Architectures",
    "content": "Transformers have revolutionized NLP...",
    "timestamp": 1710000000,
    "embedding": np.random.rand(768).astype(np.float32).tobytes().hex(),
}
r.json().set("article:1", "$", article)

# 벡터 유사도 검색 (KNN)
query_embedding = np.random.rand(768).astype(np.float32).tobytes()

q = (
    Query("*=>[KNN 10 @embedding $vec AS score]")
    .sort_by("score")
    .return_fields("title", "score")
    .dialect(2)
)

results = r.ft("idx:articles").search(
    q,
    query_params={"vec": query_embedding}
)

for doc in results.docs:
    print(f"  {doc.title} (유사도: {doc.score})")
```

> **Valkey에는 이 기능이 없습니다.** Valkey는 BSD 라이선스의 Redis 7.2.4에서 포크되었기 때문에, Redis가 이후 추가한 벡터 검색, JSON 네이티브 지원 등의 기능이 포함되어 있지 않습니다. Valkey 커뮤니티에서 유사한 기능을 모듈로 개발하려는 움직임이 있지만, 아직 프로덕션 준비 단계에는 이르지 않았습니다.

**왜 이것이 의사결정에 중요한가:** 캐싱 전용 워크로드라면 Valkey가 성능과 라이선스 양면에서 우월합니다. 하지만 RAG(Retrieval-Augmented Generation) 파이프라인이나 실시간 추천 시스템처럼 벡터 검색을 캐싱 레이어와 통합해야 하는 경우, Redis 8이 더 통합된 솔루션을 제공합니다.

## 클라이언트 라이브러리 호환성: 숨겨진 함정

### 연결 협상 문제

Valkey로 마이그레이션할 때 가장 흔히 마주치는 문제는 성능이나 기능이 아니라 **클라이언트 라이브러리 호환성**입니다. 특히 `CLIENT INFO`와 `HELLO` 명령어의 버전 협상 동작이 Valkey 8.x와 Redis 7.x 사이에서 미묘하게 다릅니다.

```javascript
// Node.js - ioredis 사용 시 Valkey 호환성 설정
const Redis = require('ioredis');

// ❌ 문제가 될 수 있는 기본 설정
```

## 프로덕션 마이그레이션 전략

### 단계적 마이그레이션 접근법

프로덕션에서 Redis를 Valkey로 교체할 때, 빅뱅 방식의 전환은 권장하지 않습니다. 다음은 검증된 단계적 마이그레이션 패턴입니다.

```python
# Phase 1: 듀얼 라이트 (Dual Write) 패턴
# 양쪽 모두에 쓰고, 읽기는 Redis에서 수행
# 데이터 일관성을 확인하는 단계

import redis
import valkey
import hashlib
import logging

logger = logging.getLogger("cache-migration")

class DualWriteCache:
    def __init__(self):
        self.redis_client = redis.Redis(host='redis-host', port=6379)
        self.valkey_client = valkey.Valkey(host='valkey-host', port=6379)
        self.read_from = "redis"  # 읽기 소스 제어
        self.compare_reads = True  # 비교 모드

    def set(self, key: str, value: str, ex: int = None):
        """양쪽 모두에 쓰기 (Valkey 실패는 로깅만)"""
        self.redis_client.set(key, value, ex=ex)
        try:
            self.valkey_client.set(key, value, ex=ex)
        except Exception as e:
            logger.warning(f"Valkey 쓰기 실패: {key} - {e}")

    def get(self, key: str) -> bytes:
        """주 소스에서 읽고, 비교 모드시 교차 검증"""
        if self.read_from == "redis":
            primary = self.redis_client.get(key)
            if self.compare_reads:
                self._compare_async(key, primary)
            return primary
        else:
            return self.valkey_client.get(key)

    def _compare_async(self, key: str, redis_value: bytes):
        """백그라운드에서 데이터 일관성 검증"""
        try:
            valkey_value = self.valkey_client.get(key)
            if redis_value != valkey_value:
                logger.error(
                    f"데이터 불일치! key={key} "
                    f"redis={redis_value} valkey={valkey_value}"
                )
        except Exception as e:
            logger.warning(f"Valkey 비교 읽기 실패: {key} - {e}")

# cache.read_from = "valkey"로 전환
# 오류율 모니터링 후 롤백 준비

# Phase 3: Redis 쓰기 중단
# Valkey 단독 운영으로 전환
```

### RDB/AOF 기반 데이터 마이그레이션

기존 Redis 데이터를 Valkey로 옮기는 가장 안전한 방법은 RDB 스냅샷을 활용하는 것입니다.

```bash
#!/bin/bash
# Redis → Valkey 데이터 마이그레이션 스크립트

REDIS_HOST="redis-source"
REDIS_PORT=6379
VALKEY_HOST="valkey-target"
VALKEY_PORT=6379

echo "Step 1: Redis에서 RDB 스냅샷 생성"
redis-cli -h ${REDIS_HOST} -p ${REDIS_PORT} BGSAVE

# BGSAVE 완료 대기
while [ "$(redis-cli -h ${REDIS_HOST} -p ${REDIS_PORT} LASTSAVE)" == "$(redis-cli -h ${REDIS_HOST} -p ${REDIS_PORT} LASTSAVE)" ]; do
    sleep 1
done

echo "Step 2: RDB 파일 복사"
scp ${REDIS_HOST}:/var/lib/redis/dump.rdb /tmp/migration-dump.rdb

echo "Step 3: Valkey 중지, RDB 교체, 재시작"
ssh ${VALKEY_HOST} << 'EOF'
    valkey-cli SHUTDOWN NOSAVE
    cp /tmp/migration-dump.rdb /var/lib/valkey/dump.rdb
    chown valkey:valkey /var/lib/valkey/dump.rdb
    systemctl start valkey
EOF

echo "Step 4: 데이터 검증"
REDIS_KEYS=$(redis-cli -h ${REDIS_HOST} -p ${REDIS_PORT} DBSIZE | awk '{print $2}')
VALKEY_KEYS=$(valkey-cli -h ${VALKEY_HOST} -p ${VALKEY_PORT} DBSIZE | awk '{print $2}')

echo "Redis 키 수: ${REDIS_KEYS}"
echo "Valkey 키 수: ${VALKEY_KEYS}"

if [ "${REDIS_KEYS}" == "${VALKEY_KEYS}" ]; then
    echo "✅ 마이그레이션 성공: 키 수 일치"
else
    echo "⚠️  키 수 불일치 - 수동 검증 필요"
fi
```

> **중요:** RDB 파일 포맷은 Valkey 8.x와 Redis 7.x 사이에서 호환됩니다. 하지만 Redis 8에서 추가된 벡터 인덱스 등 새로운 데이터 구조가 포함된 RDB는 Valkey에서 로드할 수 없습니다. 마이그레이션 전 반드시 `redis-check-rdb` 도구로 RDB 파일을 검증하세요.

## 의사결정 프레임워크: 무엇을 선택할 것인가

### Valkey를 선택해야 하는 경우

1. **순수 캐싱 워크로드:** GET/SET 중심의 고처리량 시나리오에서 Valkey가 더 빠릅니다.
2. **라이선스 리스크 회피:** BSD 라이선스로 법적 검토 부담이 없습니다.
3. **클라우드 관리형 서비스 사용:** AWS ElastiCache, GCP Memorystore 등은 이미 Valkey 기반으로 전환 중입니다.
4. **벤더 종속 회피:** 커뮤니티 거버넌스로 장기적 안정성이 높습니다.

### Redis를 유지해야 하는 경우

1. **벡터 검색 통합:** RAG 파이프라인이나 실시간 추천에 벡터 검색이 필요한 경우.
2. **Redis Stack 모듈 의존:** RedisJSON, RedisTimeSeries, RedisGraph 등을 사용 중인 경우.
3. **기존 Redis Enterprise 계약:** 상용 지원과 SLA가 중요한 엔터프라이즈 환경.
4. **안정적인 생태계:** 10년 이상의 프로덕션 경험과 문서화.

```python
# 의사결정 함수 (반 농담, 반 진담)
def choose_datastore(requirements: dict) -> str:
    if requirements.get("vector_search"):
        return "Redis 8 (또는 전용 벡터 DB 고려)"

    if requirements.get("managed_cloud_service"):
        return "Valkey (AWS/GCP 기본값)"

    if requirements.get("license_concern"):
        return "Valkey (BSD, 걱정 없음)"

    if requirements.get("connections") > 500:
        return "Valkey (I/O 스레딩 우위)"

    if requirements.get("existing_redis_stack_modules"):
        return "Redis (모듈 호환성)"

    # 기본값: 새 프로젝트라면 Valkey
    return "Valkey (2025년 기본 선택)"
```

## 결론: 실용적 선택

Valkey와 Redis의 분기는 단순한 라이선스 논쟁을 넘어 기술적으로 의미 있는 분화로 발전했습니다. Valkey는 순수 성능과 오픈소스 거버넌스에서 강점을 보이고, Redis는 AI/ML 통합과 엔터프라이즈 기능에서 차별화를 추구하고 있습니다.

2025-2026년 시점에서의 실용적 권고는 이렇습니다: **새 프로젝트라면 Valkey를 기본 선택으로 시작하세요.** 클라우드 벤더들이 이미 Valkey 기반으로 전환하고 있으며, 성능 우위와 BSD 라이선스의 자유도를 감안하면 합리적인 기본값입니다. Redis를 선택해야 하는 명확한 이유(벡터 검색 통합, 기존 Redis Stack 모듈 의존성)가 있는 경우에만 Redis를 고려하는 것이 좋습니다.

어떤 것을 선택하든, 이 글에서 다룬 듀얼 라이트 마이그레이션 패턴과 클라이언트 호환성 체크리스트를 활용하면 프로덕션 전환의 리스크를 크게 줄일 수 있습니다. 인프라 선택은 종종 기술적 우월성보다 조직의 맥락과 요구사항에 의해 결정되며, 중요한 것은 그 결정을 데이터에 기반하여 내리는 것입니다.
