# TiDB와 CockroachDB로 글로벌 분산 SQL 데이터베이스 구축하기

## 왜 분산 SQL이 필요한가

전통적인 관계형 데이터베이스는 수십 년간 애플리케이션의 데이터 계층을 지탱해 왔습니다. PostgreSQL이나 MySQL 같은 단일 노드 RDBMS는 뛰어난 안정성과 풍부한 SQL 기능을 제공하지만, 글로벌 서비스를 운영하는 순간 근본적인 한계에 부딪힙니다. 단일 마스터에 모든 쓰기가 집중되면 수직 확장의 천장에 금세 도달하고, 리전 간 복제 지연은 데이터 일관성 문제를 야기합니다.

읽기 전용 레플리카를 늘리는 방식은 읽기 부하를 분산할 수 있지만, 쓰기 확장성 문제를 해결하지 못합니다. 샤딩을 직접 구현하면 애플리케이션 복잡도가 기하급수적으로 증가하며, 크로스 샤드 조인이나 분산 트랜잭션 처리는 개발팀의 생산성을 심각하게 저하시킵니다. 2025년 기준으로 글로벌 사용자에게 100ms 이내의 응답 시간을 보장하면서도 ACID 트랜잭션을 유지해야 하는 서비스가 급격히 늘어나고 있습니다.

이 문제를 해결하기 위해 등장한 것이 NewSQL 데이터베이스입니다. NewSQL은 전통적인 RDBMS의 SQL 호환성과 ACID 보장을 유지하면서, NoSQL의 수평 확장성을 결합합니다. Google Spanner가 이 패러다임의 시작점이었다면, 오픈소스 진영에서는 TiDB와 CockroachDB가 가장 성숙한 대안으로 자리잡았습니다.

이 글에서는 두 데이터베이스의 아키텍처를 심층 비교하고, 실제 프로덕션 환경에서 마이그레이션하는 전략을 구체적인 코드와 벤치마크와 함께 살펴봅니다. 단순한 기능 비교를 넘어, **왜** 각 시스템이 특정 설계 결정을 내렸는지, 그리고 그 결정이 여러분의 워크로드에 어떤 영향을 미치는지 이해하는 것이 목표입니다.

## 아키텍처 심층 비교

### TiDB: MySQL 호환 분산 SQL

TiDB(Titanium Database)는 PingCAP이 개발한 MySQL 호환 분산 SQL 데이터베이스입니다. Google Spanner와 F1 논문에서 영감을 받았지만, MySQL 와이어 프로토콜을 완벽히 지원하여 기존 MySQL 애플리케이션을 최소한의 변경으로 마이그레이션할 수 있다는 점이 핵심 차별점입니다.

TiDB의 아키텍처는 세 가지 핵심 컴포넌트로 구성됩니다. **TiDB Server**는 SQL 파싱, 최적화, 실행을 담당하는 스테이트리스 레이어입니다. 스테이트리스이기 때문에 로드밸런서 뒤에서 무한히 수평 확장할 수 있습니다. **TiKV**는 분산 키-밸류 스토어로, 실제 데이터를 저장합니다. Raft 합의 알고리즘으로 데이터를 복제하며, 기본적으로 3개의 복제본을 유지합니다. **PD(Placement Driver)**는 클러스터의 메타데이터를 관리하고, 리전 스케줄링과 로드밸런싱을 조율하는 클러스터 브레인 역할을 합니다.

TiDB가 MySQL 호환성을 선택한 이유는 명확합니다. 전 세계적으로 MySQL은 가장 널리 사용되는 오픈소스 RDBMS이며, 기존 MySQL 사용자가 분산 데이터베이스로 전환할 때의 진입 장벽을 극적으로 낮출 수 있기 때문입니다. 실제로 `mysqldump`로 내보낸 데이터를 TiDB에 바로 적재할 수 있고, 기존 MySQL 드라이버와 ORM이 그대로 동작합니다.

```sql
-- TiDB에서의 테이블 생성: MySQL 문법 그대로 사용
CREATE TABLE orders (
    id BIGINT NOT NULL AUTO_INCREMENT,
    user_id BIGINT NOT NULL,
    product_id BIGINT NOT NULL,
    amount DECIMAL(10, 2) NOT NULL,
    status ENUM('pending', 'confirmed', 'shipped', 'delivered') DEFAULT 'pending',
    region VARCHAR(20) NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (id),
    INDEX idx_user_status (user_id, status),
    INDEX idx_region_created (region, created_at)
) ENGINE=InnoDB;

-- TiDB 고유: SHARD_ROW_ID_BITS로 핫스팟 방지
-- AUTO_INCREMENT가 단일 리전에 집중되는 문제를 해결
ALTER TABLE orders SET TIFLASH REPLICA 1;
ALTER TABLE orders SHARD_ROW_ID_BITS = 4 PRE_SPLIT_REGIONS = 3;
```

TiDB는 2024년 후반 v8.x에서 **TiFlash** 컬럼나 스토어를 대폭 강화했습니다. HTAP(Hybrid Transactional/Analytical Processing) 워크로드를 지원하여, OLTP 트랜잭션을 처리하면서 동시에 실시간 분석 쿼리를 실행할 수 있습니다. TiFlash는 Raft Learner 프로토콜을 통해 TiKV의 데이터를 비동기적으로 컬럼나 형태로 복제하므로, 분석 쿼리가 트랜잭션 성능에 영향을 주지 않습니다.

### CockroachDB: PostgreSQL 호환 분산 SQL

CockroachDB는 Cockroach Labs가 개발한 PostgreSQL 호환 분산 SQL 데이터베이스입니다. 이름처럼 "바퀴벌레"의 생존력을 목표로 하며, 노드 장애, 리전 장애, 심지어 데이터센터 전체 장애에서도 무중단 서비스를 보장하는 것이 핵심 설계 목표입니다.

CockroachDB의 아키텍처는 TiDB와 달리 **균질적(homogeneous)** 노드 구조를 채택합니다. 모든 노드가 SQL 처리와 데이터 저장을 동시에 담당하며, 별도의 코디네이터 노드가 없습니다. 이는 운영 복잡도를 크게 줄여주는 장점이 있습니다. 각 노드는 전체 SQL 레이어(파싱, 최적화, 실행)와 분산 키-밸류 스토어를 모두 포함합니다.

CockroachDB가 PostgreSQL 호환성을 선택한 이유도 전략적입니다. PostgreSQL은 고급 SQL 기능(CTE, 윈도우 함수, JSON 연산 등)에서 MySQL보다 우위에 있으며, 엔터프라이즈 환경에서 선호도가 높습니다. 특히 PostgreSQL의 풍부한 타입 시스템과 확장성은 복잡한 도메인 모델링에 유리합니다.

```sql
-- CockroachDB: PostgreSQL 문법 + 글로벌 테이블 설정
CREATE TABLE orders (
    id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
    user_id UUID NOT NULL,
    product_id UUID NOT NULL,
    amount DECIMAL(10, 2) NOT NULL,
    status TEXT DEFAULT 'pending' CHECK (status IN ('pending', 'confirmed', 'shipped', 'delivered')),
    region TEXT NOT NULL,
    created_at TIMESTAMPTZ DEFAULT now(),
    INDEX idx_user_status (user_id, status),
    INDEX idx_region_created (region, created_at)
);

-- CockroachDB 고유: 멀티 리전 설정
ALTER DATABASE shop PRIMARY REGION "us-east1";
ALTER DATABASE shop ADD REGION "eu-west1";
ALTER DATABASE shop ADD REGION "ap-northeast1";

-- 리전 기반 테이블: 각 행이 region 컬럼에 따라 해당 리전에 저장
ALTER TABLE orders SET LOCALITY REGIONAL BY ROW AS region;
```

CockroachDB v24.x에서 도입된 **Physical Cluster Replication(PCR)**은 전체 클러스터를 다른 클러스터로 비동기 복제하는 기능입니다. 이는 재해 복구 시나리오에서 RTO(Recovery Time Objective)를 분 단위로 줄여주며, 블루-그린 배포 전략에도 활용할 수 있습니다.

### 합의 알고리즘과 트랜잭션 모델

두 시스템 모두 Raft 합의 알고리즘을 사용하지만, 트랜잭션 처리 방식에서 중요한 차이가 있습니다.

TiDB는 **Percolator** 모델(Google의 분산 트랜잭션 프로토콜)을 기반으로 한 2PC(Two-Phase Commit)를 사용합니다. 트랜잭션의 타임스탬프는 PD가 중앙에서 할당하므로, 전역적으로 일관된 스냅샷 격리(Snapshot Isolation)를 제공합니다. 이 중앙 집중식 타임스탬프 할당은 단일 리전에서는 효율적이지만, 멀티 리전 환경에서는 PD까지의 라운드트립이 추가 지연을 발생시킬 수 있습니다.

CockroachDB는 **HLC(Hybrid Logical Clock)**를 사용하여 분산 환경에서 타임스탬프를 생성합니다. 이 방식은 중앙 타임스탬프 서버 없이도 직렬화 가능한(Serializable) 격리 수준을 제공할 수 있으며, 멀티 리전 환경에서 더 낮은 지연 시간을 달성할 수 있습니다. CockroachDB의 기본 격리 수준은 **Serializable**로, 이는 가장 엄격한 ANSI SQL 격리 수준입니다.

```go
// CockroachDB의 재시도 로직: Serializable 격리에서는
// 트랜잭션 충돌 시 재시도가 필요할 수 있음
func executeWithRetry(db *sql.DB, fn func(*sql.Tx) error) error {
    const maxRetries = 5
    for i := 0; i < maxRetries; i++ {
        tx, err := db.Begin()
        if err != nil {
            return fmt.Errorf("begin tx: %w", err)
        }
        
        err = fn(tx)
        if err == nil {
            return tx.Commit()
        }
        
        tx.Rollback()
        
        // CockroachDB 재시도 가능 에러 확인 (SQLSTATE 40001)
        if pqErr, ok := err.(*pq.Error); ok && pqErr.Code == "40001" {
            // 지수 백오프로 재시도
            time.Sleep(time.Duration(math.Pow(2, float64(i))) * 10 * time.Millisecond)
            continue
        }
        return err // 재시도 불가능한 에러
    }
    return fmt.Errorf("max retries exceeded")
}
```

이 차이가 왜 중요한가요? Serializable 격리는 데이터 정합성 측면에서 가장 안전하지만, 쓰기 충돌이 빈번한 워크로드에서는 재시도 비용이 증가합니다. TiDB의 Snapshot Isolation은 읽기-쓰기 혼합 워크로드에서 더 높은 처리량을 보여주는 반면, Write Skew 같은 이상 현상에 노출될 수 있습니다. 워크로드의 특성에 따라 적합한 시스템이 달라지는 핵심 요인입니다.

## 멀티 리전 전략 비교

글로벌 서비스에서 가장 중요한 것은 사용자 근처에서 데이터를 읽고 쓸 수 있는 능력입니다. 두 시스템 모두 멀티 리전을 지원하지만, 접근 방식이 크게 다릅니다.

### TiDB의 Placement Rules

TiDB는 **Placement Rules in SQL**을 통해 데이터의 물리적 배치를 세밀하게 제어합니다. 관리자는 SQL 구문으로 특정 테이블이나 파티션이 어느 리전에 저장될지 정의할 수 있습니다.

```sql
-- TiDB: Placement Policy 정의
CREATE PLACEMENT POLICY asia_policy
    PRIMARY_REGION="ap-northeast-1"
    REGIONS="ap-northeast-1,ap-southeast-1"
    FOLLOWERS=2;

CREATE PLACEMENT POLICY europe_policy
    PRIMARY_REGION="eu-west-1"
    REGIONS="eu-west-1,eu-central-1"
    FOLLOWERS=2;

-- 파티셔닝과 결합하여 리전별 데이터 배치
CREATE TABLE user_profiles (
    id BIGINT AUTO_INCREMENT,
    email VARCHAR(255) NOT NULL,
    name VARCHAR(100),
    region VARCHAR(20) NOT NULL,
    preferences JSON,
    PRIMARY KEY (id, region)
) PARTITION BY LIST COLUMNS (region) (
    PARTITION p_asia VALUES IN ('KR', 'JP', 'SG')
        PLACEMENT POLICY asia_policy,
    PARTITION p_europe VALUES IN ('DE', 'FR', 'GB')
        PLACEMENT POLICY europe_policy
);
```

이 접근 방식의 장점은 유연성입니다. 테이블 단위, 파티션 단위, 심지어 인덱스 단위로 데이터 배치를 제어할 수 있습니다. 하지만 관리자가 직접 정책을 설계하고 유지해야 하므로, 운영 오버헤드가 있습니다. PD가 정책에 따라 리전 스케줄링을 수행하지만, 최적의 정책을 찾는 것은 사용자의 몫입니다.

### CockroachDB의 멀티 리전 추상화

CockroachDB는 v21.1부터 도입된 **멀티 리전 추상화**를 통해 더 선언적인 접근 방식을 제공합니다. 사용자는 테이블의 지역성(Locality) 유형만 선언하면, CockroachDB가 자동으로 최적의 데이터 배치를 결정합니다.

```sql
-- CockroachDB: 세 가지 Locality 패턴

-- 1. REGIONAL BY TABLE: 테이블 전체가 특정 리전에
-- 읽기/쓰기가 한 리전에 집중된 테이블에 적합
ALTER TABLE system_config SET LOCALITY REGIONAL BY TABLE IN "us-east1";

-- 2. REGIONAL BY ROW: 행 단위로 리전 할당
-- 사용자별로 가까운 리전에서 서비스할 때 적합
ALTER TABLE user_sessions SET LOCALITY REGIONAL BY ROW;

-- 3. GLOBAL: 모든 리전에서 빠른 읽기 (쓰기는 느림)
-- 자주 변경되지 않는 참조 데이터에 적합
ALTER TABLE product_catalog SET LOCALITY GLOBAL;

-- REGIONAL BY ROW의 동작 원리 확인
SHOW CREATE TABLE user_sessions;
-- crdb_region 컬럼이 자동 추가되고, 게이트웨이 리전 기반으로 값 설정
```

CockroachDB의 GLOBAL 테이블은 **Non-Blocking Transaction** 기술을 사용합니다. 쓰기 시 미래 타임스탬프로 커밋하여 모든 리전의 레플리카가 즉시 일관된 읽기를 제공할 수 있게 합니다. 이는 읽기가 압도적으로 많고 쓰기가 드문 테이블(상품 카탈로그, 설정값 등)에 매우 효과적입니다.

**왜 이런 차이가 생겼을까?** TiDB는 원래 단일 리전 분산 DB로 시작하여 멀티 리전을 점진적으로 추가했기에, 세밀한 제어를 제공하되 사용자가 직접 설계합니다. CockroachDB는 처음부터 글로벌 분산을 핵심 설계 목표로 잡았기에, 더 높은 추상화 수준을 제공하여 사용자의 판단 부담을 줄입니다.

## 성능 벤치마크

실제 프로덕션과 유사한 환경에서의 벤치마크 결과를 살펴봅시다. 테스트 환경은 AWS에서 3개 리전(us-east-1, eu-west-1, ap-northeast-2)에 각 3노드를 배포한 9노드 클러스터입니다.

### 단일 리전 OLTP 성능 (sysbench)

동일 리전 내에서의 순수 트랜잭션 처리 성능입니다. 16 vCPU, 64GB RAM 인스턴스 3대, 1000만 행 기준 결과입니다.

| 메트릭 | TiDB v8.5 | CockroachDB v24.3 |
|--------|-----------|-------------------|
| Point Select (QPS) | 128,000 | 95,000 |
| Update Index (QPS) | 24,500 | 18,200 |
| Read-Write Mix (TPS) | 3,800 | 2,950 |
| P99 Latency (ms) | 8.4 | 12.1 |
| P999 Latency (ms) | 18.7 | 28.3 |

TiDB가 단일 리전 OLTP에서 약 25-35% 높은 처리량을 보여줍니다. 이는 TiDB의 분리된 아키텍처(SQL 레이어와 스토리지 레이어 분리)가 단일 리전에서 더 효율적인 리소스 활용을 가능하게 하기 때문입니다. 또한 TiDB의 Snapshot Isolation이 CockroachDB의 Serializable보다 잠금 경합이 적어 높은 동시성에서 유리합니다.

### 멀티 리전 읽기 지연 시간

리전 간 읽기 성능은 데이터 배치 전략에 크게 좌우됩니다.

| 시나리오 | TiDB | CockroachDB |
|----------|------|-------------|
| 로컬 리전 읽기 | 3.2ms | 2.8ms |
| 원격 리전 읽기 (스테일 읽기) | 4.1ms | 3.5ms |
| 원격 리전 읽기 (최신 데이터) | 68ms | 72ms |
| GLOBAL 테이블 읽기 | N/A | 3.1ms |
| TiFlash 분석 쿼리 | 450ms | N/A |

CockroachDB의 GLOBAL 테이블은 어느 리전에서든 로컬 수준의 읽기 지연을 보여줍니다. TiDB의 TiFlash는 분석 쿼리에서 독보적인 성능을 제공합니다. 두 시스템 모두 "Stale Read" (약간 오래된 데이터 허용) 기능으로 원격 읽기 지연을 크게 줄일 수 있습니다.

```sql
-- TiDB: Stale Read로 읽기 지연 최소화
-- 5초 이내의 데이터로 충분할 때
SET @ts = NOW() - INTERVAL 5 SECOND;
SELECT /*+ READ_FROM_STORAGE(TIFLASH[orders]) */ 
    region, COUNT(*), SUM(amount) 
FROM orders AS OF TIMESTAMP @ts
GROUP BY region;

-- CockroachDB: Follower Read로 가장 가까운 레플리카에서 읽기
SET CLUSTER SETTING kv.closed_timestamp.target_duration = '3s';
SELECT * FROM orders 
    AS OF SYSTEM TIME follower_read_timestamp()
    WHERE user_id = $1;
```

## 실전 마이그레이션 가이드

### MySQL에서 TiDB로 마이그레이션

TiDB의 MySQL 호환성 덕분에 마이그레이션이 비교적 간단하지만, 주의해야 할 차이점들이 있습니다.

```python
# TiDB Data Migration (DM) 도구를 사용한 온라인 마이그레이션
# dm-task.yaml 설정 예시

# Python 스크립트: 마이그레이션 전 호환성 검사
import pymysql

def check_tidb_compatibility(mysql_conn_params: dict) -> list[str]:
    """MySQL 스키마의 TiDB 호환성을 검사합니다."""
    issues = []
    conn = pymysql.connect(**mysql_conn_params)
    cursor = conn.cursor()
    
    # 1. AUTO_INCREMENT 간격 확인
    # TiDB는 성능을 위해 AUTO_INCREMENT 값이 불연속적
    cursor.execute("""
        SELECT TABLE_NAME, AUTO_INCREMENT 
        FROM INFORMATION_SCHEMA.TABLES 
        WHERE TABLE_SCHEMA = %s AND AUTO_INCREMENT IS NOT NULL
    """, (mysql_conn_params['database'],))
    
    for table, auto_inc in cursor.fetchall():
        if auto_inc and auto_inc > 0:
            issues.append(
                f"⚠️ {table}: AUTO_INCREMENT 사용 중. "
                f"TiDB에서는 ID 값이 불연속적일 수 있음 (30000, 30001 → 60000 점프 가능)"
            )
    
    # 2. 외래 키 확인 (TiDB v8.x에서 실험적 지원)
    cursor.execute("""
        SELECT TABLE_NAME, CONSTRAINT_NAME, REFERENCED_TABLE_NAME
        FROM INFORMATION_SCHEMA.KEY_COLUMN_USAGE
        WHERE TABLE_SCHEMA = %s AND REFERENCED_TABLE_NAME IS NOT NULL
    """, (mysql_conn_params['database'],))
    
    fk_count = cursor.rowcount
    if fk_count > 0:
        issues.append(
            f"⚠️ 외래 키 {fk_count}개 발견. "
            f"TiDB v8.x에서 실험적 지원되지만, 애플리케이션 레벨 제약 권장"
        )
    
    # 3. 저장 프로시저 / 트리거 확인
    cursor.execute("""
        SELECT ROUTINE_NAME, ROUTINE_TYPE 
        FROM INFORMATION_SCHEMA.ROUTINES 
        WHERE ROUTINE_SCHEMA = %s
    """, (mysql_conn_params['database'],))
    
    routines = cursor.fetchall()
    if routines:
        issues.append(
            f"⚠️ 저장 프로시저/함수 {len(routines)}개 발견. "
            f"TiDB 호환성을 개별 확인 필요"
        )
    
    cursor.close()
    conn.close()
    return issues
```

> **핵심 주의사항:** TiDB에서 `AUTO_INCREMENT`는 단조 증가를 보장하지만, 연속성은 보장하지 않습니다. 분산 환경에서 각 TiDB 서버가 ID 범위를 미리 할당받기 때문입니다. ID의 순서에 의존하는 비즈니스 로직이 있다면 반드시 수정해야 합니다.

### PostgreSQL에서 CockroachDB로 마이그레이션

CockroachDB는 PostgreSQL과 높은 호환성을 제공하지만, 몇 가지 중요한 차이가 있습니다.

```typescript
// Node.js: CockroachDB 마이그레이션 호환성 검증 도구
import { Client } from 'pg';

interface CompatibilityIssue {
  severity: 'error' | 'warning' | 'info';
  table: string;
  message: string;
}

async function checkCockroachCompatibility(
  pgClient: Client
): Promise<CompatibilityIssue[]> {
  const issues: CompatibilityIssue[] = [];

  // 1. SERIAL → UUID 변환 권장 확인
  const serialColumns = await pgClient.query(`
    SELECT c.table_name, c.column_name, c.column_default
    FROM information_schema.columns c
    WHERE c.table_schema = 'public'
      AND c.column_default LIKE 'nextval%'
  `);

  for (const row of serialColumns.rows) {
    issues.push({
      severity: 'warning',
      table: row.table_name,
      message: `컬럼 ${row.column_name}이 SERIAL 사용 중. ` +
        `CockroachDB에서는 UUID(gen_random_uuid())로 변환 권장. ` +
        `SERIAL은 단일 노드 병목 발생 가능`
    });
  }

  // 2. 지원되지 않는 확장 확인
  const extensions = await pgClient.query(`
    SELECT extname FROM pg_extension WHERE extname != 'plpgsql'
  `);

  const unsupportedExtensions = new Set([
    'postgis', 'pg_trgm', 'hstore', 'ltree', 'tablefunc'
  ]);
  // CockroachDB v24.x 지원 확장: postgis(부분), pg_trgm(부분), 
  // uuid-ossp, spatial

  for (const row of extensions.rows) {
    if (unsupportedExtensions.has(row.extname)) {
      issues.push({
        severity: 'error',
        table: 'N/A',
        message: `확장 '${row.extname}' 미지원 또는 부분 지원. 대안 확인 필요`
      });
    }
  }

  // 3. 트랜잭션 격리 수준 확인
  // CockroachDB는 Serializable만 지원
  issues.push({
    severity: 'info',
    table: 'N/A',
    message: `CockroachDB는 SERIALIZABLE 격리만 지원. ` +
      `READ COMMITTED는 v24.x에서 프리뷰 제공. ` +
      `트랜잭션 재시도 로직 필수 구현 필요`
  });

  return issues;
}
```

> **왜 UUID인가?** 분산 시스템에서 순차 ID(SERIAL, AUTO_INCREMENT)는 ID 생성이 단일 지점에 집중되어 핫스팟을 만듭니다. UUID는 각 노드가 독립적으로 생성할 수 있어 쓰기 부하가 균등하게 분산됩니다. CockroachDB에서 `gen_random_uuid()`는 UUIDv4를 생성하며, K-Sortable UUID가 필요하면 `gen_random_ulid()`를 사용할 수 있습니다.

## 어떤 것을 선택해야 할까?

최종 선택은 여러분의 상황에 달려 있습니다. 기술적 우열이 아니라, 워크로드 특성과 조직 상황에 따른 적합성의 문제입니다.

**TiDB를 선택해야 할 때:**
- 기존 MySQL 기반 시스템을 마이그레이션할 때
- HTAP 워크로드가 중요할 때 (TiFlash의 실시간 분석)
- 단일 리전에서 최대 OLTP 처리량이 필요할 때
- 대규모 배치 처리가 빈번할 때
- 이미 MySQL 생태계 도구에 익숙한 팀일 때

**CockroachDB를 선택해야 할 때:**
- 글로벌 멀티 리전이 1순위 요구사항일 때
- PostgreSQL 호환성과 고급 SQL 기능이 필요할 때
- 최고 수준의 데이터 정합성(Serializable)이 필수일 때
- 금융, 의료 등 규제 산업에서 데이터 레지던시가 중요할 때
- 운영 복잡도를 최소화하고 싶을 때 (균질적 노드)

두 시스템 모두 2025년 현재 충분히 프로덕션 레디입니다. TiDB는 글로벌에서 수천 개 기업이 사용 중이며, CockroachDB는 Fortune 500 기업 다수가 채택했습니다. 핵심은 **여러분의 워크로드를 정확히 이해하고, PoC(Proof of Concept)를 통해 실제 데이터로 검증하는 것**입니다.

## 결론

분산 SQL 데이터베이스는 더 이상 실험적 기술이 아닙니다. TiDB와 CockroachDB는 각각 MySQL과 PostgreSQL 호환성을 바탕으로, 기존 애플리케이션의 마이그레이션 장벽을 획기적으로 낮추면서도 진정한 수평 확장성을 제공합니다.

이 글에서 살펴본 것처럼, 두 시스템의 차이는 "어떤 것이 더 좋은가"의 문제가 아닙니다. TiDB의 분리형 아키텍처와 HTAP 역량은 분석과 트랜잭션을 동시에 처리해야 하는 환경에서 빛나고, CockroachDB의 균질적 노드와 선언적 멀티 리전 추상화는 글로벌 서비스의 복잡도를 극적으로 줄여줍니다.

어떤 시스템을 선택하든, 분산 데이터베이스 도입은 단순한 인프라 변경이 아니라 애플리케이션 설계 사고방식의 전환을 요구합니다. 핫스팟을 피하는 키 설계, 트랜잭션 재시도 로직, 데이터 지역성을 고려한 스키마 설계 — 이 모든 것이 분산 환경에서의 성능과 안정성을 좌우합니다. PoC부터 시작하여, 점진적으로 워크로드를 전환하는 것이 가장 안전한 접근입니다.
