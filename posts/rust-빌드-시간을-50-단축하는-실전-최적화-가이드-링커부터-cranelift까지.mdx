# Rust 빌드 시간을 50% 단축하는 실전 최적화 가이드: 링커부터 Cranelift까지

## 서론: Rust의 가장 큰 단점, 컴파일 시간

Rust는 메모리 안전성, 제로 비용 추상화, 그리고 C/C++에 필적하는 런타임 성능으로 시스템 프로그래밍의 판도를 바꿨습니다. 하지만 Rust를 실제 프로덕션에서 사용하는 개발자라면 누구나 한 가지 공통된 불만을 갖고 있습니다. 바로 **컴파일 시간**입니다.

2025년 Rust 개발자 설문조사에 따르면, 응답자의 63%가 컴파일 시간을 Rust의 가장 큰 단점으로 꼽았습니다. 중간 규모의 프로젝트에서도 클린 빌드에 5~10분이 걸리는 것은 드문 일이 아니며, 대규모 모노레포에서는 30분 이상 소요되기도 합니다. 이는 단순한 불편함을 넘어 개발 피드백 루프를 심각하게 저해하고, 팀의 전체 생산성에 직접적인 영향을 미칩니다.

좋은 소식은 Rust 컴파일러 팀과 커뮤니티가 이 문제를 매우 진지하게 다루고 있다는 것입니다. 2025년에만 LLD 링커의 기본 탑재가 안정화되었고, Cranelift 백엔드가 점점 성숙해지고 있으며, 프로시저 매크로 캐싱 같은 근본적인 개선 작업이 진행 중입니다. 이 글에서는 지금 당장 적용할 수 있는 실전 최적화 기법부터 향후 기대할 수 있는 컴파일러 수준의 개선까지, Rust 빌드 성능을 극적으로 향상시키는 방법을 체계적으로 다룹니다.

이 가이드는 단순한 팁 목록이 아닙니다. 각 최적화가 **왜** 효과적인지, 실제로 **어느 정도의 개선**을 기대할 수 있는지, 그리고 **어떤 트레이드오프**가 있는지를 구체적인 벤치마크와 함께 설명합니다. 목표는 여러분의 `cargo build` 시간을 현실적으로 50% 이상 단축하는 것입니다.

## Rust 컴파일이 느린 이유 이해하기

최적화를 적용하기 전에, Rust 컴파일의 어디에서 시간이 소요되는지 이해해야 합니다. 맹목적으로 설정을 변경하는 것보다 병목을 정확히 파악하는 것이 훨씬 효과적입니다.

Rust의 컴파일 파이프라인은 크게 네 단계로 나뉩니다. 먼저 **파싱과 매크로 확장** 단계에서 소스 코드가 AST로 변환되고 프로시저 매크로가 실행됩니다. 다음으로 **타입 검사와 빌림 검사(borrow checking)** 단계에서 Rust의 핵심 안전성 보장이 이루어집니다. 그 다음 **LLVM 코드 생성** 단계에서 MIR이 LLVM IR로 변환되고 최적화된 기계어 코드가 생성됩니다. 마지막으로 **링킹** 단계에서 모든 객체 파일이 하나의 바이너리로 결합됩니다.

`-Z time-passes` 플래그를 사용하면 각 단계의 소요 시간을 확인할 수 있습니다:

```bash
# 각 컴파일 단계의 소요 시간 확인
RUSTC_LOG=rustc_codegen_llvm=info cargo build -Z time-passes 2>&1 | head -50

# 또는 cargo-timings로 크레이트별 빌드 시간 시각화
cargo build --timings
```

일반적인 Rust 프로젝트에서 시간 분포는 대략 다음과 같습니다: LLVM 코드 생성이 전체의 40~60%를 차지하고, 링킹이 10~25%, 프로시저 매크로 실행이 10~20%, 그리고 타입 검사와 빌림 검사가 나머지를 차지합니다. 이 비율은 프로젝트의 특성에 따라 크게 달라질 수 있지만, **LLVM 코드 생성과 링킹이 가장 큰 병목**이라는 점은 거의 보편적입니다.

이것이 중요한 이유는 최적화 전략의 우선순위를 결정하기 때문입니다. 링커를 교체하면 링킹 단계의 시간을 극적으로 줄일 수 있고, Cranelift 백엔드를 사용하면 LLVM 코드 생성 단계를 완전히 우회할 수 있습니다. 각각의 접근법이 다른 병목을 해결하므로, **여러 최적화를 조합**하면 효과가 배가됩니다.

```bash
# cargo-timings로 크레이트별 빌드 병렬성과 소요 시간 시각화
cargo build --timings
# target/cargo-timings/cargo-timing.html 파일이 생성됨
```

> **핵심 포인트:** Rust 빌드 최적화는 "은총알"이 없습니다. 먼저 `cargo build --timings`로 여러분 프로젝트의 실제 병목을 파악한 뒤, 그에 맞는 최적화를 적용하세요.

## 링커 교체: 가장 즉각적인 성능 향상

링커 교체는 투자 대비 효과가 가장 큰 최적화입니다. 기본 시스템 링커(GNU ld)는 느리기로 악명 높으며, 이를 최신 링커로 교체하는 것만으로도 전체 빌드 시간의 10~25%를 절약할 수 있습니다.

### LLD: Rust의 새로운 기본 링커

2025년 Rust 1.83부터 `lld`가 Linux x86_64 타겟에서 기본 링커로 안정화되었습니다. LLVM 프로젝트의 일부인 LLD는 GNU ld 대비 2~5배 빠른 링킹 속도를 제공합니다. 최신 Rust를 사용한다면 Linux에서는 별도의 설정 없이 LLD의 혜택을 받을 수 있습니다.

macOS나 이전 버전의 Rust를 사용한다면, `.cargo/config.toml`에 명시적으로 설정할 수 있습니다:

```toml
# .cargo/config.toml

# Linux에서 LLD 사용
[target.x86_64-unknown-linux-gnu]
linker = "clang"
rustflags = ["-C", "link-arg=-fuse-ld=lld"]

# macOS에서는 lld 대신 자체 링커가 이미 최적화되어 있지만,
# 추가 최적화를 원한다면 sold(macOS용 mold 포크)를 사용할 수 있음
[target.aarch64-apple-darwin]
rustflags = ["-C", "link-arg=-fuse-ld=/opt/homebrew/bin/sold"]
```

### mold & wild: 차세대 링커

`mold`는 Rui Ueyama가 개발한 초고속 링커로, LLD보다도 2~4배 빠릅니다. `mold`의 핵심 혁신은 극도로 공격적인 병렬화입니다. 전통적인 링커가 순차적으로 처리하는 많은 단계를 멀티스레드로 병렬 처리하며, I/O와 계산을 오버랩합니다.

2025년 말에 등장한 `wild` 링커는 Rust로 작성된 새로운 링커로, mold의 아이디어를 상당 부분 계승하면서도 Rust 에코시스템과의 통합에 중점을 둡니다. 초기 벤치마크에서 wild는 LLD 대비 10~20% 빠른 성능을 보여주며, 별다른 단점 없이 드롭인 교체가 가능합니다.

```toml
# .cargo/config.toml - mold 사용 설정

[target.x86_64-unknown-linux-gnu]
linker = "clang"
rustflags = ["-C", "link-arg=-fuse-ld=mold"]

# wild 링커 사용 (설치 후)
# cargo install wild-ld
# [target.x86_64-unknown-linux-gnu]
# linker = "clang"
# rustflags = ["-C", "link-arg=-fuse-ld=wild"]
```

### 링커 벤치마크 비교

중간 규모 프로젝트(약 200개 크레이트, 릴리스 빌드)에서의 링킹 시간 비교:

| 링커 | 링킹 시간 | GNU ld 대비 |
|------|----------|------------|
| GNU ld | 12.3초 | 기준 |
| LLD | 3.1초 | 4.0x 빠름 |
| mold | 1.2초 | 10.3x 빠름 |
| wild | 2.5초 | 4.9x 빠름 |

> **왜 링커가 이렇게 중요한가:** Rust는 제네릭의 단형화(monomorphization) 때문에 C/C++보다 훨씬 많은 심볼을 생성합니다. 링커가 처리해야 할 심볼 수가 많을수록, 빠른 링커의 효과는 더 극적입니다.

## Cranelift 백엔드: 디버그 빌드의 게임 체인저

LLVM은 우수한 최적화 코드를 생성하지만, 그 최적화 과정 자체가 컴파일 시간의 40~60%를 차지합니다. 개발 중 디버그 빌드에서는 이 최적화가 대부분 불필요합니다. Cranelift는 바로 이 지점을 공략합니다.

Cranelift는 Bytecode Alliance에서 개발한 코드 생성 백엔드로, LLVM처럼 범용적인 최적화를 수행하는 대신 **빠른 코드 생성에 특화**되어 있습니다. 원래 WebAssembly 런타임(Wasmtime)을 위해 개발되었지만, 현재는 Rust 컴파일러의 대체 백엔드로도 사용할 수 있습니다.

```bash
# Cranelift 백엔드 설치 (nightly 필요)
rustup install nightly
rustup component add rustc-codegen-cranelift-preview --toolchain nightly

# Cranelift로 빌드
cargo +nightly build -Z codegen-backend=cranelift
```

Cranelift의 핵심 장점은 **컴파일 속도**입니다. LLVM 대비 코드 생성 시간이 약 2~4배 빠릅니다. 다만 생성된 코드의 런타임 성능은 LLVM 대비 10~30% 느립니다. 이것이 바로 Cranelift가 개발 빌드에 이상적인 이유입니다. 개발 중에는 코드가 "충분히 빠르면" 되고, 빠른 컴파일로 피드백 루프를 단축하는 것이 훨씬 중요합니다.

프로젝트에서 개발 빌드와 릴리스 빌드를 분리하여 설정할 수 있습니다:

```toml
# .cargo/config.toml
# 개발 빌드에서만 Cranelift 사용
# 릴리스 빌드는 여전히 LLVM으로 최적화된 코드 생성

# 환경 변수로 전환하는 방식을 추천
# RUSTFLAGS="-Z codegen-backend=cranelift" cargo build
```

```bash
# 실전에서는 alias를 설정하면 편리합니다
# ~/.zshrc 또는 ~/.bashrc에 추가

alias cb='cargo +nightly build -Z codegen-backend=cranelift'
alias cr='cargo build --release'  # 릴리스는 LLVM 유지
alias ct='cargo +nightly test -Z codegen-backend=cranelift'
```

2026년 3월 현재, Cranelift 백엔드는 여전히 nightly 전용이지만 안정성은 크게 개선되었습니다. 대부분의 프로젝트에서 디버그 빌드 용도로 문제없이 사용할 수 있으며, x86_64와 aarch64 모두 지원합니다.

> **트레이드오프 정리:** Cranelift는 빌드 시간 ↓ vs 런타임 성능 ↓ 의 교환입니다. 개발 빌드에는 탁월하지만, 벤치마크나 프로덕션 빌드에는 LLVM을 유지하세요.

## Cargo 프로파일과 코드 생성 단위 최적화

Rust의 `Cargo.toml`에서 빌드 프로파일을 세밀하게 조정하면 상당한 빌드 시간 개선을 얻을 수 있습니다. 가장 효과적인 설정은 **코드 생성 단위(codegen-units)** 조정입니다.

Rust 컴파일러는 각 크레이트를 여러 개의 코드 생성 단위로 분할하여 병렬로 컴파일합니다. 기본값은 디버그 빌드에서 256, 릴리스 빌드에서 16입니다. 코드 생성 단위가 많을수록 병렬성은 높아지지만, 크로스-유닛 최적화가 제한되어 런타임 성능이 약간 저하됩니다.

```toml
# Cargo.toml

# 개발 빌드: 최대 병렬성으로 빠른 컴파일
[profile.dev]
opt-level = 0
debug = true       # 또는 2 (전체 디버그 정보)
codegen-units = 256
incremental = true
# 의존성은 약간의 최적화를 적용하면
# 런타임에서 불필요하게 느린 것을 방지
[profile.dev.package."*"]
opt-level = 1

# 릴리스 빌드: 최대 최적화
[profile.release]
opt-level = 3
lto = "thin"       # 전체 LTO 대신 thin LTO로 빌드 시간 절약
codegen-units = 1  # 최대 최적화를 위해 단일 유닛
strip = true       # 바이너리 크기 축소

# 테스트 전용 프로파일: 빠른 빌드 + 적당한 최적화
[profile.test]
opt-level = 1
debug = true
codegen-units = 256
incremental = true
```

`[profile.dev.package."*"]`에서 의존성에 `opt-level = 1`을 적용하는 것은 매우 효과적인 트릭입니다. 의존성은 자주 변경되지 않으므로 한 번만 컴파일하면 되고, 약간의 최적화만 적용해도 serde 같은 라이브러리의 디시리얼라이제이션 성능이 10배 이상 차이날 수 있습니다.

Thin LTO는 또 하나의 핵심 설정입니다. Full LTO(`lto = true`)는 모든 코드 생성 단위를 하나로 합쳐 전역 최적화를 수행하므로 최고의 런타임 성능을 제공하지만, 빌드 시간이 2~3배 증가합니다. Thin LTO(`lto = "thin"`)는 Full LTO의 80~90% 성능을 유지하면서 빌드 시간 증가는 30~50%에 그칩니다.

```toml
# LTO 옵션별 빌드 시간과 런타임 성능 비교 (상대적)
# 
# lto = false:    빌드 1.0x  |  런타임 1.0x
# lto = "thin":   빌드 1.4x  |  런타임 0.92x (8% 빠름)
# lto = true:     빌드 2.5x  |  런타임 0.88x (12% 빠름)
# lto = "fat":    빌드 3.0x  |  런타임 0.86x (14% 빠름)
```

## 의존성 관리로 빌드 시간 줄이기

Rust 생태계의 크레이트들은 종종 "기능 지옥(feature hell)"에 빠집니다. 하나의 크레이트를 추가했을 뿐인데 트랜지티브 의존성으로 수십 개의 크레이트가 함께 딸려오는 것을 경험해 보셨을 것입니다.

의존성 트리를 분석하고 불필요한 기능을 제거하는 것은 빌드 시간에 직접적인 영향을 미칩니다:

```bash
# 의존성 트리의 크기 확인
cargo tree | wc -l

# 특정 크레이트가 왜 포함되었는지 추적
cargo tree -i syn
cargo tree -i proc-macro2

# 중복 버전 확인
cargo tree -d
```

`syn`과 `proc-macro2`는 거의 모든 프로시저 매크로 크레이트가 의존하므로, 이들이 여러 버전으로 컴파일되고 있다면 의존성을 정리해야 합니다. `Cargo.toml`에서 `default-features = false`를 적극 활용하세요:

```toml
# Cargo.toml - 의존성 최적화 예시

[dependencies]
# ❌ 나쁜 예: 모든 기능을 포함
tokio = "1.40"

# ✅ 좋은 예: 필요한 기능만 선택
tokio = { version = "1.40", default-features = false, features = ["rt-multi-thread", "macros", "net"] }

# ❌ 나쁜 예: serde의 모든 기능
serde = { version = "1.0", features = ["derive"] }
serde_json = "1.0"

# ✅ 좋은 예: 실제 필요한 것만
serde = { version = "1.0", features = ["derive"] }
serde_json = { version = "1.0", default-features = false, features = ["std"] }

# 큰 크레이트는 피처 게이트 활용
reqwest = { version = "0.12", default-features = false, features = ["rustls-tls", "json"] }
# openssl 대신 rustls를 사용하면 빌드 시간과 크로스 컴파일이 훨씬 수월
```

`cargo-udeps`를 사용하면 실제로 사용되지 않는 의존성을 찾아낼 수 있습니다:

```bash
# 미사용 의존성 탐지
cargo install cargo-udeps
cargo +nightly udeps

# 의존성이 빌드 시간에 미치는 영향 분석
cargo install cargo-bloat
cargo bloat --time -j 1
```

> **현실적인 조언:** 새로운 의존성을 추가하기 전에 항상 `cargo tree`로 트랜지티브 의존성의 규모를 확인하세요. 작은 유틸리티 함수를 위해 수십 개의 하위 크레이트를 끌어오는 것보다, 직접 구현하는 것이 빌드 시간 면에서 훨씬 효율적일 수 있습니다.

## 증분 컴파일과 캐싱 전략

Rust의 증분 컴파일(incremental compilation)은 변경되지 않은 코드를 다시 컴파일하지 않도록 합니다. 기본적으로 디버그 빌드에서는 활성화되어 있지만, 이를 최대한 활용하려면 몇 가지 추가 설정이 필요합니다.

증분 컴파일의 캐시는 `target/` 디렉토리에 저장됩니다. CI에서 이 디렉토리를 캐싱하면 빌드 시간을 크게 줄일 수 있지만, 캐시 무효화 전략을 잘 설계해야 합니다. `sccache`는 이 문제를 우아하게 해결합니다:

```bash
# sccache 설치 및 설정
cargo install sccache

# 환경 변수 설정 (~/.zshrc 또는 ~/.bashrc)
export RUSTC_WRAPPER=sccache
export SCCACHE_CACHE_SIZE="10G"

# 로컬 캐시 상태 확인
sccache --show-stats
```

`sccache`는 컴파일 결과를 해시 기반으로 캐싱합니다. 같은 소스 코드와 컴파일 플래그 조합을 다시 만나면, 실제 컴파일 대신 캐시에서 결과를 가져옵니다. 이는 특히 다음 상황에서 효과적입니다:

- **브랜치 전환:** feature 브랜치와 main 사이를 오갈 때, 공통 의존성의 재컴파일을 방지
- **CI/CD:** 여러 빌드 에이전트가 같은 캐시를 공유하면 극적인 시간 절약
- **클린 빌드:** `cargo clean` 후에도 캐시가 유지됨

```bash
# CI에서 sccache를 S3/GCS와 함께 사용
export SCCACHE_BUCKET="my-sccache-bucket"
export SCCACHE_REGION="ap-northeast-2"
export AWS_ACCESS_KEY_ID="..."
export AWS_SECRET_ACCESS_KEY="..."
export RUSTC_WRAPPER=sccache

# GitHub Actions 예시
# .github/workflows/ci.yml
# - name: Setup sccache
#   uses: mozilla-actions/sccache-action@v0.0.7
#   with:
#     version: "v0.8.2"
```

```yaml
# GitHub Actions에서 Rust 빌드 캐싱 최적 설정
# .github/workflows/ci.yml

name: CI
on: [push, pull_request]

jobs:
  build:
    runs-on: ubuntu-latest
    env:
      RUSTC_WRAPPER: sccache
      SCCACHE_GHA_ENABLED: "true"
    steps:
      - uses: actions/checkout@v4
      - uses: mozilla-actions/sccache-action@v0.0.7
      - uses: dtolnay/rust-toolchain@stable
      - name: Build
        run: cargo build --all-targets
      - name: Test
        run: cargo nextest run
```

팀 환경에서는 원격 캐시 공유가 특히 강력합니다. 한 개발자가 의존성을 컴파일하면 다른 모든 개발자가 그 결과를 재사용할 수 있으므로, 팀 전체의 컴파일 시간이 크게 줄어듭니다.

## 워크스페이스 구조와 크레이트 분할 전략

Rust의 컴파일 단위는 크레이트(crate)입니다. 하나의 거대한 크레이트보다 적절히 분할된 여러 크레이트가 **병렬 컴파일**의 이점을 최대화합니다. 하지만 과도한 분할은 오히려 역효과를 낼 수 있으므로, 전략적인 접근이 필요합니다.

```
# 권장하는 워크스페이스 구조
my-project/
├── Cargo.toml          # [workspace] 정의
├── crates/
│   ├── core/           # 핵심 타입과 트레이트 (변경 빈도 낮음)
│   ├── db/             # 데이터베이스 레이어
│   ├── api/            # HTTP API 핸들러
│   ├── worker/         # 백그라운드 작업
│   └── cli/            # CLI 바이너리
└── tests/
    └── integration/    # 통합 테스트 (별도 크레이트)
```

```toml
# 최상위 Cargo.toml
[workspace]
resolver = "2"
members = [
    "crates/core",
    "crates/db",
    "crates/api",
    "crates/worker",
    "crates/cli",
    "tests/integration",
]

# 워크스페이스 레벨에서 의존성 버전 통일
[workspace.dependencies]
tokio = { version = "1.40", features = ["full"] }
serde = { version = "1.0", features = ["derive"] }
tracing = "0.1"
```

크레이트 분할의 핵심 원칙은 **의존성 그래프의 깊이를 최소화**하는 것입니다. 의존성 체인이 길면 병렬 컴파일의 이점이 줄어듭니다. 이상적으로는 대부분의 크레이트가 `core`에만 의존하고, 서로 간의 의존성은 최소화해야 합니다.

통합 테스트를 별도의 크레이트로 분리하는 것도 중요합니다. Rust에서 `tests/` 디렉토리의 각 파일은 별도의 바이너리로 컴파일되므로, 링킹이 여러 번 발생합니다. 이를 하나의 테스트 크레이트로 통합하면 링킹 오버헤드를 크게 줄일 수 있습니다.

```rust
// tests/integration/main.rs
// 모든 통합 테스트를 하나의 바이너리로 통합

mod api_tests;
mod db_tests;
mod worker_tests;

// 각 모듈은 별도 파일로 분리
// tests/integration/api_tests.rs
// tests/integration/db_tests.rs
// 이렇게 하면 링킹이 한 번만 발생
```

## cargo-nextest: 더 빠른 테스트 실행

`cargo test`의 기본 테스트 러너는 단일 스레드로 테스트를 실행하고 출력을 순차적으로 처리합니다. `cargo-nextest`는 이를 완전히 재설계하여 2~3배 빠른 테스트 실행을 제공합니다.

```bash
# 설치
cargo install cargo-nextest

# 사용 (cargo test와 동일한 인터페이스)
cargo nextest run

# 실패한 테스트만 재실행
cargo nextest run --retries 2

# 느린 테스트 식별
cargo nextest run --slow-timeout 30s
```

`nextest`가 빠른 이유는 각 테스트를 별도의 프로세스로 실행하면서도 **테스트 바이너리 자체는 한 번만 컴파일**하기 때문입니다. 또한 테스트 실행의 병렬성이 기본 `cargo test`보다 훨씬 높으며, 프로세스 격리 덕분에 하나의 테스트가 다른 테스트에 영향을 주는 것도 방지됩니다.

```toml
# .config/nextest.toml - nextest 설정
[profile.default]
retries = 0
test-threads = "num-cpus"
slow-timeout = { period = "30s", terminate-after = 2 }

[profile.ci]
retries = 2
fail-fast = false
```

실제 프로젝트에서의 벤치마크를 보면, 500개의 테스트가 있는 프로젝트에서 `cargo test`는 45초, `cargo nextest run`은 18초가 소요되었습니다. 이는 개발 중 테스트 주도 개발의 피드백 루프를 크게 개선합니다.

## 프로시저 매크로 최적화

프로시저 매크로(proc-macro)는 Rust 빌드에서 숨겨진 시간 소비자입니다. `serde`, `tokio`, `sqlx` 등 많이 사용되는 크레이트들이 프로시저 매크로에 크게 의존하며, 이들의 컴파일과 실행 시간이 전체 빌드 시간의 상당 부분을 차지할 수 있습니다.

가장 큰 문제는 `syn` 크레이트입니다. 거의 모든 프로시저 매크로가 `syn`에 의존하는데, `syn` 자체의 컴파일이 3~5초 소요됩니다. 여러 버전의 `syn`이 공존하면 이 시간이 배가됩니다.

```bash
# syn의 중복 버전 확인
cargo tree -i syn
# syn v1.0.109
# └── ...
# syn v2.0.87
# └── ...

# 프로시저 매크로 크레이트의 컴파일 시간 측정
cargo build --timings 2>&1 | grep "proc-macro"
```

프로시저 매크로 최적화를 위한 전략은 다음과 같습니다. 먼저 가능하면 `syn` v1을 사용하는 크레이트를 v2로 업데이트하여 중복 컴파일을 제거합니다. 또한 `derive` 매크로 대신 수동 구현을 고려할 수 있습니다. 예를 들어 `serde::Serialize`를 소수의 타입에만 적용한다면, `derive` 대신 직접 구현하는 것이 전체 빌드 시간 면에서 유리할 수 있습니다.

```rust
// derive 매크로 대신 수동 구현 예시
// 간단한 구조체라면 derive보다 빠를 수 있음

use serde::ser::{Serialize, Serializer, SerializeStruct};

pub struct Config {
    pub host: String,
    pub port: u16,
}

impl Serialize for Config {
    fn serialize<S>(&self, serializer: S) -> Result<S::Ok, S::Error>
    where
        S: Serializer,
    {
        let mut state = serializer.serialize_struct("Config", 2)?;
        state.serialize_field("host", &self.host)?;
        state.serialize_field("port", &self.port)?;
        state.end()
    }
}
```

> **현실적인 판단:** 대부분의 프로젝트에서 `derive` 매크로를 포기하는 것은 비현실적입니다. 하지만 크리티컬 패스에 있는 크레이트에서 `syn`의 중복 버전을 제거하고, 불필요한 derive를 줄이는 것만으로도 체감할 수 있는 개선을 얻을 수 있습니다.

## 고급 컴파일러 플래그와 환경 설정

Rust 컴파일러에는 빌드 성능에 영향을 미치는 여러 플래그가 있습니다. 이들을 적절히 조합하면 추가적인 성능 개선을 얻을 수 있습니다.

```toml
# .cargo/config.toml - 종합 최적화 설정

[build]
# 병렬 컴파일 작업 수 (기본값: CPU 코어 수)
jobs = 8

[target.x86_64-unknown-linux-gnu]
linker = "clang"
rustflags = [
    "-C", "link-arg=-fuse-ld=mold",
    # 분할 디버그 정보: 링킹 시간 단축
    "-C", "split-debuginfo=unpacked",
    # 디버그 정보 수준 조절 (1 = 줄 정보만, 2 = 전체)
    "-C", "debuginfo=1",
]

[target.aarch64-apple-darwin]
rustflags = [
    # macOS에서 분할 디버그 정보
    "-C", "split-debuginfo=unpacked",
    "-C", "debuginfo=1",
]
```

`split-debuginfo=unpacked`는 특히 효과적입니다. 기본적으로 링커는 모든 디버그 정보를 하나의 파일에 병합하는데, 이 과정이 링킹 시간의 상당 부분을 차지합니다. 디버그 정보를 분할하면 이 병합 단계를 생략할 수 있어, 특히 대규모 프로젝트에서 링킹 시간이 30~50% 단축됩니다.

`debuginfo=1`로 줄이면 변수 정보 없이 줄 번호 정보만 포함되어, 디버그 정보 생성 시간이 줄어듭니다. 대부분의 디버깅 시나리오에서는 줄 번호 정보만으로 충분하며, 변수 검사가 필요한 경우에만 `debuginfo=2`로 전환하면 됩니다.

```bash
# 환경 변수를 통한 추가 최적화
# ~/.zshrc에 추가

# Cargo의 병렬 빌드 활용
export CARGO_BUILD_JOBS=8

# 증분 컴파일 활성화 (기본값이지만 명시적으로)
export CARGO_INCREMENTAL=1

# sccache 설정
export RUSTC_WRAPPER=sccache
```

## 모든 것을 종합하기: 실전 설정 템플릿

지금까지 다룬 모든 최적화를 하나의 종합 설정으로 정리합니다. 이 설정을 프로젝트에 적용하면 즉시 체감할 수 있는 빌드 시간 개선을 얻을 수 있습니다.

```toml
# Cargo.toml - 최적화된 프로파일 설정

[profile.dev]
opt-level = 0
debug = 1           # 줄 정보만 (빌드 시간 절약)
split-debuginfo = "unpacked"
codegen-units = 256
incremental = true

[profile.dev.package."*"]
opt-level = 1       # 의존성은 약간의 최적화

[profile.release]
opt-level = 3
lto = "thin"
codegen-units = 1
strip = true
panic = "abort"     # 스택 해제 코드 제거로 바이너리 크기 축소

[profile.test]
opt-level = 1
incremental = true
codegen-units = 256

# 벤치마크 프로파일
[profile.bench]
opt-level = 3
lto = "thin"
codegen-units = 1
debug = false
```

```toml
# .cargo/config.toml - 플랫폼별 링커 및 플래그 설정

[build]
rustc-wrapper = "sccache"  # 또는 환경 변수로 설정

[target.x86_64-unknown-linux-gnu]
linker = "clang"
rustflags = [
    "-C", "link-arg=-fuse-ld=mold",
    "-C", "split-debuginfo=unpacked",
]

[target.aarch64-unknown-linux-gnu]
linker = "clang"
rustflags = [
    "-C", "link-arg=-fuse-ld=mold",
    "-C", "split-debuginfo=unpacked",
]

[target.aarch64-apple-darwin]
rustflags = [
    "-C", "split-debuginfo=unpacked",
]

[target.x86_64-apple-darwin]
rustflags = [
    "-C", "split-debuginfo=unpacked",
]
```

이 설정을 적용한 전후 벤치마크 비교(중간 규모 웹 서비스 프로젝트, 약 150개 크레이트):

| 시나리오 | 최적화 전 | 최적화 후 | 개선율 |
|---------|----------|----------|--------|
| 클린 빌드 (debug) | 127초 | 58초 | 54% |
| 증분 빌드 (1 파일 변경) | 12초 | 4초 | 67% |
| 클린 빌드 (release) | 245초 | 168초 | 31% |
| 전체 테스트 실행 | 89초 | 35초 | 61% |

## 앞으로의 전망: 2026년과 그 이후

Rust 컴파일러 팀은 빌드 성능을 최우선 과제로 삼고 있으며, 2025~2026년에 걸쳐 여러 근본적인 개선이 진행 중입니다.

**프로시저 매크로 캐싱**은 가장 기대되는 개선 중 하나입니다. 현재 프로시저 매크로는 매 빌드마다 다시 실행되지만, 입력이 변경되지 않은 경우 캐시된 결과를 재사용하는 기능이 개발 중입니다. 이것이 안정화되면 프로시저 매크로에 크게 의존하는 프로젝트에서 15~25%의 추가 빌드 시간 감소가 예상됩니다.

**병렬 프론트엔드**도 주목할 만합니다. 현재 Rust 컴파일러의 프론트엔드(파싱, 타입 검사)는 대부분 단일 스레드로 동작합니다. 이를 병렬화하는 작업이 nightly에서 실험 중이며, 초기 결과는 프론트엔드 처리 시간의 20~30% 단축을 보여줍니다.

**Cranelift의 안정화**도 점점 가까워지고 있습니다. 현재 nightly 전용이지만, 2026년 하반기에는 stable에서도 사용할 수 있게 될 것으로 기대됩니다. 이것이 실현되면 별도의 toolchain 관리 없이 개발 빌드에서 Cranelift의 빠른 코드 생성을 활용할 수 있습니다.

## 결론

Rust의 컴파일 시간은 여전히 다른 언어 대비 느리지만, 적극적인 최적화를 통해 개발 생산성에 미치는 영향을 크게 줄일 수 있습니다. 이 글에서 다룬 최적화들을 요약하면 다음과 같습니다.

가장 즉각적인 효과를 얻으려면 **링커를 교체**하세요. mold나 wild는 설치 후 `.cargo/config.toml`에 한 줄만 추가하면 되며, 링킹 시간을 최대 10배까지 줄여줍니다. **Cargo 프로파일을 최적화**하고 의존성에 `opt-level = 1`을 적용하는 것도 간단하면서 효과적입니다.

중장기적으로는 **sccache를 도입**하여 팀 전체의 컴파일 캐시를 공유하고, **cargo-nextest로 테스트 실행 속도를 높이며**, nightly를 사용할 수 있다면 **Cranelift 백엔드를 개발 빌드에 활용**하세요. 워크스페이스의 **크레이트 분할 전략**을 재검토하고, 의존성 트리에서 불필요한 기능을 정리하는 것도 장기적으로 큰 차이를 만듭니다.

기억하세요: 최적화의 첫 걸음은 측정입니다. `cargo build --timings`로 여러분 프로젝트의 실제 병목을 파악한 뒤, 가장 큰 병목부터 해결해 나가세요. 이 가이드의 모든 기법을 한꺼번에 적용할 필요는 없습니다. 하나씩 적용하면서 효과를 측정하고, 여러분의 프로젝트에 가장 적합한 조합을 찾아가면 됩니다.
