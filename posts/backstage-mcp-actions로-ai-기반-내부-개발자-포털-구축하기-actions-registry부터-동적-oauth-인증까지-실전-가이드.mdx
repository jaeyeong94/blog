# Backstage MCP Actions로 AI 기반 내부 개발자 포털 구축하기

내부 개발자 포털(Internal Developer Portal, IDP)은 플랫폼 엔지니어링의 핵심 구성 요소입니다. 소프트웨어 카탈로그, 서비스 템플릿, 문서, CI/CD 파이프라인 등 수많은 도구와 정보를 한곳에 모아 개발자 경험을 통합하는 역할을 합니다. Spotify가 만들어 오픈소스로 공개한 Backstage는 이 분야에서 사실상의 표준으로 자리잡았으며, CNCF Graduated 프로젝트로서 수백 개 기업이 프로덕션 환경에서 운영하고 있습니다.

하지만 2025년 말부터 Backstage에 흥미로운 변화가 일어나고 있습니다. AI 에이전트가 개발자 포털과 직접 상호작용할 수 있는 Model Context Protocol(MCP) 통합이 도입된 것입니다. 이제 Cursor, Claude Desktop, ChatGPT 같은 AI 도구에서 Backstage의 액션을 직접 호출할 수 있습니다. 더 이상 브라우저를 열고 UI를 클릭할 필요 없이, AI 에이전트에게 "프로덕션 서비스의 상태를 확인해줘"라고 말하면 Backstage 카탈로그에서 데이터를 가져와 답변합니다.

이 글에서는 Backstage v1.40에서 도입된 Actions Registry와 MCP 서버 통합의 아키텍처를 심층 분석하고, v1.43에서 추가된 동적 클라이언트 등록(Dynamic Client Registration)을 통한 OAuth 기반 사용자 인증까지 다룹니다. 실제 프로덕션 환경에서 AI 에이전트가 안전하게 플랫폼 액션을 호출할 수 있는 시스템을 단계별로 구축해 보겠습니다.

## Actions Registry: 분산된 플러그인 액션의 통합

### 기존 Backstage 플러그인의 한계

Backstage는 플러그인 아키텍처로 설계되어 있습니다. Kubernetes 플러그인, GitHub Actions 플러그인, PagerDuty 플러그인 등 각각이 독립적인 API 엔드포인트를 제공합니다. 문제는 이러한 플러그인들의 기능이 각 플러그인 내부에 잠겨 있다는 것입니다. Kubernetes 플러그인이 제공하는 "Pod 재시작" 기능을 다른 곳에서 재사용하려면 해당 플러그인의 API를 직접 호출해야 했고, 이를 위한 표준화된 방법이 없었습니다.

Scaffolder(소프트웨어 템플릿 엔진)가 이 문제를 부분적으로 해결했습니다. `createTemplateAction`으로 정의된 액션은 템플릿 워크플로우에서 재사용할 수 있었지만, 이것은 Scaffolder 컨텍스트에서만 동작했습니다. 플러그인 자체의 비즈니스 로직을 범용적으로 노출하는 메커니즘은 존재하지 않았습니다.

### Actions Registry의 등장

v1.40에서 도입된 Actions Registry는 이 문제를 근본적으로 해결합니다. 모든 백엔드 플러그인이 자신의 액션을 중앙 레지스트리에 등록하면, 해당 액션은 어디서든 발견하고 호출할 수 있게 됩니다. 액션 정의는 Zod 스키마로 타입 안전하게 이루어지며, 입출력 스키마, 설명, 권한 정책이 함께 등록됩니다.

```typescript
// plugins/kubernetes-backend/src/actions.ts
import { createBackendModule } from '@backstage/backend-plugin-api';
import { actionsRegistryExtensionPoint } from '@backstage/plugin-actions-node';
import { z } from 'zod';

export const kubernetesActionsModule = createBackendModule({
  pluginId: 'kubernetes',
  moduleId: 'actions',
  register(reg) {
    reg.registerInit({
      deps: {
        actionsRegistry: actionsRegistryExtensionPoint,
      },
      async init({ actionsRegistry }) {
        // Pod 목록 조회 액션 등록
        actionsRegistry.register({
          name: 'list_pods',
          title: 'List Kubernetes Pods',
          description:
            'Returns a list of pods for a given namespace and cluster, ' +
            'including their status, restart count, and resource usage.',
          schema: {
            input: z => z.object({
              cluster: z.string().describe('Target cluster name'),
              namespace: z.string().describe('Kubernetes namespace'),
              labelSelector: z
                .string()
                .optional()
                .describe('Label selector to filter pods'),
            }),
            output: z => z.object({
              pods: z.array(z.object({
                name: z.string(),
                status: z.string(),
                restarts: z.number(),
                age: z.string(),
              })),
            }),
          },
          // 실제 핸들러 구현
          async handler(ctx) {
            const { cluster, namespace, labelSelector } = ctx.input;
            const k8sClient = await getClusterClient(cluster);
            const pods = await k8sClient.listNamespacedPod(
              namespace,
              undefined,
              undefined,
              undefined,
              undefined,
              labelSelector,
            );
            return {
              pods: pods.body.items.map(pod => ({
                name: pod.metadata?.name ?? 'unknown',
                status: pod.status?.phase ?? 'Unknown',
                restarts: pod.status?.containerStatuses?.[0]
                  ?.restartCount ?? 0,
                age: getAge(pod.metadata?.creationTimestamp),
              })),
            };
          },
        });

        // Pod 재시작 액션 등록
        actionsRegistry.register({
          name: 'restart_deployment',
          title: 'Restart Kubernetes Deployment',
          description:
            'Performs a rolling restart of a deployment by patching ' +
            'the pod template annotation with the current timestamp.',
          schema: {
            input: z => z.object({
              cluster: z.string().describe('Target cluster name'),
              namespace: z.string().describe('Kubernetes namespace'),
              deploymentName: z.string().describe('Deployment to restart'),
            }),
            output: z => z.object({
              success: z.boolean(),
              message: z.string(),
            }),
          },
          async handler(ctx) {
            const { cluster, namespace, deploymentName } = ctx.input;
            const k8sClient = await getClusterClient(cluster);
            await k8sClient.patchNamespacedDeployment(
              deploymentName,
              namespace,
              {
                spec: {
                  template: {
                    metadata: {
                      annotations: {
                        'kubectl.kubernetes.io/restartedAt':
                          new Date().toISOString(),
                      },
                    },
                  },
                },
              },
            );
            return {
              success: true,
              message: `Deployment ${deploymentName} restarted`,
            };
          },
        });
      },
    });
  },
});
```

이 코드에서 주목할 점은 **액션이 플러그인과 함께 자동으로 설치된다**는 것입니다. Kubernetes 백엔드 플러그인을 설치하면 `list_pods`와 `restart_deployment` 액션이 자동으로 Actions Registry에 등록됩니다. 별도의 설정이 필요 없습니다.

### Zod 스키마의 역할

Actions Registry가 Zod를 사용하는 이유는 단순한 타입 검증을 넘어섭니다. Zod 스키마는 JSON Schema로 자동 변환되며, 이 JSON Schema가 MCP 프로토콜을 통해 AI 에이전트에게 전달됩니다. AI 에이전트는 이 스키마를 읽고 각 액션의 입력 파라미터가 무엇인지, 어떤 타입인지, 필수인지 선택인지를 이해합니다. `.describe()` 메서드로 추가한 설명은 AI가 파라미터의 의미를 파악하는 데 핵심적인 역할을 합니다.

```typescript
// .describe()가 AI 에이전트의 파라미터 이해를 돕는 방식
const schema = z.object({
  // AI가 "어떤 클러스터?" 라고 묻는 대신 적절한 값을 추론할 수 있음
  cluster: z.string().describe(
    'Target Kubernetes cluster name. ' +
    'Common values: prod-us-east, staging-eu-west, dev-local'
  ),
  // 열거형으로 가능한 값을 제한
  environment: z.enum(['production', 'staging', 'development'])
    .describe('Target deployment environment'),
  // 선택적 파라미터와 기본값
  timeout: z.number()
    .optional()
    .default(30)
    .describe('Operation timeout in seconds'),
});
```

## MCP 서버 통합: AI 에이전트가 Backstage를 만나다

### Model Context Protocol이란

Model Context Protocol(MCP)은 Anthropic이 2024년 말에 공개한 오픈 프로토콜로, AI 모델이 외부 도구와 데이터 소스에 접근하는 방식을 표준화합니다. AI 에이전트가 사용할 수 있는 "도구"를 JSON-RPC 기반으로 정의하고, 도구의 스키마를 자동으로 AI에게 노출하며, AI가 도구 호출 결과를 받아 추론에 활용하는 방식입니다. MCP는 빠르게 업계 표준으로 자리잡아, Cursor, Claude Desktop, Windsurf, VS Code 등 주요 AI 도구들이 MCP 클라이언트를 내장하고 있습니다.

Backstage의 MCP 통합이 강력한 이유는 Actions Registry에 등록된 모든 액션이 자동으로 MCP 도구로 노출된다는 점입니다. 플러그인 개발자가 MCP에 대해 전혀 알 필요가 없습니다. 액션을 등록하면 MCP 서버가 알아서 처리합니다.

### MCP Actions 백엔드 플러그인 설치

```bash
# Backstage 프로젝트에 MCP 플러그인 추가
yarn --cwd packages/backend add @backstage/plugin-mcp-actions-backend
```

```typescript
// packages/backend/src/index.ts
import { createBackend } from '@backstage/backend-defaults';

const backend = createBackend();

// 기존 플러그인들
backend.add(import('@backstage/plugin-catalog-backend'));
backend.add(import('@backstage/plugin-scaffolder-backend'));
backend.add(import('@backstage/plugin-kubernetes-backend'));

// MCP Actions 백엔드 플러그인 추가
backend.add(import('@backstage/plugin-mcp-actions-backend'));

// 커스텀 액션 모듈 추가
backend.add(import('./plugins/kubernetes-actions'));
backend.add(import('./plugins/catalog-actions'));

backend.start();
```

```yaml
# app-config.yaml
mcp:
  actions:
    # MCP 서버 기본 설정
    server:
      # MCP 서버가 리스닝할 경로
      path: /api/mcp
      # SSE(Server-Sent Events) 트랜스포트 활성화
      transport: sse
    # 노출할 액션 필터 (선택사항)
    # 지정하지 않으면 모든 등록된 액션이 노출됨
    allowedActions:
      - 'kubernetes:list_pods'
      - 'kubernetes:restart_deployment'
      - 'catalog:get_entity'
      - 'catalog:list_entities'
      - 'scaffolder:execute_template'
```

이 설정만으로 Backstage 인스턴스가 MCP 서버로 동작합니다. AI 클라이언트에서 연결하면 등록된 모든 액션을 도구로 사용할 수 있습니다.

### AI 클라이언트 설정

Claude Desktop에서 Backstage MCP 서버에 연결하는 설정은 매우 간단합니다:

```json
{
  "mcpServers": {
    "backstage": {
      "url": "https://backstage.internal.company.com/api/mcp/sse",
      "headers": {
        "Authorization": "Bearer ${BACKSTAGE_TOKEN}"
      }
    }
  }
}
```

Cursor에서는 `.cursor/mcp.json` 파일로 설정합니다:

```json
{
  "mcpServers": {
    "backstage-platform": {
      "url": "https://backstage.internal.company.com/api/mcp/sse",
      "headers": {
        "Authorization": "Bearer ${BACKSTAGE_TOKEN}"
      }
    }
  }
}
```

연결이 완료되면 AI 에이전트는 Backstage에 등록된 모든 액션을 인식합니다. 예를 들어 Claude Desktop에서 "프로덕션 클러스터의 payment-service Pod 상태를 확인해줘"라고 요청하면, Claude는 자동으로 `kubernetes:list_pods` 액션을 호출하고 결과를 정리해서 보여줍니다.

## 동적 클라이언트 등록: 제로 트러스트 AI 인증

### 정적 토큰의 한계

v1.40의 초기 MCP 구현에서는 정적 토큰(static token) 기반 인증을 사용했습니다. 관리자가 미리 토큰을 생성하고, AI 클라이언트 설정에 해당 토큰을 넣는 방식이었습니다. 이 방식은 몇 가지 심각한 문제가 있었습니다.

첫째, 토큰이 특정 사용자에 연결되지 않으므로 감사 로그(audit log)에서 "MCP 서비스 계정이 배포를 실행함"이라고만 기록됩니다. 누가 실제로 요청했는지 알 수 없습니다. 둘째, 사용자별 권한 제어가 불가능합니다. 주니어 개발자의 AI 에이전트도 시니어 개발자와 동일한 권한으로 동작합니다. 셋째, 토큰이 유출되면 해당 토큰으로 모든 액션을 호출할 수 있습니다.

### v1.43의 Dynamic Client Registration

v1.43에서 도입된 동적 클라이언트 등록(Dynamic Client Registration)은 이 문제를 근본적으로 해결합니다. OAuth 2.0 플로우를 통해 각 AI 클라이언트가 사용자의 실제 ID로 인증됩니다. 작동 흐름은 다음과 같습니다:

1. AI 에이전트(Claude Desktop, Cursor 등)가 MCP 액션을 호출하려고 시도합니다.
2. Backstage MCP 서버가 인증이 필요하다고 응답합니다.
3. 사용자의 브라우저가 열리며 Backstage 로그인 화면이 표시됩니다.
4. 사용자가 기존 SSO(GitHub, Google, Microsoft 등)로 로그인합니다.
5. MCP 클라이언트에 대한 접근 권한을 승인합니다.
6. AI 에이전트가 해당 사용자의 토큰으로 액션을 호출합니다.

```yaml
# app-config.yaml - 동적 클라이언트 등록 설정
auth:
  providers:
    # 기존 인증 프로바이더 설정
    github:
      development:
        clientId: ${GITHUB_CLIENT_ID}
        clientSecret: ${GITHUB_CLIENT_SECRET}

mcp:
  actions:
    server:
      path: /api/mcp
      transport: sse
    auth:
      # 동적 클라이언트 등록 활성화
      dynamicClientRegistration:
        enabled: true
        # 클라이언트 토큰 만료 시간
        tokenLifetimeMinutes: 60
        # 허용할 리다이렉트 URI 패턴
        allowedRedirectUris:
          - 'http://localhost:*'
          - 'vscode://*'
          - 'cursor://*'
```

이 설정으로 AI 클라이언트는 더 이상 사전 설정된 토큰 없이도 Backstage에 연결할 수 있습니다. 최초 연결 시 OAuth 플로우가 자동으로 시작되며, 사용자가 한 번 인증하면 토큰 만료까지 추가 인증 없이 사용할 수 있습니다.

### 권한 정책과 AI 액션의 통합

동적 인증의 핵심 가치는 Backstage의 기존 권한 시스템과 완전히 통합된다는 점입니다. Backstage Permission Framework에서 정의한 권한 정책이 MCP 호출에도 동일하게 적용됩니다.

```typescript
// plugins/permissions-backend/src/policy.ts
import {
  PolicyDecision,
  AuthorizeResult,
} from '@backstage/plugin-permission-common';
import {
  PermissionPolicy,
  PolicyQuery,
} from '@backstage/plugin-permission-node';
import { BackstageIdentityResponse } from '@backstage/plugin-auth-node';

export class PlatformPermissionPolicy implements PermissionPolicy {
  async handle(
    request: PolicyQuery,
    user?: BackstageIdentityResponse,
  ): Promise<PolicyDecision> {
    // MCP를 통한 호출이든 UI를 통한 호출이든 동일한 정책 적용
    const userGroups = user?.identity.ownershipEntityRefs ?? [];

    // 프로덕션 배포 관련 액션은 SRE 팀만 허용
    if (request.permission.name === 'kubernetes.restart_deployment') {
      const targetCluster = request.permission.attributes?.cluster;
      if (
        targetCluster?.startsWith('prod') &&
        !userGroups.includes('group:default/sre-team')
      ) {
        return { result: AuthorizeResult.DENY };
      }
    }

    // Scaffolder 템플릿 실행은 승인된 사용자만
    if (request.permission.name === 'scaffolder.execute_template') {
      if (!userGroups.includes('group:default/developers')) {
        return { result: AuthorizeResult.DENY };
      }
    }

    return { result: AuthorizeResult.ALLOW };
  }
}
```

이 권한 정책에서 주목할 점은 **MCP를 통한 호출과 웹 UI를 통한 호출이 완전히 동일한 권한 검사를 받는다**는 것입니다. 주니어 개발자가 AI 에이전트에게 "프로덕션 클러스터의 payment-service를 재시작해줘"라고 요청하더라도, 해당 개발자가 SRE 팀에 속하지 않으면 액션이 거부됩니다. AI 에이전트는 특권적 존재가 아니라 사용자의 권한을 그대로 물려받는 인터페이스일 뿐입니다.

## 커스텀 액션 개발: 실전 플랫폼 액션 구축

### 서비스 카탈로그 액션

실제 프로덕션에서 가장 많이 사용되는 액션은 카탈로그 관련 액션입니다. 서비스 소유자 확인, 의존성 그래프 조회, 서비스 메타데이터 검색 등이 여기에 해당합니다.

```typescript
// plugins/catalog-actions/src/module.ts
import { createBackendModule } from '@backstage/backend-plugin-api';
import { actionsRegistryExtensionPoint } from '@backstage/plugin-actions-node';
import { catalogServiceRef } from '@backstage/plugin-catalog-node';
import { z } from 'zod';

export const catalogActionsModule = createBackendModule({
  pluginId: 'catalog',
  moduleId: 'platform-actions',
  register(reg) {
    reg.registerInit({
      deps: {
        actionsRegistry: actionsRegistryExtensionPoint,
        catalog: catalogServiceRef,
      },
      async init({ actionsRegistry, catalog }) {
        // 서비스 정보 조회
        actionsRegistry.register({
          name: 'get_service_info',
          title: 'Get Service Information',
          description:
            'Returns comprehensive information about a service ' +
            'including owner, lifecycle, dependencies, and metadata.',
          schema: {
            input: z => z.object({
              serviceName: z.string()
                .describe('Name of the service in the catalog'),
              includeRelations: z.boolean()
                .optional()
                .default(true)
                .describe('Include dependency and API relations'),
            }),
            output: z => z.object({
              name: z.string(),
              owner: z.string(),
              lifecycle: z.string(),
              description: z.string().optional(),
              tags: z.array(z.string()),
              dependencies: z.array(z.string()),
              apis: z.array(z.string()),
              techStack: z.array(z.string()),
            }),
          },
          async handler(ctx) {
            const entity = await catalog.getEntityByRef(
              `component:default/${ctx.input.serviceName}`,
              { credentials: ctx.credentials },
            );

            if (!entity) {
              throw new Error(
                `Service ${ctx.input.serviceName} not found`,
              );
            }

            const relations = ctx.input.includeRelations
              ? entity.relations ?? []
              : [];

            return {
              name: entity.metadata.name,
              owner: (
                entity.spec as { owner?: string }
              )?.owner ?? 'unknown',
              lifecycle: (
                entity.spec as { lifecycle?: string }
              )?.lifecycle ?? 'unknown',
              description: entity.metadata.description,
              tags: entity.metadata.tags ?? [],
              dependencies: relations
                .filter(r => r.type === 'dependsOn')
                .map(r => r.targetRef),
              apis: relations
                .filter(r => r.type === 'providesApi')
                .map(r => r.targetRef),
              techStack: (
                entity.metadata.annotations?.[
                  'backstage.io/tech-stack'
                ] ?? ''
              ).split(',').filter(Boolean),
            };
          },
        });

        // 온콜 담당자 조회
        actionsRegistry.register({
          name: 'get_oncall',
          title: 'Get On-Call Engineer',
          description:
            'Returns the current on-call engineer for a given ' +
            'service based on PagerDuty integration.',
          schema: {
            input: z => z.object({
              serviceName: z.string()
                .describe('Name of the service'),
            }),
            output: z => z.object({
              engineer: z.string(),
              escalationLevel: z.number(),
              schedule: z.string(),
              contactSlack: z.string().optional(),
            }),
          },
          async handler(ctx) {
            const entity = await catalog.getEntityByRef(
              `component:default/${ctx.input.serviceName}`,
              { credentials: ctx.credentials },
            );
            const pagerdutyServiceId =
              entity?.metadata.annotations?.[
                'pagerduty.com/service-id'
              ];

            // PagerDuty API로 온콜 정보 조회
            const oncall = await fetchPagerDutyOncall(
              pagerdutyServiceId,
            );
            return oncall;
          },
        });
      },
    });
  },
});
```

### 인프라 프로비저닝 액션

Crossplane이나 Terraform과 연동하여 인프라를 프로비저닝하는 액션도 MCP를 통해 노출할 수 있습니다. 개발자가 AI 에이전트에게 "staging 환경에 Redis 클러스터를 생성해줘"라고 요청하면, 적절한 Backstage 템플릿이 실행됩니다.

```typescript
// plugins/infra-actions/src/module.ts
actionsRegistry.register({
  name: 'provision_database',
  title: 'Provision Database Instance',
  description:
    'Creates a new managed database instance through ' +
    'Crossplane. Supports PostgreSQL and Redis.',
  schema: {
    input: z => z.object({
      type: z.enum(['postgresql', 'redis'])
        .describe('Database engine type'),
      environment: z.enum(['development', 'staging', 'production'])
        .describe('Target environment'),
      size: z.enum(['small', 'medium', 'large'])
        .describe(
          'Instance size. small: 1 vCPU/2GB, ' +
          'medium: 2 vCPU/8GB, large: 4 vCPU/16GB',
        ),
      serviceName: z.string()
        .describe(
          'Name of the service that will use this database. ' +
          'Used for naming and tagging.',
        ),
      highAvailability: z.boolean()
        .optional()
        .default(false)
        .describe(
          'Enable multi-AZ deployment. ' +
          'Required for production.',
        ),
    }),
    output: z => z.object({
      instanceId: z.string(),
      connectionString: z.string(),
      status: z.string(),
      estimatedReadyTime: z.string(),
    }),
  },
  async handler(ctx) {
    const { type, environment, size, serviceName, highAvailability } =
      ctx.input;

    // 프로덕션 환경에서 HA 비활성화 방지
    if (environment === 'production' && !highAvailability) {
      throw new Error(
        'High availability is required for production databases',
      );
    }

    // Crossplane 클레임 생성
    const claim = await createCrossplaneClaim({
      apiVersion: 'database.platform.company.com/v1alpha1',
      kind: type === 'postgresql' ? 'PostgreSQLInstance' : 'RedisCluster',
      metadata: {
        name: `${serviceName}-${type}-${environment}`,
        namespace: `team-${ctx.user.ownershipEntityRefs[0]}`,
        labels: {
          'app.kubernetes.io/managed-by': 'backstage',
          'platform.company.com/environment': environment,
          'platform.company.com/service': serviceName,
          'platform.company.com/requested-by': ctx.user.userEntityRef,
        },
      },
      spec: {
        size,
        highAvailability,
      },
    });

    return {
      instanceId: claim.metadata.name,
      connectionString: `Pending - will be available in Secret ` +
        `${claim.metadata.name}-conn`,
      status: 'Provisioning',
      estimatedReadyTime: type === 'postgresql' ? '5-8 minutes' : '2-3 minutes',
    };
  },
});
```

> **중요:** 인프라 프로비저닝 같은 고위험 액션은 반드시 Backstage Permission Framework와 연동하여 적절한 권한 검사를 수행해야 합니다. 프로덕션 데이터베이스 생성은 팀 리드 이상만 가능하도록 정책을 설정하는 것이 권장됩니다.

## Scaffolder와의 통합: 템플릿에서 액션 재사용

v1.43에서 Actions Registry는 Scaffolder와 통합되었습니다. 이는 등록된 액션을 소프트웨어 템플릿의 단계(step)로 직접 사용할 수 있다는 의미입니다. 기존에는 Scaffolder 전용 액션을 별도로 정의해야 했지만, 이제는 하나의 액션 정의가 MCP 도구와 Scaffolder 단계 모두에서 동작합니다.

```yaml
# templates/new-microservice/template.yaml
apiVersion: scaffolder.backstage.io/v1beta3
kind: Template
metadata:
  name: create-microservice
  title: Create New Microservice
  description: |
    새로운 마이크로서비스를 생성합니다. Go 또는 TypeScript 기반의 
    서비스 스캐폴딩, GitHub 레포 생성, CI/CD 파이프라인 설정,
    데이터베이스 프로비저닝을 자동으로 수행합니다.
spec:
  owner: platform-team
  type: service
  parameters:
    - title: Service Details
      required:
        - serviceName
        - language
        - owner
      properties:
        serviceName:
          title: Service Name
          type: string
          pattern: '^[a-z][a-z0-9-]*$'
        language:
          title: Language
          type: string
          enum: ['go', 'typescript']
        owner:
          title: Owner Team
          type: string
          ui:field: OwnerPicker

    - title: Infrastructure
      properties:
        database:
          title: Database Type
          type: string
          enum: ['none', 'postgresql', 'redis']
        environment:
          title: Initial Environment
          type: string
          enum: ['development', 'staging']
          default: 'development'

  steps:
    # 1. 템플릿에서 코드 스캐폴딩
    - id: fetch-template
      name: Fetch Service Template
      action: fetch:template
      input:
        url: './skeleton-${{ parameters.language }}'
        values:
          serviceName: ${{ parameters.serviceName }}
          owner: ${{ parameters.owner }}

    # 2. GitHub 레포 생성 및 코드 푸시
    - id: publish
      name: Publish to GitHub
      action: publish:github
      input:
        repoUrl: 'github.com?owner=company&repo=${{ parameters.serviceName }}'
        defaultBranch: main

    # 3. Actions Registry에서 등록된 액션 직접 호출!
    - id: provision-db
      name: Provision Database
      if: ${{ parameters.database !== 'none' }}
      action: actions:provision_database
      input:
        type: ${{ parameters.database }}
        environment: ${{ parameters.environment }}
        size: small
        serviceName: ${{ parameters.serviceName }}

    # 4. 카탈로그에 등록
    - id: register
      name: Register in Catalog
      action: catalog:register
      input:
        repoContentsUrl: ${{ steps.publish.output.repoContentsUrl }}
        catalogInfoPath: /catalog-info.yaml
```

`actions:provision_database` 단계가 Actions Registry에 등록된 동일한 `provision_database` 액션을 호출합니다. 코드 중복 없이 하나의 비즈니스 로직이 MCP와 Scaffolder 양쪽에서 재사용됩니다. 이것이 바로 "Write Once, Expose Everywhere" 패턴의 핵심입니다.

### 보안 강화 설정

프로덕션 환경에서 MCP 서버를 운영할 때는 추가적인 보안 레이어가 필요합니다.

mcp:

      audit:

          - secretValue

## 실전 사용 시나리오

MCP 통합이 완료된 Backstage에서 개발자의 일상적인 워크플로우가 어떻게 변화하는지 살펴보겠습니다.

**시나리오 1: 인시던트 대응**

개발자가 Claude Desktop에서 대화합니다:

- "payment-service가 느려졌다는 알림을 받았어. 현재 상태를 확인해줘."
- Claude가 `catalog:get_service_info`로 서비스 정보를 조회하고, `kubernetes:list_pods`로 Pod 상태를 확인합니다.
- "Pod 3개 중 1개가 CrashLoopBackOff 상태예요. 최근 5분간 재시작이 12번 발생했습니다. payment-service-7b4d의 메모리 사용량이 limit에 도달한 것으로 보입니다."
- "해당 Pod만 재시작하고, 온콜 담당자에게 알려줘."
- Claude가 `kubernetes:restart_deployment`를 호출하고, `catalog:get_oncall`로 온콜 담당자를 찾아 Slack 채널 정보를 제공합니다.

**시나리오 2: 새 서비스 생성**

- "사용자 알림을 처리할 Go 마이크로서비스를 만들어줘. PostgreSQL 필요하고, staging에 배포할 거야."
- Claude가 `scaffolder:execute_template`을 적절한 파라미터로 호출합니다.
- GitHub 레포 생성, CI/CD 파이프라인 설정, 데이터베이스 프로비저닝이 자동으로 진행됩니다.
- 모든 것이 개발자의 ID로 실행되므로 감사 로그에도 정확히 기록됩니다.

**시나리오 3: 아키텍처 검토**

- "우리 서비스 중에서 PostgreSQL 16 미만을 사용하는 서비스 목록을 뽑아줘."
- Claude가 `catalog:list_entities`를 호출하여 데이터베이스 의존성 정보를 분석합니다.
- 버전 업그레이드가 필요한 서비스 목록과 각 서비스의 소유 팀을 정리해서 보여줍니다.

이러한 시나리오들은 기존에 수동으로 여러 도구를 오가며 수행하던 작업을 자연어 대화 한 번으로 해결할 수 있게 합니다. 핵심은 AI 에이전트가 새로운 권한을 갖는 것이 아니라, 기존 사용자의 권한을 자연어 인터페이스로 접근할 수 있게 된다는 점입니다.

## 마이그레이션 전략과 도입 단계

Backstage MCP 통합은 한 번에 전체 도입하기보다 점진적으로 적용하는 것이 효과적입니다. 1단계로 읽기 전용 카탈로그 액션(서비스 정보 조회, 의존성 확인 등)부터 시작하여 리스크를 최소화합니다. 2단계에서 Kubernetes 모니터링 같은 관측성 액션을 추가합니다. 3단계에서 Scaffolder 통합과 인프라 프로비저닝 같은 쓰기 액션을 도입하되, 반드시 권한 정책과 확인 단계를 함께 설정합니다.

기존 Backstage 인스턴스가 있다면 MCP 플러그인 추가는 `yarn add` 한 줄과 설정 파일 몇 줄로 완료됩니다. 새로운 인프라가 필요하지 않으며, 기존 인증/권한 시스템이 그대로 활용됩니다. 초기 투자 대비 효과가 매우 높은 개선입니다.

## 결론

Backstage의 MCP 통합은 내부 개발자 포털의 패러다임을 바꾸고 있습니다. 더 이상 개발자 포털은 브라우저에서만 접근하는 웹 애플리케이션이 아닙니다. AI 에이전트가 동일한 비즈니스 로직에 접근할 수 있는 플랫폼 API 허브로 진화하고 있습니다.

Actions Registry의 "Write Once, Expose Everywhere" 패턴은 플러그인 개발자의 부담을 줄이면서도 액션의 활용 범위를 극대화합니다. 동적 클라이언트 등록을 통한 OAuth 인증은 AI 에이전트를 특권적 서비스 계정이 아닌 사용자의 확장으로 취급하는 제로 트러스트 아키텍처를 실현합니다. 그리고 Scaffolder와의 통합은 수동 워크플로우와 AI 기반 자동화가 동일한 액션 위에서 공존할 수 있게 합니다.

플랫폼 엔지니어링의 궁극적 목표는 개발자가 인프라의 복잡성을 의식하지 않고 비즈니스 로직에 집중할 수 있게 하는 것입니다. MCP 통합은 그 목표에 한 걸음 더 다가가는 도구입니다. AI 에이전트에게 "staging에 Redis 캐시 좀 만들어줘"라고 말하면 적절한 권한 검사를 거쳐 Crossplane 클레임이 생성되고, 몇 분 후 연결 정보가 전달됩니다. 이것이 플랫폼 엔지니어링이 지향하는 개발자 경험입니다.
