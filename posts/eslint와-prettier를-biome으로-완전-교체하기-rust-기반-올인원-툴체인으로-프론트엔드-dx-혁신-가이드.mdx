# ESLint와 Prettier를 Biome으로 완전 교체하기: Rust 기반 올인원 툴체인으로 프론트엔드 DX 혁신 가이드

## Introduction

프론트엔드 개발자라면 누구나 한 번쯤 ESLint와 Prettier 설정의 늪에 빠져본 경험이 있을 것입니다. `.eslintrc.js`, `.prettierrc`, `eslint-config-prettier`, `eslint-plugin-prettier` — 이 조합이 서로 충돌하지 않도록 맞추는 것만으로도 상당한 시간을 소비하게 됩니다. 거기에 TypeScript ESLint 플러그인, React 플러그인, import 정렬 플러그인까지 더하면 `node_modules`에 수십 개의 의존성이 쌓이고, CI 파이프라인에서 린팅 단계가 빌드보다 오래 걸리는 아이러니한 상황이 벌어집니다.

2024년부터 JavaScript 생태계에서 빠르게 존재감을 키워온 **Biome**은 이 문제에 대한 근본적인 해답을 제시합니다. Rome 프로젝트에서 포크되어 독립적으로 성장한 Biome은 Rust로 작성된 단일 바이너리로 포매팅, 린팅, import 정렬을 모두 처리합니다. 2025년 3월에 출시된 Biome 2.0은 타입 인식(type-aware) 린팅, GritQL 기반 플러그인 시스템, 도메인(domain) 기반 규칙 자동 활성화라는 세 가지 핵심 기능을 도입하며 게임 체인저로 자리매김했습니다. 2026년 현재 v2.3에 이르러 423개 이상의 린트 규칙과 Vue, Svelte, Astro에 대한 실험적 지원까지 갖추고 있습니다.

이 글에서는 실제 프로덕션 프로젝트에서 ESLint + Prettier 조합을 Biome으로 완전히 교체하는 전체 과정을 다룹니다. 단순한 설치 가이드를 넘어, **왜** 교체해야 하는지에 대한 구체적인 벤치마크 데이터, 마이그레이션 중 마주치는 실제 함정들, 그리고 Biome 2.x의 고급 기능을 활용한 팀 차원의 코드 품질 전략까지 포괄적으로 안내합니다.

성능 수치부터 말씀드리면, 10,000줄 규모의 모노레포에서 Biome의 린팅 속도는 약 200ms인 반면 ESLint는 3~5초가 소요됩니다 — 약 15배의 차이입니다. 포매팅의 경우 1,000개 파일 처리에 Biome은 50ms, Prettier + ESLint 조합은 1~2초가 걸립니다. 이러한 속도 차이는 로컬 개발 경험뿐 아니라 CI/CD 파이프라인 비용에도 직접적인 영향을 미칩니다.

## 왜 ESLint + Prettier 조합을 떠나야 하는가

### 설정 복잡성의 누적

전형적인 React + TypeScript 프로젝트의 ESLint 설정을 살펴보면 문제의 심각성을 체감할 수 있습니다. 먼저 필요한 의존성 목록부터 압도적입니다:

```json
{
  "devDependencies": {
    "eslint": "^9.0.0",
    "prettier": "^3.3.0",
    "@typescript-eslint/parser": "^8.0.0",
    "@typescript-eslint/eslint-plugin": "^8.0.0",
    "eslint-plugin-react": "^7.35.0",
    "eslint-plugin-react-hooks": "^5.0.0",
    "eslint-plugin-import": "^2.30.0",
    "eslint-config-prettier": "^9.1.0",
    "eslint-plugin-prettier": "^5.2.0",
    "eslint-plugin-jsx-a11y": "^6.10.0"
  }
}
```

10개의 의존성, 각각이 자체 버전 사이클을 가지고 있으며, 업데이트 시 호환성 문제가 빈번하게 발생합니다. ESLint 9의 flat config 전환만으로도 수많은 플러그인이 호환성을 잃었고, 커뮤니티가 따라잡는 데 수개월이 걸렸습니다.

반면 Biome은 하나의 의존성으로 이 모든 것을 대체합니다:

```json
{
  "devDependencies": {
    "@biomejs/biome": "^2.3.0"
  }
}
```

> **핵심 포인트:** 의존성 수가 줄어든다는 것은 단순히 `package.json`이 깔끔해지는 것 이상의 의미를 가집니다. 보안 취약점 노출 면적의 축소, lock 파일 충돌 빈도 감소, 신규 팀원 온보딩 시간 단축으로 이어집니다.

### 성능 격차의 구조적 원인

ESLint가 느린 것은 단순한 구현 문제가 아니라 구조적 한계입니다. ESLint는 JavaScript로 작성되어 있으며, 각 파일마다 AST를 파싱하고 수백 개의 규칙을 순회하며 검사합니다. 플러그인마다 별도의 파서가 필요할 수 있고(`@typescript-eslint/parser`, `@babel/eslint-parser`), 이는 동일 파일을 여러 번 파싱하는 결과로 이어집니다.

Biome은 Rust의 제로 코스트 추상화(zero-cost abstraction)와 병렬 처리 능력을 기반으로 설계되었습니다. 단일 파서가 CST(Concrete Syntax Tree)를 생성하고, 포매터와 린터가 동일한 트리를 공유합니다. 멀티코어 CPU의 모든 코어를 활용하는 병렬 파일 처리는 프로젝트 규모가 커질수록 더 큰 이점을 발휘합니다.

```bash
# 실제 벤치마크: 500개 TypeScript 파일 프로젝트
$ time npx eslint . --max-warnings=0
real    0m4.872s

$ time npx biome check .
real    0m0.312s
```

4.8초 대 0.3초 — 약 15배의 차이는 파일 저장 시마다 반복되고, pre-commit 훅에서 체감됩니다. CI에서 하루 수십 번 실행된다면, 이 차이는 월 단위로 상당한 비용 절감으로 이어집니다.

### Prettier와의 미묘한 비호환성

ESLint와 Prettier를 함께 사용할 때 가장 흔한 문제는 규칙 충돌입니다. `eslint-config-prettier`가 충돌하는 ESLint 규칙을 비활성화해주지만, 이 과정에서 실제로 유용한 ESLint 규칙까지 꺼지는 경우가 있습니다. 또한 `--fix`와 `--write`의 실행 순서에 따라 결과가 달라지는 레이스 컨디션도 알려진 문제입니다.

Biome은 포매터와 린터가 동일한 엔진 안에서 동작하기 때문에 이런 충돌이 원천적으로 불가능합니다. `biome check --write` 한 번의 명령으로 포매팅, 린팅 수정, import 정렬이 원자적으로 처리됩니다.

## Biome 2.x 핵심 기능 심층 분석

### 타입 인식 린팅 — TypeScript 컴파일러 없이

Biome 2.0의 가장 혁신적인 기능은 TypeScript 컴파일러(tsc)에 의존하지 않는 자체 타입 추론 엔진입니다. `@typescript-eslint`의 타입 인식 규칙을 사용하려면 `tsconfig.json` 경로를 ESLint에 전달하고 tsc를 통해 전체 프로젝트의 타입 정보를 로딩해야 했습니다. 이 과정은 대규모 프로젝트에서 10초 이상 소요될 수 있으며, 메모리 사용량도 급증합니다.

Biome의 접근법은 다릅니다. 자체 추론 엔진이 필요한 타입 정보만 가볍게 추론하여 린트 규칙에 제공합니다. Vercel의 스폰서십으로 개발된 이 엔진은 tsc의 전체 타입 시스템을 복제하지 않으면서도 린팅에 필요한 수준의 타입 정보를 충분히 제공합니다.

```typescript
// Biome의 타입 인식 규칙이 잡아내는 예시들

// noFloatingPromises: await 없이 Promise를 무시하는 코드 감지
async function fetchData() {
  // ⚠️ Biome 경고: 이 Promise가 처리되지 않았습니다
  fetch('/api/data');

  // ✅ 올바른 사용
  await fetch('/api/data');
}

// useExhaustiveSwitchCase: 유니온 타입의 모든 케이스 검사
type Status = 'idle' | 'loading' | 'success' | 'error';

function handleStatus(status: Status) {
  switch (status) {
    case 'idle':
      return '대기 중';
    case 'loading':
      return '로딩 중';
    // ⚠️ Biome 경고: 'success'와 'error' 케이스가 누락되었습니다
  }
}
```

`biome.json`에서 타입 인식 린팅을 활성화하는 방법은 간단합니다:

```json
{
  "$schema": "https://biomejs.dev/schemas/2.3.0/schema.json",
  "linter": {
    "enabled": true,
    "rules": {
      "correctness": {
        "noFloatingPromises": "error",
        "useExhaustiveSwitchCase": "warn"
      },
      "suspicious": {
        "noMisleadingInstantiator": "error"
      }
    }
  }
}
```

### GritQL 플러그인 시스템

Biome 2.0에서 도입된 GritQL 기반 플러그인 시스템은 커스텀 린트 규칙 작성을 획기적으로 단순화했습니다. ESLint 플러그인을 만들려면 AST 방문자 패턴을 이해하고, 규칙 메타데이터를 정의하며, 별도의 npm 패키지로 배포해야 했습니다. Biome에서는 GritQL 패턴 하나로 동일한 목적을 달성할 수 있습니다.

예를 들어, 팀 내에서 `console.log`를 프로덕션 코드에서 금지하되, `console.error`와 `console.warn`은 허용하는 규칙을 만들고 싶다면:

```gritql
# biome-plugins/no-console-log.grit
`console.log($args)` => .
```

이 패턴을 `biome.json`에 등록합니다:

```json
{
  "plugins": ["./biome-plugins/no-console-log.grit"],
  "linter": {
    "enabled": true
  }
}
```

더 복잡한 예시로, React 컴포넌트에서 인라인 스타일 객체 사용을 감지하는 플러그인을 살펴보겠습니다:

```gritql
# biome-plugins/no-inline-style.grit
`<$component style={$styleObj} $rest />` where {
  $styleObj <: not `styles.$name`
}
```

> **왜 GritQL인가?** GritQL은 코드 패턴 매칭에 특화된 쿼리 언어로, 원래 Grit.io에서 개발되었습니다. 2025년 말 GritQL이 Biome 조직으로 편입되면서, Biome의 CST에 직접 쿼리를 실행하고 커스텀 진단 메시지를 보고할 수 있게 되었습니다. AST 방문자 코드를 수십 줄 작성하는 대신, 선언적 패턴 한 줄로 같은 결과를 얻을 수 있다는 점이 핵심입니다.

### 도메인(Domain) 기반 규칙 자동 활성화

Biome 2.0의 도메인 시스템은 프로젝트의 `package.json` 의존성을 분석하여 관련 린트 규칙을 자동으로 활성화합니다. React 프로젝트라면 React 관련 규칙이, Next.js 프로젝트라면 Next.js 특화 규칙이 추가 설정 없이 적용됩니다.

```json
{
  "linter": {
    "domains": {
      "react": "all",
      "next": "recommended",
      "test": "recommended"
    }
  }
}
```

이 설정만으로 React Hooks 규칙, Next.js Image 최적화 규칙, 테스트 프레임워크 모범 사례 규칙이 활성화됩니다. 새로운 도메인이 추가될 때마다 별도의 플러그인을 설치할 필요 없이 Biome 업데이트 하나로 최신 규칙을 적용받을 수 있습니다.

## 실전 마이그레이션 가이드

### Step 1: 현재 상태 감사

마이그레이션을 시작하기 전에, 현재 ESLint 설정에서 사용 중인 규칙과 플러그인을 정확히 파악해야 합니다. Biome은 이를 위한 자동화된 마이그레이션 도구를 제공합니다:

```bash
# Biome 설치
npm install --save-dev @biomejs/biome@latest

# 기본 설정 파일 생성
npx biome init

# ESLint 설정을 Biome으로 마이그레이션 (자동)
npx biome migrate eslint --write

# Prettier 설정을 Biome으로 마이그레이션 (자동)
npx biome migrate prettier --write
```

`biome migrate eslint` 명령은 `.eslintrc.*` 파일을 분석하여 대응하는 Biome 규칙으로 변환합니다. 변환할 수 없는 규칙은 주석으로 표시되므로, 어떤 규칙을 GritQL 플러그인으로 대체해야 할지 명확하게 파악할 수 있습니다.

### Step 2: biome.json 정밀 설정

자동 마이그레이션 결과를 기반으로 프로젝트에 맞게 설정을 조정합니다:

```json
{
  "$schema": "https://biomejs.dev/schemas/2.3.0/schema.json",
  "organizeImports": {
    "enabled": true
  },
  "formatter": {
    "enabled": true,
    "indentStyle": "space",
    "indentWidth": 2,
    "lineWidth": 100,
    "lineEnding": "lf"
  },
  "javascript": {
    "formatter": {
      "quoteStyle": "single",
      "trailingCommas": "all",
      "semicolons": "always",
      "arrowParentheses": "always"
    }
  },
  "css": {
    "formatter": {
      "enabled": true,
      "indentStyle": "space",
      "indentWidth": 2
    },
    "linter": {
      "enabled": true
    }
  },
  "linter": {
    "enabled": true,
    "domains": {
      "react": "all",
      "next": "recommended"
    },
    "rules": {
      "recommended": true,
      "correctness": {
        "noUnusedVariables": "error",
        "noUnusedImports": "error",
        "useExhaustiveDependencies": "warn"
      },
      "style": {
        "noNonNullAssertion": "warn",
        "useConst": "error",
        "useTemplate": "error"
      },
      "suspicious": {
        "noExplicitAny": "warn",
        "noDoubleEquals": "error"
      },
      "performance": {
        "noAccumulatingSpread": "error"
      }
    }
  },
  "files": {
    "ignore": [
      "node_modules",
      ".next",
      "dist",
      "coverage",
      "*.generated.ts"
    ]
  }
}
```

### Step 3: Import 정렬 커스터마이징

Biome 2.x의 import/export 정리기는 import 병합, 커스텀 그룹 정의 등 `eslint-plugin-import`보다 강력한 기능을 제공합니다:

```json
{
  "assist": {
    "actions": {
      "source": {
        "organizeImports": {
          "level": "error",
          "options": {
            "groups": [
              { "static": ":BUILTIN:" },
              { "static": ":PACKAGE:" },
              { "static": ":ALIAS:", "match": "@/*" },
              { "static": ":RELATIVE:" },
              { "static": ":TYPE:" }
            ]
          }
        }
      }
    }
  }
}
```

이 설정을 적용하면 import 문이 자동으로 다음 순서로 정렬됩니다:

```typescript
// 1. Node.js 내장 모듈
import path from 'node:path';

// 2. 외부 패키지
import { useState, useEffect } from 'react';
import { clsx } from 'clsx';

// 3. 프로젝트 내부 별칭 경로
import { Button } from '@/components/Button';
import { useAuth } from '@/hooks/useAuth';

// 4. 상대 경로 import
import { helper } from './utils';

// 5. 타입 import
import type { User } from '@/types';
```

### Step 4: 모노레포 설정

Biome 2.x는 중첩 설정 파일을 자동 감지합니다. 모노레포의 각 패키지에 별도의 `biome.json`을 두고, 루트 설정을 상속받을 수 있습니다:

```
monorepo/
├── biome.json              # 루트: 공통 규칙
├── packages/
│   ├── web/
│   │   └── biome.json      # extends 루트, React 도메인 활성화
│   ├── api/
│   │   └── biome.json      # extends 루트, Node.js 규칙 추가
│   └── shared/
│       └── biome.json      # extends 루트
```

패키지별 설정 파일 예시:

```json
{
  "$schema": "https://biomejs.dev/schemas/2.3.0/schema.json",
  "extends": ["../../biome.json"],
  "linter": {
    "domains": {
      "react": "all"
    },
    "rules": {
      "style": {
        "useFilenamingConvention": {
          "level": "error",
          "options": {
            "filenameCases": ["PascalCase", "camelCase"]
          }
        }
      }
    }
  }
}
```

### Step 5: CI/CD 파이프라인 업데이트

기존 ESLint + Prettier CI 스텝을 Biome으로 교체합니다:

```yaml
# .github/workflows/ci.yml
name: CI
on: [push, pull_request]

jobs:
  quality:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 22

      - name: Install dependencies
        run: npm ci

      # 기존: ESLint + Prettier (2개 스텝, ~12초)
      # - run: npx eslint . --max-warnings=0
      # - run: npx prettier --check .

      # 신규: Biome 하나로 통합 (~0.8초)
      - name: Code quality check
        run: npx biome ci .
```

`biome ci` 명령은 포매팅 검사, 린팅, import 정렬 검증을 한 번에 수행하며, CI에 최적화된 출력 형식을 제공합니다. GitHub Actions에서는 SARIF 리포터를 사용하여 PR에 인라인 코멘트를 달 수도 있습니다:

```yaml
      - name: Biome CI
        run: npx biome ci . --reporter=github
```

### Step 6: Git Hooks 설정

`lint-staged`와 함께 사용하던 pre-commit 훅도 간소화됩니다:

```json
{
  "lint-staged": {
    "*.{js,ts,jsx,tsx,css,json}": [
      "biome check --write --no-errors-on-unmatched"
    ]
  }
}
```

또는 Biome이 제공하는 별도의 Git 훅 지원을 활용할 수도 있습니다. `lefthook`이나 `husky`와 함께 사용하면 됩니다:

```yaml
# .lefthook.yml
pre-commit:
  commands:
    biome:
      glob: "*.{js,ts,jsx,tsx,css,json}"
      run: npx biome check --write --staged {staged_files}
      stage_fixed: true
```

## 마이그레이션 시 주의할 함정들

### Prettier와의 미세한 포매팅 차이

Biome의 포매터는 Prettier와 97% 이상 호환되지만, 일부 엣지 케이스에서 다른 결과를 생성합니다. 특히 긴 삼항 연산자 체인, JSX 내부의 조건부 렌더링, 그리고 오브젝트 디스트럭처링에서 차이가 날 수 있습니다:

```typescript
// Prettier 결과
const result =
  condition1 && condition2
    ? longValueName1
    : condition3
      ? longValueName2
      : defaultValue;

// Biome 결과 (약간 다를 수 있음)
const result = condition1 && condition2
  ? longValueName1
  : condition3
    ? longValueName2
    : defaultValue;
```

이런 차이가 팀에서 문제가 된다면, 마이그레이션 초기에 `npx biome format --write .`을 실행하여 전체 코드베이스를 한 번에 재포매팅하고, 해당 커밋을 `.git-blame-ignore-revs`에 등록하는 것을 권장합니다:

```bash
# 전체 재포매팅
npx biome format --write .
git add -A
git commit -m "chore: migrate formatting from Prettier to Biome"

# git blame에서 해당 커밋 무시
echo "$(git rev-parse HEAD) # Biome migration formatting" >> .git-blame-ignore-revs
git config blame.ignoreRevsFile .git-blame-ignore-revs
```

### 아직 지원되지 않는 ESLint 규칙

Biome 2.3 기준으로 423개 이상의 규칙을 지원하지만, 일부 ESLint 플러그인의 특수한 규칙은 아직 대응하지 못합니다. 대표적인 예:

- `eslint-plugin-jsx-a11y`의 일부 고급 접근성 규칙
- `eslint-plugin-testing-library`의 특정 쿼리 패턴 규칙
- `eslint-plugin-storybook`의 Storybook 특화 규칙

이런 경우 두 가지 전략이 있습니다:

1. **GritQL 플러그인으로 자체 구현**: 패턴이 단순한 경우 몇 줄의 GritQL로 대체 가능
2. **ESLint 병행 실행**: 전환 기간 동안 Biome이 커버하지 못하는 규칙만 ESLint에서 실행

```json
{
  "scripts": {
    "lint": "biome check .",
    "lint:a11y": "eslint . --config eslint.a11y.config.js",
    "lint:all": "npm run lint && npm run lint:a11y"
  }
}
```

점진적으로 ESLint 의존성을 줄여나가면서, Biome의 규칙 커버리지가 확대되면 완전히 제거할 수 있습니다.

### IDE 설정 동기화

JetBrains IDE(WebStorm, IntelliJ)의 경우, Biome IDEA 플러그인을 설치하면 내장 포매터 및 검사기를 Biome으로 대체할 수 있습니다. 2025년 말 업데이트된 플러그인은 LSP 기반으로 동작하여 실시간 진단이 크게 개선되었습니다.

## 고급 활용: 팀 차원의 코드 품질 전략

### 점진적 규칙 강화

새 프로젝트가 아닌 기존 프로젝트에 Biome을 도입할 때, 모든 규칙을 한 번에 `error`로 설정하면 수백 개의 기존 위반 사항이 쏟아집니다. Biome의 suppression 기능을 활용한 점진적 접근이 효과적입니다:

```bash
# 현재 위반 사항을 모두 자동 suppress 처리
npx biome check --write --unsafe .

# 이후 새 코드에서는 규칙이 적용됨
# 기존 suppress 주석은 tech debt로 관리
```

```typescript
// Biome이 자동으로 추가한 suppression 주석
// biome-ignore lint/suspicious/noExplicitAny: 레거시 코드, TODO: 타입 정의 필요
function processData(data: any) {
  // ...
}
```

이 접근법의 장점은 **새 코드에 대한 품질 기준은 즉시 적용**하면서, 기존 코드는 점진적으로 개선할 수 있다는 것입니다. CI에서 suppression 주석의 수를 추적하여 기술 부채 감소 추이를 모니터링할 수도 있습니다.

### 멀티 파일 분석 활용

Biome 2.0의 멀티 파일 분석 기능으로 가능해진 대표적인 규칙이 `noImportCycles`입니다:

```typescript
// utils/format.ts
import { validate } from './validate'; // ← validate.ts가 format.ts를 import하면 순환!

export function formatDate(date: Date) {
  if (validate(date)) {
    return date.toISOString();
  }
  return null;
}
```

순환 import는 런타임에서 `undefined` 참조나 초기화 순서 문제를 일으킬 수 있는데, 이전에는 `eslint-plugin-import`의 `no-cycle` 규칙으로 감지했으나 매우 느렸습니다(전체 의존성 그래프를 매번 재구성). Biome은 프로젝트 로딩 시 한 번 구축한 의존성 그래프를 재활용하여 이 검사를 거의 무료로 수행합니다.

### 커스텀 도메인 규칙 조합

팀의 코딩 컨벤션을 하나의 biome.json에 담아 사내 npm 패키지로 배포할 수 있습니다:

```json

  "name": "@mycompany/biome-config",
  "version": "1.0.0",
  "files": ["biome.json"]
}
```

```json
      "style": {
        "useNamingConvention": {

  }
}
```

## Conclusion

ESLint와 Prettier는 JavaScript 생태계의 코드 품질을 한 단계 끌어올린 위대한 도구들입니다. 하지만 두 도구의 조합이 가진 구조적 한계 — 느린 실행 속도, 복잡한 설정, 플러그인 호환성 문제 — 는 프로젝트 규모가 커질수록 개발 경험을 저해하는 요인이 됩니다. Biome 2.x는 이 문제들에 대한 현실적이고 즉시 적용 가능한 대안을 제공합니다.

특히 타입 인식 린팅, GritQL 플러그인, 도메인 기반 자동 구성이라는 세 가지 혁신은 단순한 속도 개선을 넘어 코드 품질 관리의 패러다임 자체를 바꿉니다. tsc 없이 타입 수준의 버그를 잡고, 선언적 패턴 언어로 팀 규칙을 정의하며, `package.json`만으로 관련 규칙이 자동 활성화되는 경험은 한번 맛보면 돌아가기 어렵습니다.

마이그레이션이 부담스럽다면, Biome의 `migrate eslint`과 `migrate prettier` 명령이 대부분의 무거운 작업을 자동으로 처리해준다는 점을 기억하세요. 점진적으로 도입하면서 팀이 적응할 시간을 주되, CI부터 전환하는 것을 권장합니다. CI에서의 속도 개선 효과는 즉시 체감되며, 팀원들의 동의를 얻는 가장 강력한 근거가 됩니다.

15배 빠른 린팅, 90% 줄어든 설정 파일, 하나의 의존성 — 숫자가 이미 충분히 말해주고 있습니다. 지금이 전환할 때입니다.
