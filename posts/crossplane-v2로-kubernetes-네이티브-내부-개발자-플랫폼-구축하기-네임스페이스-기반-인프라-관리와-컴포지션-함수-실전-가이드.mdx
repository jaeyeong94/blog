# Crossplane v2로 Kubernetes 네이티브 내부 개발자 플랫폼 구축하기

## 도입: 왜 지금 Crossplane v2인가

인프라 관리의 풍경이 빠르게 변하고 있습니다. Terraform이 IaC(Infrastructure as Code) 시대를 열었다면, Crossplane은 그 다음 단계—**인프라를 Kubernetes API로 관리하는 시대**—를 열고 있습니다. 2025년 10월 CNCF Graduated 프로젝트로 승격된 Crossplane은 이제 Kubernetes, Prometheus, Envoy와 같은 반열에 올라섰으며, 엔터프라이즈 환경에서의 신뢰성과 성숙도를 공식적으로 인정받았습니다.

Crossplane v2는 단순한 마이너 업데이트가 아닙니다. **네임스페이스 기반 리소스 관리, 모든 Kubernetes 리소스에 대한 컴포지션 지원, Operations를 통한 운영 워크플로우** 등 근본적인 아키텍처 변화를 가져왔습니다. 이전 버전에서 불편했던 Claim과 클러스터 스코프 제약이 사라지면서, 플랫폼 엔지니어가 진정한 셀프서비스 개발자 플랫폼을 구축하기가 훨씬 쉬워졌습니다.

플랫폼 엔지니어링이 단순한 유행어를 넘어 조직의 핵심 역량으로 자리잡은 지금, Crossplane v2는 "내부 개발자 플랫폼(IDP)을 어떻게 구축할 것인가"라는 질문에 가장 설득력 있는 답을 제시합니다. Terraform의 plan/apply 모델과 달리, Crossplane은 Kubernetes의 선언적 조정(reconciliation) 패턴을 그대로 활용합니다. 원하는 상태를 선언하면, 컨트롤러가 지속적으로 실제 상태를 맞춰갑니다. 드리프트 감지? 자동입니다. 장애 복구? 컨트롤러가 알아서 합니다.

이 글에서는 Crossplane v2의 핵심 변화를 깊이 있게 살펴보고, 컴포지션 함수 파이프라인을 활용한 실제 플랫폼 구축, 그리고 프로덕션 환경에서의 운영 패턴까지 다룹니다. 단순히 "무엇이 바뀌었는가"가 아니라 **"왜 이렇게 바뀌었고, 이것이 여러분의 플랫폼에 어떤 의미인가"**에 초점을 맞추겠습니다.

## Crossplane v2의 핵심 변화: 네임스페이스가 모든 것을 바꾼다

### 네임스페이스 기반 Composite Resource

Crossplane v1에서 가장 큰 불편 중 하나는 Composite Resource(XR)가 항상 클러스터 스코프라는 것이었습니다. 개발자에게 네임스페이스 수준의 접근을 제공하려면 Claim이라는 별도 추상화를 거쳐야 했고, 이는 불필요한 복잡성을 더했습니다.

v2에서는 XR이 기본적으로 네임스페이스 스코프입니다. 이것이 왜 중요할까요? 네임스페이스는 Kubernetes의 핵심 격리 단위입니다. RBAC, NetworkPolicy, ResourceQuota 모두 네임스페이스를 기준으로 동작합니다. XR이 네임스페이스에 속하게 되면서, 기존 Kubernetes 보안 모델과 완벽하게 통합됩니다.

```yaml
# Crossplane v2: 네임스페이스 기반 XRD 정의
apiVersion: apiextensions.crossplane.io/v2
kind: CompositeResourceDefinition
metadata:
  name: applications.platform.example.io
spec:
  scope: Namespaced  # 기본값, 네임스페이스 스코프
  group: platform.example.io
  names:
    kind: Application
    plural: applications
  versions:
    - name: v1
      served: true
      referenceable: true
      schema:
        openAPIV3Schema:
          type: object
          properties:
            spec:
              type: object
              properties:
                image:
                  type: string
                  description: 컨테이너 이미지
                replicas:
                  type: integer
                  default: 2
                database:
                  type: object
                  properties:
                    engine:
                      type: string
                      enum: ["postgres", "mysql"]
                    storageGB:
                      type: integer
                      default: 20
```

개발자는 이제 자신의 네임스페이스에서 직접 XR을 생성합니다. Claim이라는 중간 계층이 사라졌습니다:

```yaml
# 개발자가 자신의 네임스페이스에서 직접 생성
apiVersion: platform.example.io/v1
kind: Application
metadata:
  namespace: team-backend  # 자신의 네임스페이스
  name: user-service
spec:
  image: registry.example.io/user-service:v2.1.0
  replicas: 3
  database:
    engine: postgres
    storageGB: 50
  crossplane:
    compositionRef:
      name: standard-app
```

> **핵심 포인트:** `spec.crossplane` 아래에 Crossplane 관련 필드가 모였습니다. 개발자는 `spec` 최상위의 비즈니스 로직 필드에만 집중하면 되고, Crossplane 내부 동작은 무시할 수 있습니다. 이 작은 변화가 개발자 경험(DX)에 큰 차이를 만듭니다.

### 네임스페이스 기반 Managed Resource

XR뿐 아니라 Managed Resource(MR)도 네임스페이스 스코프가 됩니다. 이전에는 AWS RDS 인스턴스, S3 버킷 등 모든 클라우드 리소스가 클러스터 스코프로 생성되어, 팀 간 격리가 어려웠습니다.

```yaml
# v2: 네임스페이스 기반 S3 버킷
apiVersion: s3.aws.m.upbound.io/v1beta1
kind: Bucket
metadata:
  namespace: team-data  # 팀의 네임스페이스에 속함
  name: analytics-raw-data
spec:
  forProvider:
    region: ap-northeast-2
    tags:
      team: data-engineering
      environment: production
```

이제 RBAC만으로 "data-engineering 팀은 자신의 네임스페이스에서만 S3 버킷을 생성할 수 있다"는 정책을 자연스럽게 구현할 수 있습니다:

```yaml
# 팀별 인프라 접근 제어
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: team-data
  name: infra-creator
rules:
  - apiGroups: ["s3.aws.m.upbound.io"]
    resources: ["buckets"]
    verbs: ["create", "get", "list", "watch", "update", "patch"]
  - apiGroups: ["rds.aws.m.upbound.io"]
    resources: ["instances"]
    verbs: ["create", "get", "list", "watch", "update", "patch"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  namespace: team-data
  name: data-team-infra
subjects:
  - kind: Group
    name: data-engineering
    apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role
  name: infra-creator
  apiGroup: rbac.authorization.k8s.io
```

> **왜 이렇게 바뀌었나:** 클러스터 스코프 리소스는 멀티테넌시의 적입니다. 모든 팀의 리소스가 한 곳에 섞이면, 권한 관리가 ClusterRole 수준에서만 가능하고, 이름 충돌을 피하기 위한 네이밍 컨벤션에 의존해야 합니다. 네임스페이스 기반으로 전환함으로써, Crossplane은 Kubernetes의 기존 멀티테넌시 패턴을 그대로 활용할 수 있게 되었습니다.

## 모든 Kubernetes 리소스를 컴포지션하라

Crossplane v2의 가장 흥미로운 변화 중 하나는 **컴포지션이 더 이상 Crossplane 리소스에 제한되지 않는다**는 것입니다. Deployment, Service, ConfigMap 같은 네이티브 Kubernetes 리소스는 물론, CloudNativePG의 PostgreSQL Cluster나 Cert-Manager의 Certificate 같은 서드파티 CRD도 컴포지션에 포함할 수 있습니다.

이전에는 Kubernetes 네이티브 리소스를 컴포지션에 포함하려면 `provider-kubernetes`의 `Object` 리소스로 감싸야 했습니다. 이는 우회적이고 디버깅이 어려웠습니다. v2에서는 이 제약이 완전히 사라졌습니다.

```yaml
# v2 Composition: Kubernetes 네이티브 리소스 + 클라우드 리소스 통합
apiVersion: apiextensions.crossplane.io/v1
kind: Composition
metadata:
  name: standard-app
spec:
  compositeTypeRef:
    apiVersion: platform.example.io/v1
    kind: Application
  mode: Pipeline
  pipeline:
    - step: build-resources
      functionRef:
        name: function-kcl
      input:
        apiVersion: krm.kcl.dev/v1alpha1
        kind: KCLInput
        spec:
          source: |
            # KCL로 리소스 생성 로직 작성
            oxr = option("params").oxr
            
            # Kubernetes Deployment - 네이티브 리소스 직접 컴포지션
            deployment = {
                apiVersion = "apps/v1"
                kind = "Deployment"
                metadata.name = oxr.metadata.name
                spec = {
                    replicas = oxr.spec.replicas
                    selector.matchLabels.app = oxr.metadata.name
                    template = {
                        metadata.labels.app = oxr.metadata.name
                        spec.containers = [{
                            name = "app"
                            image = oxr.spec.image
                            ports = [{ containerPort = 8080 }]
                            resources = {
                                requests = { cpu = "100m", memory = "256Mi" }
                                limits = { cpu = "500m", memory = "512Mi" }
                            }
                        }]
                    }
                }
            }
            
            # Kubernetes Service - 네이티브 리소스
            service = {
                apiVersion = "v1"
                kind = "Service"
                metadata.name = oxr.metadata.name
                spec = {
                    selector.app = oxr.metadata.name
                    ports = [{ port = 80, targetPort = 8080 }]
                }
            }
            
            # CloudNativePG Cluster - 서드파티 CRD
            database = {
                apiVersion = "postgresql.cnpg.io/v1"
                kind = "Cluster"
                metadata.name = "{}-db".format(oxr.metadata.name)
                spec = {
                    instances = 3
                    storage.size = "{}Gi".format(oxr.spec.database.storageGB)
                    postgresql.parameters = {
                        shared_buffers = "256MB"
                        max_connections = "200"
                    }
                }
            }
            
            items = [deployment, service, database]
    
    - step: detect-readiness
      functionRef:
        name: function-auto-ready
```

이 예제에서 하나의 `Application` XR이 Kubernetes Deployment, Service, 그리고 CloudNativePG PostgreSQL 클러스터를 모두 관리합니다. 개발자는 이미지와 데이터베이스 설정만 지정하면, 전체 스택이 자동으로 프로비저닝되고 지속적으로 조정됩니다.

> **왜 이것이 게임 체인저인가:** 이전에는 "앱 배포"와 "인프라 프로비저닝"이 별도 워크플로우였습니다. Helm으로 앱을 배포하고, Terraform으로 데이터베이스를 만들고, 둘을 수동으로 연결해야 했습니다. Crossplane v2에서는 하나의 선언으로 전체 스택을 관리합니다. 이것이 진정한 플랫폼 추상화입니다.

## 컴포지션 함수 파이프라인: 로직을 코드로 표현하기

Crossplane v1의 Patch & Transform 방식은 단순한 매핑에는 적합했지만, 복잡한 조건부 로직이나 동적 리소스 생성에는 한계가 명확했습니다. 컴포지션 함수(Composition Functions)는 이 문제를 근본적으로 해결합니다.

### 함수 파이프라인의 동작 원리

컴포지션 함수는 gRPC 서버로 동작하는 컨테이너입니다. Crossplane이 XR의 현재 상태(observed)와 원하는 상태(desired)를 함수에 전달하면, 함수가 변환된 desired 상태를 반환합니다. 여러 함수를 파이프라인으로 연결하여, 각 단계가 이전 단계의 결과를 받아 점진적으로 리소스를 구성합니다.

```
XR(observed) → [Function A] → [Function B] → [Function C] → desired resources
```

이 아키텍처가 강력한 이유는, **각 함수가 독립적이고 재사용 가능하다**는 것입니다. 보안 정책 적용 함수, 모니터링 사이드카 주입 함수, 네트워크 정책 생성 함수 등을 조합하여 조직의 표준 플랫폼을 구축할 수 있습니다.

### 실전 예제: 멀티 환경 배포 파이프라인

프로덕션 수준의 플랫폼에서는 환경별(dev/staging/prod) 설정 차이, 보안 정책 적용, 모니터링 설정 등 복잡한 로직이 필요합니다. 컴포지션 함수 파이프라인으로 이를 우아하게 해결할 수 있습니다:

```yaml
apiVersion: apiextensions.crossplane.io/v1
kind: Composition
metadata:
  name: production-app
spec:
  compositeTypeRef:
    apiVersion: platform.example.io/v1
    kind: Application
  mode: Pipeline
  pipeline:
    # Step 1: 기본 리소스 생성 (KCL로 타입 안전하게)
    - step: generate-base-resources
      functionRef:
        name: function-kcl
      input:
        apiVersion: krm.kcl.dev/v1alpha1
        kind: KCLInput
        spec:
          source: |
            oxr = option("params").oxr
            env = oxr.spec.get("environment", "dev")
            
            # 환경별 리소스 설정
            _env_config = {
                dev = { replicas = 1, cpu = "200m", memory = "256Mi" }
                staging = { replicas = 2, cpu = "500m", memory = "512Mi" }
                prod = { replicas = 3, cpu = "1000m", memory = "1Gi" }
            }
            
            cfg = _env_config[env]
            
            deployment = {
                apiVersion = "apps/v1"
                kind = "Deployment"
                metadata = {
                    name = oxr.metadata.name
                    labels = {
                        "app.kubernetes.io/name" = oxr.metadata.name
                        "app.kubernetes.io/environment" = env
                    }
                }
                spec = {
                    replicas = cfg.replicas
                    selector.matchLabels = {
                        "app.kubernetes.io/name" = oxr.metadata.name
                    }
                    template = {
                        metadata.labels = {
                            "app.kubernetes.io/name" = oxr.metadata.name
                            "app.kubernetes.io/environment" = env
                        }
                        spec.containers = [{
                            name = "app"
                            image = oxr.spec.image
                            resources = {
                                requests = { cpu = cfg.cpu, memory = cfg.memory }
                                limits = { cpu = cfg.cpu, memory = cfg.memory }
                            }
                        }]
                    }
                }
            }
            
            items = [deployment]
    
    # Step 2: 네트워크 정책 자동 생성
    - step: add-network-policies
      functionRef:
        name: function-kcl
      input:
        apiVersion: krm.kcl.dev/v1alpha1
        kind: KCLInput
        spec:
          source: |
            oxr = option("params").oxr
            dxr = option("params").dxr
            
            # 기존 리소스 유지
            items = dxr.spec.crossplane.resourceRefs or []
            
            # NetworkPolicy 추가
            netpol = {
                apiVersion = "networking.k8s.io/v1"
                kind = "NetworkPolicy"
                metadata.name = "{}-netpol".format(oxr.metadata.name)
                spec = {
                    podSelector.matchLabels = {
                        "app.kubernetes.io/name" = oxr.metadata.name
                    }
                    policyTypes = ["Ingress", "Egress"]
                    ingress = [{
                        from = [{
                            namespaceSelector.matchLabels = {
                                "kubernetes.io/metadata.name" = oxr.metadata.namespace
                            }
                        }]
                    }]
                    egress = [{
                        to = [{
                            namespaceSelector.matchLabels = {
                                "kubernetes.io/metadata.name" = "kube-system"
                            }
                        }]
                    }, {
                        ports = [{ port = 443, protocol = "TCP" }]
                    }]
                }
            }
            
            items += [netpol]
    
    # Step 3: 관측성 설정 (ServiceMonitor, PodDisruptionBudget)
    - step: add-observability
      functionRef:
        name: function-kcl
      input:
        apiVersion: krm.kcl.dev/v1alpha1
        kind: KCLInput
        spec:
          source: |
            oxr = option("params").oxr
            env = oxr.spec.get("environment", "dev")
            
            items = []
            
            # 프로덕션에만 ServiceMonitor 추가
            if env == "prod":
                items += [{
                    apiVersion = "monitoring.coreos.com/v1"
                    kind = "ServiceMonitor"
                    metadata.name = "{}-monitor".format(oxr.metadata.name)
                    spec = {
                        selector.matchLabels = {
                            "app.kubernetes.io/name" = oxr.metadata.name
                        }
                        endpoints = [{ port = "metrics", interval = "30s" }]
                    }
                }]
            
            # staging/prod에 PDB 추가
            if env in ["staging", "prod"]:
                items += [{
                    apiVersion = "policy/v1"
                    kind = "PodDisruptionBudget"
                    metadata.name = "{}-pdb".format(oxr.metadata.name)
                    spec = {
                        minAvailable = 1
                        selector.matchLabels = {
                            "app.kubernetes.io/name" = oxr.metadata.name
                        }
                    }
                }]
    
    # Step 4: Readiness 자동 감지
    - step: auto-ready
      functionRef:
        name: function-auto-ready
```

이 파이프라인에서 주목할 점은 **각 단계가 독립적인 관심사**를 담당한다는 것입니다. 기본 리소스 생성, 네트워크 보안, 관측성이 분리되어 있어 각각을 독립적으로 테스트하고 업데이트할 수 있습니다.

### CUE를 활용한 타입 안전 컴포지션

KCL 외에도 CUE 언어를 사용할 수 있습니다. CUE는 강력한 타입 시스템과 검증 기능을 제공하여, 설정 실수를 컴파일 타임에 잡아낼 수 있습니다:

```yaml
apiVersion: apiextensions.crossplane.io/v1
kind: Composition
metadata:
  name: validated-storage
spec:
  compositeTypeRef:
    apiVersion: platform.example.io/v1
    kind: StorageBucket
  mode: Pipeline
  pipeline:
    - step: cue-generate
      functionRef:
        name: function-cue
      input:
        apiVersion: cue.fn.crossplane.io/v1beta1
        kind: CUEInput
        name: storage-bucket
        export:
          target: Resources
          value: |
            // CUE의 타입 시스템으로 검증
            #BucketConfig: {
                region: "ap-northeast-2" | "us-east-1" | "eu-west-1"
                versioning: bool | *true  // 기본값 true
                encryption: "AES256" | "aws:kms"
            }
            
            config: #BucketConfig & {
                region: "ap-northeast-2"
                encryption: "aws:kms"
            }
            
            apiVersion: "s3.aws.m.upbound.io/v1beta1"
            kind: "Bucket"
            spec: forProvider: {
                region: config.region
                serverSideEncryptionConfiguration: [{
                    rule: [{
                        applyServerSideEncryptionByDefault: [{
                            sseAlgorithm: config.encryption
                        }]
                    }]
                }]
                versioning: [{
                    enabled: config.versioning
                }]
            }
    - step: auto-ready
      functionRef:
        name: function-auto-ready
```

> **왜 함수 파이프라인인가:** Patch & Transform은 "설정의 설정"이었습니다. YAML로 로직을 표현하려다 보니 가독성이 떨어지고, 디버깅이 거의 불가능했습니다. 함수 파이프라인은 **실제 프로그래밍 언어(Go, Python, KCL, CUE)로 로직을 작성**할 수 있게 해줍니다. 테스트, 코드 리뷰, IDE 지원이 가능해지면서, 컴포지션이 진정한 "코드"가 되었습니다.

## Operations: 선언적 Day-2 운영

Crossplane v2의 또 다른 혁신은 **Operations**입니다. 기존 Crossplane은 리소스의 지속적 조정(continuous reconciliation)에 특화되어 있었지만, 일회성 운영 작업—인증서 갱신 확인, 롤링 업그레이드, 백업 검증 등—은 다루기 어려웠습니다.

Operations는 Kubernetes Job과 유사하게 **함수 파이프라인을 완료까지 실행**합니다. 일반 컴포지션과 달리 지속적으로 조정하지 않고, 특정 작업을 수행하고 결과를 보고합니다.

### CronOperation으로 자동화된 운영

```yaml
# 매일 아침 6시에 인증서 만료 확인
apiVersion: ops.crossplane.io/v1alpha1
kind: CronOperation
metadata:
  name: cert-expiry-checker
spec:
  schedule: "0 6 * * *"  # 매일 06:00
  mode: Pipeline
  pipeline:
    - step: check-certificates
      functionRef:
        name: function-cert-checker
      input:
        apiVersion: cert-checker.fn.example.io/v1
        kind: CertCheckInput
        spec:
          warningDaysBeforeExpiry: 30
          namespaces:
            - production
            - staging
    - step: notify-slack
      functionRef:
        name: function-slack-notifier
      input:
        apiVersion: slack.fn.example.io/v1
        kind: SlackNotification
        spec:
          channel: "#platform-alerts"
          onlyOnWarning: true
```

### EventOperation으로 이벤트 기반 자동화

```yaml
# 리소스 변경에 반응하는 이벤트 기반 운영
apiVersion: ops.crossplane.io/v1alpha1
kind: EventOperation
metadata:
  name: auto-backup-on-scale
spec:
  watch:
    apiVersion: postgresql.cnpg.io/v1
    kind: Cluster
  trigger:
    fieldPath: spec.instances
    onChange: true
  mode: Pipeline
  pipeline:
    - step: create-backup
      functionRef:
        name: function-cnpg-backup
      input:
        apiVersion: backup.fn.example.io/v1
        kind: BackupInput
        spec:
          retentionDays: 30
          target: "s3://backups/pre-scale/"
```

> **왜 Operations가 필요한가:** 인프라 관리는 프로비저닝만이 아닙니다. 프로덕션 환경에서는 Day-2 운영—모니터링, 백업, 업그레이드, 규정 준수 확인—이 전체 작업의 80%를 차지합니다. Operations는 이러한 운영 작업도 선언적으로 관리할 수 있게 해줍니다. 스크립트와 크론잡으로 관리하던 운영 자동화가 이제 Crossplane의 일급 시민이 되었습니다.

## Managed Resource Definition으로 클러스터 경량화

Crossplane Provider를 설치하면 수백 개의 CRD가 한꺼번에 등록됩니다. `provider-aws`의 경우 1,000개 이상의 CRD가 생성되어, API 서버에 상당한 부하를 줄 수 있습니다. v2에서는 **Managed Resource Definition(MRD)**을 통해 실제로 사용하는 리소스만 선택적으로 활성화할 수 있습니다.

```yaml
# 필요한 리소스만 선택적으로 활성화
apiVersion: apiextensions.crossplane.io/v1alpha1
kind: ManagedResourceDefinition
metadata:
  name: buckets.s3.aws.m.upbound.io
spec:
  managedResourceDefinitionRef:
    name: s3-bucket
---
apiVersion: apiextensions.crossplane.io/v1alpha1
kind: ManagedResourceDefinition
metadata:
  name: instances.rds.aws.m.upbound.io
spec:
  managedResourceDefinitionRef:
    name: rds-instance
```

이 접근법의 장점은 분명합니다:
- **API 서버 부하 감소**: 불필요한 CRD를 등록하지 않아 메모리와 etcd 사용량이 줄어듭니다.
- **보안 표면 축소**: 활성화되지 않은 리소스 타입은 생성 자체가 불가능합니다.
- **빠른 시작**: Provider 시작 시간이 대폭 단축됩니다.

벤치마크에 따르면, 전체 AWS Provider 대비 필요한 10-20개 리소스만 활성화했을 때 **메모리 사용량이 약 60-70% 감소**하고, **API 서버 응답 시간이 약 40% 향상**되는 것으로 보고됩니다.

## 실전: 내부 개발자 플랫폼 구축 아키텍처

지금까지 살펴본 Crossplane v2의 기능들을 종합하여, 프로덕션 수준의 내부 개발자 플랫폼을 설계해 봅시다.

### 플랫폼 계층 구조

```
┌─────────────────────────────────────────────┐
│           개발자 인터페이스                       │
│  (kubectl, ArgoCD, Backstage, 자체 포탈)       │
├─────────────────────────────────────────────┤
│           플랫폼 API (XRD)                      │
│  Application, Database, Cache, Queue         │
├─────────────────────────────────────────────┤
│         컴포지션 함수 파이프라인                   │
│  [리소스생성] → [보안정책] → [관측성] → [검증]    │
├─────────────────────────────────────────────┤
│         Crossplane Providers                 │
│  AWS / GCP / Azure / Kubernetes / Helm       │
├─────────────────────────────────────────────┤
│         클라우드 인프라                          │
│  RDS, S3, EKS, CloudSQL, GKE, ...          │
└─────────────────────────────────────────────┘
```

### ArgoCD와의 GitOps 통합

Crossplane과 ArgoCD를 결합하면 완전한 GitOps 기반 인프라 관리가 가능합니다:

```yaml
# ArgoCD Application - 팀의 인프라 정의를 Git에서 동기화
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: team-backend-infra
  namespace: argocd
spec:
  project: platform
  source:
    repoURL: https://github.com/example/team-backend-infra
    targetRevision: main
    path: overlays/production
  destination:
    server: https://kubernetes.default.svc
    namespace: team-backend
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
    syncOptions:
      - CreateNamespace=true
```

팀의 Git 리포지토리에 Crossplane XR 매니페스트를 커밋하면, ArgoCD가 이를 클러스터에 동기화하고, Crossplane이 실제 클라우드 리소스를 프로비저닝합니다. 이 파이프라인에서 **이중 조정(double reconciliation)**이 일어납니다: ArgoCD가 Git→클러스터를, Crossplane이 클러스터→클라우드를 조정합니다.

### 환경별 구성 전략

```yaml
# base/application.yaml - 환경 공통 정의
apiVersion: platform.example.io/v1
kind: Application
metadata:
  name: payment-service
spec:
  image: registry.example.io/payment-service
  database:
    engine: postgres

---
# overlays/production/patch.yaml - 프로덕션 오버레이
apiVersion: platform.example.io/v1
kind: Application
metadata:
  name: payment-service
spec:
  replicas: 5
  database:
    storageGB: 200
    backup:
      enabled: true
      retentionDays: 90
  crossplane:
    compositionRef:
      name: ha-app  # 고가용성 컴포지션
```

## 마이그레이션 전략: Terraform에서 Crossplane v2로

기존 Terraform 인프라를 Crossplane으로 마이그레이션하는 것은 빅뱅 전환이 아니라 점진적 접근이 필요합니다.

### 단계별 마이그레이션 로드맵

**Phase 1: 공존 (1-2개월)**
- 새 인프라는 Crossplane으로 생성
- 기존 Terraform 인프라는 그대로 유지
- `provider-terraform`으로 기존 모듈 재사용

```yaml
# provider-terraform으로 기존 모듈 활용
apiVersion: tf.upbound.io/v1beta1
kind: Workspace
metadata:
  namespace: team-backend
  name: legacy-vpc
spec:
  forProvider:
    source: Remote
    module: |
      module "vpc" {
        source  = "terraform-aws-modules/vpc/aws"
        version = "5.1.0"
        
        name = "team-backend-vpc"
        cidr = "10.0.0.0/16"
        
        azs             = ["ap-northeast-2a", "ap-northeast-2c"]
        private_subnets = ["10.0.1.0/24", "10.0.2.0/24"]
        public_subnets  = ["10.0.101.0/24", "10.0.102.0/24"]
      }
```

**Phase 2: 점진적 전환 (2-4개월)**
- Crossplane 네이티브 Provider로 리소스를 하나씩 import
- 컴포지션으로 표준 플랫폼 API 구축
- 팀별로 새 API 마이그레이션

**Phase 3: 완전 전환 (4-6개월)**
- 모든 인프라가 Crossplane 관리 하에
- Terraform 상태 파일 아카이빙
- Operations로 Day-2 운영 자동화

> **중요한 점:** 마이그레이션의 목표는 도구를 바꾸는 것이 아니라, **플랫폼 추상화를 구축하는 것**입니다. Terraform에서 Crossplane으로 1:1 전환하는 것은 의미가 없습니다. 개발자가 사용하는 상위 수준 API(Application, Database 등)를 설계하고, 그 구현체로 Crossplane을 활용하는 것이 올바른 접근입니다.

## 프로덕션 운영 팁과 주의사항

### 성능 튜닝

Crossplane 컨트롤러의 기본 설정은 소규모 환경에 맞춰져 있습니다. 프로덕션에서는 다음 설정을 조정하세요:

```yaml
# Crossplane Helm values - 프로덕션 설정
args:
  - --max-reconcile-rate=100  # 기본값 10, 대규모 환경에서 증가
  - --poll-interval=5m        # 클라우드 API 폴링 간격
resources:
  requests:
    cpu: "500m"
    memory: "1Gi"
  limits:
    cpu: "2"
    memory: "4Gi"
```

### 디버깅 전략

Crossplane 리소스의 상태 추적은 `kubectl`로 가능합니다:

```bash
# XR과 하위 리소스 전체 상태 확인
kubectl get application user-service -n team-backend -o yaml

# 이벤트 확인
kubectl get events -n team-backend --field-selector involvedObject.name=user-service

# 컴포지션 함수 로그 확인
kubectl logs -n crossplane-system -l pkg.crossplane.io/function=function-kcl

# Crossplane 트레이스 (리소스 계층 추적)
kubectl get managed -l crossplane.io/composite=user-service
```

### 보안 모범 사례

1. **ProviderConfig 분리**: 팀별로 다른 클라우드 자격증명을 사용하도록 ProviderConfig를 네임스페이스별로 설정합니다.
2. **Webhook 검증**: 컴포지션 함수에 입력값 검증 로직을 포함하여, 잘못된 설정이 클라우드에 전달되는 것을 방지합니다.
3. **감사 로깅**: Kubernetes Audit Log와 Crossplane 이벤트를 중앙 로깅 시스템에 수집합니다.

## 결론: 플랫폼 엔지니어링의 미래

Crossplane v2는 단순한 IaC 도구가 아닙니다. **Kubernetes 네이티브 플랫폼 엔지니어링 프레임워크**입니다. 네임스페이스 기반 리소스 관리로 멀티테넌시가 자연스러워졌고, 모든 Kubernetes 리소스에 대한 컴포지션으로 진정한 풀스택 추상화가 가능해졌으며, Operations로 Day-2 운영까지 선언적으로 관리할 수 있게 되었습니다.

AI 에이전트가 인프라를 관리하는 시대가 다가오고 있습니다. 에이전트가 효과적으로 작동하려면 **일관된 API, 선언적 인터페이스, 자동 조정**이 필요합니다. Crossplane v2의 Kubernetes API 기반 접근법은 이러한 요구에 정확히 부합합니다. 에이전트는 kubectl을 통해 인프라를 발견하고, 선언적 매니페스트로 의도를 표현하며, 컨트롤러에게 실행을 위임할 수 있습니다.

플랫폼 엔지니어링을 시작하려는 팀이든, 기존 Terraform 기반 인프라를 현대화하려는 팀이든, Crossplane v2는 진지하게 검토할 가치가 있습니다. CNCF Graduated라는 성숙도, 활발한 커뮤니티, 그리고 v2의 혁신적 변화는 Crossplane이 앞으로 수년간 플랫폼 엔지니어링의 핵심 도구로 자리잡을 것임을 보여줍니다.
